# 第 12 章 多渠道：一个大脑，七张嘴

---

## 第三个平台开始失控

Agent 能聊天了。我把它接上了 Telegram，花了一天。工作正常。

老板说："能不能也接上钉钉？" 又花了一天。也跑起来了。

然后是飞书。然后是企业微信。然后是 QQ。

到第三个平台的时候，我发现了问题。

不是"接不上"的问题——每个平台的 API 都能调通。是"接上之后出了一堆 bug"的问题。

钉钉的消息长度上限和 Telegram 不一样。Telegram 能发 4,096 个字符的消息，钉钉大概 20,000 个，企业微信只有 2,048 个。Agent 生成了一段 6,000 字的回复，在 Telegram 上好好的，在企业微信上直接报错——消息太长，发送失败。

Telegram 支持 Markdown 格式。Agent 的回复里有 `**加粗**` 和 `### 标题`，在 Telegram 上渲染得很漂亮。在企业微信上，用户看到的是原始的星号和井号。一个用户截图发到群里吐槽："这个机器人连格式都不会排。"

更离谱的是超时问题。Agent 有一个"问用户"的功能——它遇到不确定的事情会反问用户，然后等用户回答。在 WebSocket 连接的 Web 界面上，这很简单——用户随时在看屏幕，几秒钟就回复了。但在 Telegram 上，用户可能问完问题就关了 App 去吃饭了。Agent 的"问用户"功能挂着一个等待 Promise，等了 14 个小时，直到有人重启了网关才释放。这 14 个小时里，这个用户的会话完全冻结了。

最让我崩溃的是：这些 bug 中有好几个，我两周前在 Telegram 上已经修过了。修完忘了同步到钉钉。因为每个平台的代码是独立写的，修了一个不影响另一个——直到你自己都忘了哪个修了哪个没修。

**多渠道不是一个"接 API"的问题。它是一个分布式系统问题，只是伪装成了集成任务。**

---

## 核心判断

> **在渠道代码里复制的每一行逻辑，都是一个等着在某一个渠道上炸掉的 bug。**

平台表面上看起来很像——收消息、调 AI、发回复。但差异藏在细节里：Telegram 渲染 Markdown，企业微信不渲染。Telegram 能内联上传 50MB 的文件，钉钉给你一个有时效的 webhook URL。QQ 需要被动回复令牌，5 分钟过期。每一个泄漏到核心处理流程里的渠道特定行为，都是随着新平台接入而指数级增长的维护负担。

目标是：**渠道代码只负责传输——收字节、发字节。** 其他所有事情——会话管理、超时处理、事件流处理、格式适配——属于共享基础设施。

---

## 一个平台一套代码，为什么会失控

让我们算一笔账。

假设你有 5 个渠道，每个渠道的代码里都需要这些逻辑：

| 重复的逻辑 | 每个渠道的代码行数 | Bug 风险 |
|-----------|------------------|---------|
| 会话创建和恢复 | ~25 行 | 一个渠道忘了清理，会话泄漏 |
| 带超时的"问用户" | ~15 行 | 一个渠道没加超时，Promise 永远挂起 |
| 发送文件的 URL 拼接 | ~10 行 | 一个渠道的 URL 格式拼错了 |
| 事件流处理 | ~40 行 | 不同渠道的错误消息不一致 |
| 消息分割 | ~5 行 | 一个渠道忘了分割，触发 API 限制 |

加起来大概 95 行逻辑。5 个渠道，就是 475 行几乎一模一样的代码，每一份都在独立地漂移。

打个比方：你开了 5 家连锁店，每家店都有自己的收银系统、库存管理、员工排班。第一家店的收银 bug 修了，其他四家不知道。第一家店的排班规则改了，其他四家还是旧的。每开一家新店，不是复制粘贴就完事——你还得复制粘贴之前所有修过的 bug。

这还只是 5 个渠道。如果你要接入 7 个、10 个、15 个呢？

## ChannelManager：一个管家管所有渠道

解决渠道生命周期问题的方法是：**不要让每个渠道自己管自己。找一个管家。**

这个管家叫 `ChannelManager`。它的职责很简单：管理所有渠道的启动、停止、状态监控和热重载。

每个渠道，不管底层用什么技术（WebSocket、长轮询、私有 SDK），都暴露相同的两个函数：

- `stop()` —— 优雅关闭
- `broadcast(text)` —— 给所有已知会话推送一条消息

就这两个。不多不少。

管家的工作流程：

```
启动时：
  遍历所有渠道 → 跳过没配置的 → 逐个启动 → 捕获个别失败（不阻塞其他渠道）

运行中：
  监控状态 → 连接中 / 已连接 / 错误 / 未配置

热重载：
  对比新旧配置 → 只重启凭据有变化的渠道 → 其他渠道不受影响
```

这里有一个关键设计选择：**一个渠道启动失败，不能拖垮其他渠道。**

早期版本用的是 `Promise.all()` ——所有渠道同时启动，任何一个抛异常，全部失败。听起来不合理，但代码写起来最简单。直到有一天，钉钉的凭据过期了，整个网关启动失败——连 Telegram 和 Web 界面都不能用了。

改完之后：每个渠道独立启动，独立捕获异常。钉钉凭据错了？记一条错误日志，标记状态为 `error`，Telegram 照常运行。

这就像连锁店的管理系统：一家店断网了，其他店照常营业。你不会因为北京的店断网就把上海的店也关了。

## 共享逻辑：从 120 行到 30 行

找到了管家之后，下一步是把重复的逻辑提取出来。

AgentClaw 把所有渠道共享的逻辑提取到了一个叫 `channel-utils.ts` 的模块里。这个模块提供一组工厂函数，每个函数创建一个回调，匹配工具执行上下文（ToolExecutionContext）的接口。

核心的共享逻辑有三块：

### 第一块：带超时的"问用户"

Agent 有一个 `ask_user` 工具。它向用户提一个问题，然后等用户回答。在 Web 界面上，这个等待通过 WebSocket 解决——用户发下一条消息时，Promise 就 resolve 了。

但在 Telegram、钉钉、QQ 上，用户可能不回复。可能关了 App。可能去开会了。可能睡着了。

如果"问用户"没有超时保护，一个不回复的用户就能永久冻结一个会话。Agent 循环卡在那里，LLM 迭代预算被白白占用，其他用户不受影响（异步），但这个用户的会话永远挂着。

AgentClaw 生产环境出过这个事故：一个 Telegram 用户触发了 `ask_user`，然后去睡觉了。会话挂了 14 个小时，直到有人重启网关。

修复方案是机械的：

```
ask_user 流程：
  1. 把问题发给用户
  2. 把 resolve 回调存到 pendingPrompts 映射里
  3. 启动 5 分钟计时器
  4. 如果用户回复了 → 清除计时器，用回答 resolve
  5. 如果计时器到了 → 从映射里删除，用 "[用户未在 5 分钟内回复]" resolve
```

LLM 收到超时消息，就像收到一个正常的工具返回值。它会自己适应——"用户没回复，我用默认方案继续"。没有挂起，不需要人工干预，没有静默失败。

因为这个逻辑在所有渠道上完全一样，它放在 `createPromptUser()` 工厂函数里——一份实现，所有渠道共用。新渠道不可能"忘记加超时"，因为它从来不需要自己实现这个逻辑。

### 第二块：文件发送

有些渠道能直接上传文件（Telegram 支持内联文件上传，图片会预览，文档会显示下载按钮）。有些渠道不能（钉钉给你一个有时效的 webhook URL，企业微信限制更多）。

对于不能直接上传文件的渠道，退化方案是生成一个下载链接。用户点链接下载文件。

`createLinkSendFile()` 工厂函数根据渠道的能力返回不同的实现。能上传的走上传，不能上传的走链接。上层代码不需要知道区别。

### 第三块：事件流处理

Agent 的回复是流式生成的——一个字一个字地出来。在 Web 界面上，这很自然——WebSocket 实时推送每个字。但在很多聊天平台上，你不能一个字一个字地发消息。你需要先把所有字攒起来，然后一次性发出去。

`processSimpleEventLoop()` 工厂函数处理这个流程：累积文本 → 按平台限制分割 → 逐条发送。

## 消息分割：每个平台的限制都不一样

每个聊天平台都有消息长度上限。而且没有两个平台的上限是一样的：

| 平台 | 消息长度上限 | 超出后会发生什么 |
|------|------------|----------------|
| Telegram | 4,096 字符 | API 报错，消息没发出去 |
| 钉钉 | ~20,000 字符 | 静默截断，用户看到半截消息 |
| QQ | ~4,000 字符 | API 报错 |
| 飞书 | ~30,000 字符 | 因消息类型而异 |
| 企业微信 | ~2,048 字符 | API 报错 |
| WhatsApp | ~65,536 字符 | 很少触发 |

LLM 不知道这些限制。它生成了一段 6,000 字的回复——因为问题确实需要这么多内容。如果不做分割，这段回复要么报错，要么被截断——用户看到半截答案，没有任何提示说后面还有内容。

AgentClaw 的 `splitMessage()` 函数处理这个问题。分割策略是优先级有序的：

1. 先尝试在段落边界分割（双换行符）
2. 段落太长就在行边界分割（单换行符）
3. 行也太长就在词边界分割（空格）
4. 最后兜底：直接在限制处硬切

每个平台传入自己的限制——企业微信用 `splitMessage(text, 2048)`，Telegram 用 `splitMessage(text, 4096)`。函数本身不关心平台，只关心"在不超过限制的前提下，怎么切分最合理"。

## 平台提示：告诉 AI 它在跟谁说话

有一个很微妙的问题。

Agent 生成了一段漂亮的回复，里面有 Markdown 标题、加粗文字、项目符号列表。在 Web 界面上，渲染得很好看。在企业微信上，用户看到的是原始的星号和井号。在 Telegram 上，Markdown 确实能渲染——但语法跟标准 CommonMark 不完全一样。

你可以在发送前做后处理——每个渠道写一套 Markdown 转换器。但这意味着你要解析 Markdown，这很脆弱，而且你丢失了语义意图。

更好的方案是：**在 LLM 生成回复之前，告诉它用户的平台支持什么格式。**

AgentClaw 在系统提示词里注入一行平台提示（Platform Hint）：

| 渠道 | 提示 |
|------|------|
| Telegram | "不要使用 Markdown 格式（不会渲染）。媒体发送设置 auto_send: true。" |
| 钉钉 | "支持 Markdown 格式。" |
| 企业微信 | "不要使用 Markdown。消息尽量简短。" |
| 飞书 | "支持富文本和 Markdown。" |
| QQ | "有限的 Markdown 支持。消息尽量简短。" |
| CLI | "纯文本输出。避免过多 Markdown。" |
| WebSocket（Web 界面） | （不加提示——完整渲染支持） |

这条提示的成本大概是 15 到 25 个 token。它省掉了每条回复的后处理，而且从根本上消除了格式错乱的问题。**防止错误输出的最便宜方式，是让错误输出根本不被生成。**

提示在会话创建时注入一次，不是每条消息都注入。它成为冻结系统提示词快照的一部分——如果用的是 Anthropic 的模型，它会被 prompt prefix cache 覆盖，后续调用实际上不额外花钱。

这就像你告诉一个翻译"这段话要翻译给日本客户"，他自然会用敬语、避免俚语。你不需要事后逐句改他的译文——他在生成阶段就知道该用什么风格。

## 那些不能抽象的部分

不是所有逻辑都适合放到共享代码里。有些行为是真正平台特定的。

**Telegram** 支持内联文件上传——图片直接预览，文档显示下载按钮。它还支持流式输出——通过编辑草稿消息，回复看起来像在"实时打字"。我们支持的其他渠道没有这个能力。

**钉钉** 用会话 webhook 模型——消息到达时附带一个 webhook URL，有一定时效。你必须通过这个 webhook 回复，不能用通用 API。这意味着"发回复"的底层机制跟 Telegram 的 `bot.api.sendMessage()` 完全不同。

**QQ** 需要被动回复令牌。每条收到的消息携带一个 `msg_id` 和序列号。你的回复必须引用这些信息，而且令牌会过期。如果你的 Agent 处理了 6 分钟（复杂的工具链），回复令牌就死了，你需要一个降级机制。

**WhatsApp** 通过非官方库管理自己的持久化认证状态，首次设置需要扫描 QR 码，而且出于安全原因只在自聊模式下运行。

**飞书** 支持富文本消息和 Markdown，但两者的 API 端点不同。富文本消息可以包含交互式卡片（按钮、下拉菜单），但构造卡片的 JSON 结构比纯文本复杂得多。Agent 不需要知道这些区别——平台提示告诉它"支持 Markdown"就够了，卡片交互是飞书渠道文件自己的事。

**企业微信** 的消息长度限制是所有渠道里最紧的——只有 2,048 个字符。而且它不支持 Markdown。这意味着在企业微信上，Agent 的每条回复都可能触发分割，而且分割后的纯文本可能丢失原本的结构化信息。平台提示"消息尽量简短"不是客气话——是字面意思。

这些差异属于各自的渠道文件，不属于共享工具模块。

判断标准很简单：**如果删掉这段逻辑只影响一个渠道，它就属于那个渠道的文件。如果影响两个或更多渠道，就提取出来。**

## 完整架构：从用户消息到 AI 回复

把所有层叠在一起，数据流长这样：

完整数据流可以拆成六步：

1. 用户从 Telegram、钉钉、QQ、飞书、企业微信、WhatsApp 或 Web 发来消息。
2. 渠道文件只负责传输：用平台 SDK 认证，把收到的消息映射成文本或内容块，然后调用共享处理管线。
3. `channel-utils.ts` 处理共享逻辑：`createPromptUser()` 负责 5 分钟超时，`createLinkSendFile()` 负责 URL 拼接，`processSimpleEventLoop()` 负责累积、分割和发送。
4. `Orchestrator.processInputStream(sessionId, input, toolContext)` 接管任务。AI 大脑完全不知道渠道的存在，只发出 `tool_call`、`response_chunk`、`response_complete` 等事件。
5. `platform-hints.ts` 在会话创建时注入平台提示，告诉 LLM 当前渠道支持什么格式。
6. `ChannelManager` 管理生命周期和状态：启动、停止、广播、热重载，以及已连接、断开、错误、未配置等状态。

Orchestrator 永远不知道它在服务哪个渠道。它接收文本（或带图片的内容块），通过 Agent 循环处理，发出事件。渠道层把这些事件翻译成平台特定的 API 调用。平台提示是唯一到达 LLM 的渠道感知信息——而且它只是系统提示词里的一句话。

**大脑不需要知道嘴巴长什么样。大脑只需要思考，嘴巴负责说话。**

让我们用一个具体例子走一遍。一个企业微信用户发了一条消息："帮我查一下下周北京的天气，顺便推荐几个适合下雨天去的室内景点。"

1. 企业微信渠道文件收到消息，映射成文本
2. 调用共享管线 `processAndReply(sessionId, text, toolContext)`
3. Orchestrator 创建会话（如果不存在），注入平台提示："不要使用 Markdown。消息尽量简短。"
4. Agent 循环开始：LLM 调用 → web_search 工具查天气 → web_search 工具查景点 → LLM 整理回复
5. LLM 生成了 3,200 字的回复（因为它不知道企业微信的限制）
6. 事件流处理器累积文本，调用 `splitMessage(text, 2048)`
7. 分割成 2 条消息，依次通过企业微信 API 发送
8. 用户看到两条消息，格式干净，没有星号和井号

如果同一个问题从 Telegram 进来：平台提示不同，LLM 生成的回复风格会略有不同；消息分割用 4,096 的限制，可能只分成 1 条；Telegram 渠道还会尝试用 Markdown 渲染标题和加粗。

同一个大脑，不同的嘴巴，适应不同的听众。

## 我们评估过但放弃的方案

在找到当前架构之前，我们认真考虑过几个替代方案。

**通用消息格式。** 设计一个渠道无关的富文本抽象语法树（AST），每个渠道负责渲染。我们做了原型。问题在于：每个平台的"Markdown"差异大到通用 AST 要么退化成纯文本（最低公共标准），要么需要为每个平台写渲染器——这比直接让 LLM 通过平台提示生成合适的文本更难维护。

**队列架构。** 消息进来、响应出去、渠道作为独立消费者。加了 Redis 或 RabbitMQ 的基础设施复杂度，但我们的规模不需要。单个 Node.js 进程通过 async/await 就能并发处理所有渠道。等到 10,000 个以上并发会话时再重新考虑。

**每个渠道用不同的 LLM 人格。** 钉钉用户倾向于问工作相关的问题，Telegram 用户什么都问。我们考虑过路由到不同的模型或不同的系统提示词。最终放弃了——维护多个人格的认知开销超过了任何质量收益。一个大脑，一个人格，渠道适配的格式。

## 从"试试看"到"稳定跑"

多渠道的演进不是一步到位的。AgentClaw 的渠道系统经历了三个阶段。

**阶段一：各自为政。** 每个渠道一个文件，每个文件里复制粘贴了会话管理、超时处理、消息分割等逻辑。3 个渠道，代码能跑。5 个渠道，开始出 bug。7 个渠道，每个渠道都有自己的"个性"——同样一个 bug，在 Telegram 上修了，在钉钉上还在。

**阶段二：提取共享逻辑。** 把重复的代码提到 `channel-utils.ts`，用工厂函数创建回调。每个渠道的代码从 120 行降到 30 行。bug 修一处，所有渠道都修了。

**阶段三：管家上线。** `ChannelManager` 统一管理生命周期，支持热重载、状态监控、独立容错。新增一个渠道只需要写 30 行代码：认证、消息映射、回复映射。

这三个阶段，对应的是一个很朴素的工程原则：**先让它跑起来，再让它好维护，最后让它好扩展。** 不要一上来就设计完美的抽象——你不知道哪些逻辑会重复，直到你真的写了三遍。

如果你从一开始就试图设计一个"完美的渠道抽象层"，大概率会过度设计——你会预判哪些逻辑需要共享，但你的预判可能是错的。真正的共享模式是在写了三个渠道之后才浮现的。第一个渠道写完整逻辑。第二个渠道复制粘贴，微调。第三个渠道你开始烦了——"这已经是第三遍了"。这时候你知道该提取什么了。

---

## 真实数据：新增一个渠道有多简单

以飞书渠道为例。飞书用私有 SDK，有自己的消息格式、文件上传方式和事件订阅机制。

在阶段一（各自为政），新增飞书渠道需要：写认证逻辑、写会话管理、写超时处理、写文件发送、写消息分割、写事件流处理。大概 120 行代码，而且要从 Telegram 渠道复制粘贴大量逻辑，手动调整 API 调用。

在阶段三（管家 + 共享逻辑），新增飞书渠道需要：

1. 认证（接入飞书 SDK，~8 行）
2. 收到消息 → 映射成文本 → 调 `processAndReply`（~10 行）
3. 发送回复 → 调飞书 API（~8 行）
4. 发送文件 → 调飞书文件上传（~4 行）

总共约 30 行。超时？`createPromptUser()` 已经处理了。消息分割？`splitMessage()` 已经处理了。事件流？`processSimpleEventLoop()` 已经处理了。

**从 120 行到 30 行，不是因为飞书更简单，而是因为共享基础设施承担了 75% 的工作。**

## 本章小结

1. **多渠道不是"接 API"的问题。** 它是分布式系统问题——会话管理、超时处理、消息分割、格式适配，每个渠道都要做，每个渠道都不一样。

2. **渠道代码只负责传输。** 收字节、发字节。其他逻辑属于共享基础设施。

3. **ChannelManager 统一管理生命周期。** 启动、停止、状态监控、热重载。一个渠道挂了不影响其他渠道。

4. **5 分钟超时是必须的，不是建议。** 没有超时的 `ask_user` 会永久冻结会话。这不是理论风险，是生产事故。

5. **平台提示是最便宜的格式适配。** 一行话、15 到 25 个 token，在系统提示词里注入一次。比事后处理每条回复便宜得多，效果也更好。

6. **消息分割必须按平台限制。** 没有两个平台的限制是一样的。共享的 `splitMessage()` 函数让每个渠道传入自己的限制，逻辑只写一次。

7. **先跑起来，再好维护，最后好扩展。** 不要一上来就设计完美抽象。写了三遍之后，你自然知道哪些该提取。

**任务 1：审计重复代码。**

如果你手头有涉及多个渠道（或多种输出格式）的代码，检查一下：会话管理、超时处理、消息分割这些逻辑，是不是在每个渠道里都写了一遍？列一个清单，标出哪些逻辑重复了、重复了几份。

**任务 2：设计一个新渠道的接入清单。**

假设你要给你的 Agent 接入一个新的聊天平台（比如 Slack 或 Line）。画一个清单，标出：

- 必须平台特定的部分（认证、消息格式映射、文件上传方式）
- 应该共享的部分（会话管理、超时处理、消息分割、事件流处理）
- 需要注入的平台提示（这个平台支持什么格式？消息长度上限是多少？）

如果"应该共享"的部分比"必须特定"的多，说明你的架构是健康的。如果反过来，说明有大量逻辑泄漏到了渠道层。
