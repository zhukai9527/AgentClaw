# 第 14 章 我们犯过的错，以及你不必再犯

---

## 一个 cat 命令炸掉了一切

那天晚上，我让 AgentClaw 帮我查看一个日志文件。

这本来是最简单的任务——用 shell 工具执行 `cat server.log`，把文件内容返回给我。一行命令，零难度。

然后我的屏幕开始疯狂滚动。

Agent 把整个日志文件——三万多行、将近 2MB 的文本——全部塞进了对话上下文。下一轮调用 LLM 时，上下文窗口直接被撑满。Agent 开始"忘事"：它记不住我最开始说的"只看最后 50 行错误"这个需求了。它试图分析所有日志，但信息太多，它开始胡言乱语。它甚至试图把日志内容写入另一个文件，然后读取那个文件——结果那个文件也很大，上下文被撑得更满了。

整个会话废了。

这不是模型的错。模型很聪明，它只是被塞了太多信息。这也不是用户的错。用户说"看看日志"是完全合理的请求。

这是框架的错。框架没有做输出溢出处理（Output Overflow Handling）。一个 `cat` 命令就能炸掉整个上下文，而框架没有任何防护。

这个 bug 修复花了两天。不是因为修复本身复杂——最终只是加了一个"输出超过阈值就截断，只给模型看预览"的机制。而是因为我们一开始根本没想到这个问题。谁会担心一个 `cat` 命令呢？

**这一章，我要把 AgentClaw 从零到一过程中犯过的所有错误摊开给你看。**不是那种"我们遇到了挑战，然后克服了"的励志故事。而是真实的、具体的、有时候甚至有点丢人的错误——以及我们从中学到的东西。

---

## 为什么要写错误

你可能觉得，一本技术书应该讲"怎么做对"。讲最佳实践、讲架构设计、讲经过验证的方案。

但真实的工程不是这样的。

真实的工程是：你做了一个看起来很合理的决定，运行了一个多月，才发现它是个错误。然后你花两周时间把它删掉，重写。然后你发现重写的版本也有问题。然后再改。

AgentClaw 的 commit 历史里，有大量被删除的代码。中间件链（Middleware Chain）——删了。能力协商（Capability Negotiation）——删了。自定义校验器（Custom Validator）——删了。好几个"聪明"的设计方案，写的时候觉得很好，用了几周发现是负担，最后都删了。

**被删掉的代码比保留下来的代码更有教育价值。**因为它告诉你：哪些方案看起来聪明，实际上行不通。哪些设计看起来优雅，实际上增加了复杂度。哪些"最佳实践"在 Agent 这个领域里不适用。

我不打算美化这些错误。每一个错误，我都会写清楚四件事：

1. **根因**：到底是什么导致了问题？
2. **误判**：我们当时为什么觉得那样做是对的？
3. **修复成本**：花了多少时间和精力来修正？
4. **教训**：下次怎么避免？

## 错误一：工具描述比工具实现更影响效果

### 发生了什么

AgentClaw 有一个 `execute_code` 工具，让 Agent 能执行代码。工具描述里写的是"Execute JavaScript code"——执行 JavaScript 代码。

然后我们发现，某些模型——特别是 Doubao 的一些版本——会反复在这个工具里写 Python 代码。它 `import pdfplumber`，然后报 SyntaxError。它又试了一次，换了个 Python 库，又报 SyntaxError。又试了一次。

连续五次。

工具描述里明明写了"JavaScript"，模型为什么视而不见？

### 根因

大语言模型不是在"读"工具描述，它是在"看"工具描述。这两者的区别很微妙，但很重要。

"读"意味着逐字逐句理解，不遗漏任何细节。"看"意味着扫一眼，抓住大意，然后根据自己的经验填补细节。

模型的经验告诉它：`execute_code` 这种工具通常是通用的，能执行任何语言的代码。所以即使描述里写了"JavaScript"，它也会根据任务需要——比如处理 PDF——自动选择 Python，因为它"知道"Python 处理 PDF 更方便。

这不是模型的 bug。这是模型的工作方式。它用概率来决策，而不是用规则来决策。"JavaScript"这个词在描述里出现了，但"处理 PDF 用 Python"这个先验知识更强。

### 误判

我们当时觉得，工具描述是给模型看的"说明书"，写清楚就行了。"Execute JavaScript code"——多清楚啊。模型应该能理解。

但事实是：**工具描述不是说明书，是控制指令。**它需要对抗模型的先验知识，而不是假设模型会遵守。

### 修复成本

这个错误导致了三轮修复：

第一轮：把描述改成"Execute **JavaScript ONLY** (NOT Python)"。加粗、加括号、加否定。模型还是偶尔犯错。

第二轮：在系统提示词里加了一条规则："execute_code 工具只支持 JavaScript，不支持 Python。"这减少了犯错频率，但没有完全消除。

第三轮：在工具参数里加了语言校验，如果检测到 Python 语法（`import`、`def`、`print(`），直接返回错误提示，而不是让解释器报 SyntaxError。

三轮修复，改了三个不同位置的文件。

### 教训

**工具描述的质量比工具实现的质量更影响 Agent 的行为。**

一个描述模糊的工具，即使实现完美，Agent 也会用错。一个描述精准的工具，即使实现简单，Agent 也能用对。

具体来说：

- 关键约束要在描述里重复强调，不要只写一次。模型的注意力不是均匀分布的，开头和结尾的词比中间的词更容易被注意到。
- 对抗模型的先验知识需要多层防护：描述里写、系统提示词里写、参数校验里拦。指望一层防护就够了，是不现实的。
- 测试工具描述时，不要只测试"正常路径"。要故意让模型犯错，看看它在什么情况下会不遵守描述。

这个教训后来影响了 AgentClaw 所有工具的设计。每个工具的 `description` 字段，我们都会反复打磨，确保关键约束被放在最显眼的位置。

## 错误二：没有溢出处理，一个 cat 就能炸掉上下文

### 发生了什么

就是开篇讲的那个故事。`cat server.log` 把三万行日志塞进了上下文，整个会话废了。

但这个问题不只是 `cat`。任何输出很大的工具都有这个风险：`grep` 匹配了太多结果、`file_read` 读了一个大文件、`web_fetch` 抓了一个很长的网页。只要工具输出超过上下文窗口的可用空间，Agent 就会开始"忘事"。

### 根因

Agent 的上下文窗口（Context Window）是有限的。即使是 128K token 的大窗口，扣掉系统提示词、工具定义、历史对话之后，留给工具输出的空间可能只有 20-30K token。一个中等大小的文件就能把它填满。

更糟糕的是，上下文被填满后，Agent 不会"报错"。它会继续运行，但开始丢失早期信息。用户最初的需求、之前的对话历史、甚至系统提示词的一部分，都可能被挤出去。Agent 变成了一个"失忆"的状态——它还在运行，但已经不记得自己在做什么了。

这比直接报错更危险。报错至少你知道出了问题。失忆的 Agent 会继续执行，但方向完全错了，而你可能要等到结果出来才发现。

### 误判

我们当时觉得，工具输出越大越好——信息越多，Agent 做出的决策越准确。

这个想法在理论上是对的。如果 Agent 能处理无限的信息，给它越多信息确实越好。但现实中，上下文是有限的，信息越多，每条信息分到的"注意力"就越少。给 Agent 三万行日志，它反而不知道该看哪一行了。

**信息过载不只是浪费 token，它会降低决策质量。**

### 修复成本

修复方案分两层：

第一层：工具输出截断。每个工具的输出都有一个上限（比如 8000 字符）。超过上限的内容被写入磁盘文件，只给模型看一个预览摘要。

第二层：溢出提示词设计。截断后的提示词引导 Agent 利用预览信息，而不是鼓励它去读完整文件。

第二层尤其重要。我们最初写的溢出提示是"输出已截断，请用 file_read 查看完整内容"。这导致 Agent 发起了 6 个工具调用（grep + 5 次 file_read），消耗了 134K token，耗时 63 秒。

改成"预览中的信息通常足够回答问题"之后，工具调用降到 1 个，token 降到 28K（减少 79%），耗时降到 7.7 秒（减少 88%）。

**一句话的措辞差异，带来了 79% 的成本节省。**

### 教训

- 工具输出必须有上限。不设上限等于把炸弹交给 Agent。
- 溢出后的提示词设计比截断本身更重要。提示词是在"引导 Agent 的行为"，不是在"通知 Agent 一个事实"。
- 给 Agent 最小够用的信息，而不是最多可能的信息。这是反直觉的——我们本能地觉得"多给点信息总比少给好"。但在有限的上下文里，少即是多。

## 错误三：子代理不共享预算，导致迭代爆炸

### 发生了什么

AgentClaw 有一个子代理（Subagent）系统。当主 Agent 遇到复杂任务时，它可以启动一个子代理来处理子任务。这听起来很合理——分工协作嘛。

然后我们发现，子代理没有迭代次数限制。

主 Agent 说"帮我分析这个文件"，子代理开始工作。子代理觉得信息不够，又启动了一个子子代理。子子代理又启动了子子子代理。每一层都在消耗迭代次数，但没有任何全局上限。

有一次，一个简单的"帮我看一个文件"的任务，最终消耗了几十次工具调用和大量 token。用户等了很久，最后得到的结果还不如直接自己看文件。

### 根因

问题出在"预算"的设计上。主 Agent 有一个迭代预算（IterationBudget）——最多执行 N 次工具调用。但子代理启动时，没有继承或共享这个预算。子代理有自己独立的计数器，从零开始。

这就像一个公司里，每个部门有自己的预算，但部门经理可以无限创建子部门，每个子部门又有自己的预算。理论上总预算被控制了，实际上可以无限膨胀。

### 误判

我们当时觉得，子代理应该有独立的预算，因为子任务的复杂度不同。有些子任务只需要 2 次工具调用，有些需要 20 次。如果共享预算，简单子任务可能把预算用完，导致复杂子任务无法执行。

这个想法有道理，但忽略了一个关键问题：**子代理可以递归创建子代理。**独立预算在单层时没问题，在多层递归时就变成了无限膨胀。

### 修复成本

修复方案是让父子代理共享同一个 `IterationBudget` 对象。子代理的每次工具调用都计入全局预算。当全局预算用完时，所有代理——不管层级——都必须停止。

这个改动本身不大，但测试很复杂。需要验证：父子共享预算后，简单子任务不会因为父代理已经用了大部分预算而无法执行；复杂子任务不会因为预算不够而中途失败；递归子代理不会绕过预算限制。

### 教训

- 任何可以递归创建的实体，都必须有全局上限。不只是迭代次数，还有内存、文件句柄、网络连接。
- "独立预算"在单层时是合理的，在多层时是危险的。设计时要问：这个实体可以被递归创建吗？如果可以，就需要全局预算。
- 测试时要故意构造"最坏情况"——最大深度的递归、最多层级的嵌套——而不是只测正常的单层情况。

## 错误四：记忆提取频率太高，变成隐性 token 消耗

### 发生了什么

AgentClaw 的记忆系统会在每轮对话后自动提取值得长期保存的信息。用户说"我喜欢用 Tab 缩进"，Agent 会把这条偏好存入记忆。下次对话时，Agent 会加载这些记忆，知道用户的偏好。

这听起来很棒。问题是：**每轮对话都提取记忆，成本很高。**

记忆提取需要调用 LLM——让模型分析对话内容，判断哪些信息值得保存。这不是免费的。每次提取消耗几百到几千 token，取决于对话长度。

如果用户跟 Agent 聊了 20 轮，就有 20 次记忆提取调用。如果同时有 10 个用户在聊天，就有 200 次调用。这些 token 消耗，用户是看不到的——它们不在用户的对话里，而是在后台默默发生。

我们发现，记忆提取的 token 消耗占到了总消耗的 15-20%。这是一个不小的数字。

### 根因

记忆提取的初衷是好的：让 Agent 自动学习，不要求用户主动说"记住这个"。但实现时，我们把它做成了"每轮必提取"，而不是"有价值时才提取"。

大部分对话轮次其实不包含值得长期保存的信息。用户说"好的"、"继续"、"谢谢"——这些不需要提取记忆。但我们的系统每轮都调用 LLM 来分析，大部分时候分析的结果是"没有值得保存的信息"。

**为"没有结果"付费，是最贵的付费方式。**

### 误判

我们当时觉得，"自动提取"比"手动提取"好。用户不需要操心记忆的事，Agent 自己会处理。这在用户体验上是对的，在成本上是错的。

更好的方案是：降低提取频率，只在"可能有价值"的轮次才提取。比如：用户第一次提到偏好时提取，后续重复提到时不提取。用户提供了新的事实信息时提取，只是确认或追问时不提取。

### 修复成本

修复方案是加了一个"提取决策器"——先用一个轻量级的规则判断当前轮次是否值得提取，再决定是否调用 LLM。规则包括：对话长度超过阈值、包含关键词（"记住"、"偏好"、"以后"）、包含新的事实性信息。

这把记忆提取的频率大幅降低，后台 token 消耗也跟着降了下来。

### 教训

- 自动化不等于无成本。每一个"自动"操作都有 token 代价，要问自己：这个操作值得每次都做吗？
- 后台 LLM 调用是隐性成本。用户看到的是对话本身，看不到记忆提取、摘要生成、标题生成等后台调用的 token 消耗。这些隐性成本加起来不可忽视。
- "少调用 LLM"是一种设计原则。能用规则判断的，不要调用 LLM。能用缓存解决的，不要重新计算。能延迟执行的，不要立即执行。

## 错误五：过度设计——中间件链、能力协商为什么被删

### 发生了什么

AgentClaw 的工具系统，最初设计得非常"优雅"。

我们有一个中间件链（Middleware Chain）：每个工具调用都要经过一系列中间件——权限检查、参数校验、日志记录、超时控制、重试逻辑。每个中间件做一件事，可以自由组合。看起来很模块化，很可扩展。

我们还有能力协商（Capability Negotiation）：Agent 启动时，会和工具系统"协商"——你能做什么？你支持哪些参数？你的限制是什么？然后 Agent 根据协商结果决定怎么使用工具。

我们甚至有自定义校验器（Custom Validator）：每个工具可以定义自己的参数校验规则，不只是类型检查，还有业务逻辑校验——比如"搜索关键词不能超过 100 个字符"。

这些设计写出来很漂亮。代码结构清晰，职责分明，扩展性好。

然后我们把它们全删了。

### 根因

问题不是这些设计"错了"。在传统软件工程里，它们是合理的模式。中间件链在 Web 框架里广泛使用，能力协商在分布式系统里很常见，自定义校验器在表单处理里很有用。

但在 Agent 系统里，它们增加了复杂度，却没有带来相应的价值。

**中间件链**的问题是：Agent 的工具调用频率很高，每轮对话可能有 3-5 次工具调用。每次调用都要经过 5-6 个中间件，每个中间件都有自己的逻辑和异常处理。当中间件之间有交互时（比如权限检查依赖参数校验的结果），调试变得非常困难。一个工具调用失败了，你不知道是中间件的哪一层出了问题。

**能力协商**的问题是：Agent 的工具列表在运行时基本不变。你不需要每次都"协商"——工具能做什么，在工具定义里已经写清楚了。能力协商在动态系统（比如微服务之间）里有意义，在静态系统（Agent 和它的工具）里是过度设计。

**自定义校验器**的问题是：LLM 传入的参数本来就不靠谱。你花了很多精力写校验规则，但 LLM 可能传入你完全没想到的值。与其写复杂的校验框架，不如做清晰的类型转换和明确报错——把 `"False"` 转成 `false`，把 `"123"` 转成 `123`；无法转换的值就让工具直接失败，并把错误原因返回给 Agent。

### 误判

我们当时用了传统软件工程的思维来设计 Agent 系统。传统工程强调"可扩展性"、"模块化"、"关注点分离"。这些原则在大型系统里是对的。但在 Agent 系统里，**简单比优雅更重要**。

Agent 系统的复杂度不在代码结构，而在 LLM 的不确定性。LLM 会传入奇怪的参数、会忽略工具描述、会在你没想到的地方犯错。这些不确定性比代码结构的复杂度更难处理。增加代码层面的抽象，只会让调试更困难。

### 修复成本

删除这些设计花了一周。不是因为删除本身复杂，而是因为需要确保删除后系统仍然正常工作。中间件里的逻辑不能直接丢掉——有些是必要的（比如超时控制），需要移到工具执行层。能力协商里的信息不能丢掉——需要找到其他方式让 Agent 知道工具的限制。

最终的方案是：每个工具自己处理自己的逻辑。超时？工具自己设。参数校验？工具自己做。日志？框架统一记录。没有中间件，没有协商，没有自定义校验器。

代码量明显减少。调试时间减少了更多——因为每个工具都是自包含的，出了问题直接看工具代码就行。

### 教训

- **不要用传统软件工程的直觉来设计 Agent 系统。**Agent 系统的核心挑战是 LLM 的不确定性，不是代码结构的优雅性。增加抽象层不会减少不确定性，只会让问题更难定位。
- **能删的设计就是好设计。**如果一个模块删掉后系统仍然正常工作，说明这个模块本来就不需要。定期问自己：这个模块真的在创造价值吗？
- **简单是 Agent 框架的核心竞争力。**Agent 框架的用户——不管是人类还是 LLM——都更喜欢简单的接口。复杂的接口只会增加出错的概率。

## 错误六："先试试看"的代价——连续制造 fix

### 发生了什么

2026 年 3 月的某一周，AgentClaw 的 commit 历史看起来像这样：

```
fix: execute_code 支持 Python 语法检测
fix: execute_code 语法检测误报修复
fix: execute_code 支持更多 Python 模式
fix: execute_code 检测逻辑重构
fix: execute_code 最终修复
```

五个 fix commit，都跟同一个工具有关。

起因是：我们想让 `execute_code` 工具能检测到 Python 代码并给出友好提示。第一个实现用正则匹配 `import` 关键字。然后发现 `import` 在 JavaScript 里也有效（ES modules），误报了。改了匹配规则。然后发现有些 Python 代码不用 `import`（比如纯计算），漏报了。又改了。改来改去，越改越复杂。

同一周还有：

```
fix: max iterations 提示消息优化
fix: max iterations 提示消息再次优化
fix: max iterations 提示消息最终版
```

三个 fix，都是同一件事：当 Agent 达到最大迭代次数时，给用户显示什么消息。

还有：

```
fix: pdf-parse 在 Node.js 报 DOMMatrix 错误
fix: pdf-parse 入口路径修复
```

两个 fix，因为没有先查文档就直接用了 pdf-parse 库，结果 v2 有兼容性问题，降级到 v1 又发现入口文件不同。

这一周，我们提交了一堆 fix commit。其中大部分是"修了又修"——同一个问题改了两三次才真正修好。

### 根因

根因不是技术能力不足。根因是**工作方式有问题**。

我们采用了"先试试看"的方式：看到一个问题，直接动手改，改完测试，发现还有问题，再改。这在修简单 bug 时是有效的——改一行代码，测试通过，完事。但在处理复杂问题时，这种方式会导致"修复的修复的修复"。

"先试试看"的问题在于：**你没有在动手之前想清楚问题的全貌。**

`execute_code` 的问题，本质上是一个设计决策：你到底想让这个工具支持哪些语言？只支持 JavaScript？还是也支持 Python？如果支持 Python，需要什么额外的依赖？如果不支持，怎么给用户友好的提示？

这些决策应该在动手之前做好。但我们没有。我们看到"模型在 execute_code 里写 Python"这个现象，直接动手加了一个正则检测。然后发现正则太简单了，改。然后发现改了还是不够，又改。

每一轮"试试看"都浪费一个 commit。更重要的是，每一轮都可能引入新的 bug——正则匹配误报了合法的 JavaScript `import` 语句，这就是"修了一个问题，引入了另一个问题"。

### 误判

我们当时觉得，快速迭代比深入思考更高效。"先做出来看看效果，不行再改"——这在产品原型阶段是对的，但在框架开发阶段是错的。

框架的代码会被大量复用。一个有问题的工具描述，会影响所有使用这个工具的对话。一个有问题的溢出处理，会影响所有输出很大的场景。框架层面的 bug 影响面比应用层面大得多，修复成本也高得多。

**在框架层面，"先试试看"的成本是应用层面的 N 倍。**

### 修复成本

这些 fix commit 累计改动量远超预期。如果在一开始就深入研究、想清楚方案，可能只需要几个 commit 就够了。

时间成本更明显：一周的开发时间，大部分花在了"修了又修"上，而不是在新功能上。

### 教训

这条教训后来被写进了 AgentClaw 的开发纪律里：

> **一次做对，不要迭代试错。**先研究清楚（查文档、看源码、grep 现有模式），再动手。

具体来说：

- **第三方库先查文档再用。**不要猜 API，不要"试试看"。`node -e "console.log(Object.keys(require('pkg')))"` 一行命令就能看到库的接口。
- **设计决策先想清楚再动手。**涉及新机制时，先问自己：数据从哪来？是否需要硬编码？有没有现成抽象可复用？同类功能在代码库里怎么做的？
- **一个 bug 改完后，问自己：这是个例还是通病？**如果是通病，一次修一类，不要一个一个修。

## 错误七：修了 A 坏了 B——影响范围不检查

### 发生了什么

2026 年 3 月 22 日，一天之内连续四次"改了 A 坏了 B"。

第一次：修 `auto_send` 参数的布尔值处理。LLM 传入 `"False"`（字符串），JavaScript 里 `"False"` 是 truthy（非空字符串都是 true），所以自动发送被错误地启用了。修了 `"False"` 的情况。然后发现 LLM 还会传 `"True"`（大写 T），不匹配 `=== "true"`（小写 t）。修了一个，漏了一个。

第二次：删一个不再需要的 Skill，顺手删了关联的 `fetch.py` 文件。然后发现另一个功能依赖这个文件。

第三次：给 restart.ps1 加了日志重定向（`-RedirectStandardOutput`），想看 gateway 的启动日志。结果 PowerShell 的重定向会截断子进程的环境变量继承，导致 PATH 里没有 `C:\WINDOWS\system32`，`cmd.exe` 报 ENOENT。

第四次：删了一段业务逻辑，但忘了删关联的 CSS 类。死代码留在了样式表里。

四次事故，同一个根因：**改了代码后没有检查影响范围。**

### 根因

"改了 A 坏了 B"在任何软件项目里都会发生。但在 Agent 框架里，这个问题特别严重，因为 Agent 的组件之间耦合度很高。

Agent 的工具定义会影响 LLM 的行为。LLM 的行为会影响上下文的使用。上下文的使用会影响记忆的提取。记忆的提取会影响系统提示词的构建。系统提示词会影响工具的使用方式。

这是一个循环依赖的链条。改动链条上的任何一环，都可能影响其他环节。

传统的"改了 A 坏了 B"通常发生在代码层面——你改了一个函数的签名，调用方编译报错。编译器会告诉你。但在 Agent 系统里，很多"坏了 B"不会在编译时报错，而是在运行时表现为 LLM 的奇怪行为——更难发现，更难定位。

### 误判

我们当时觉得，"明显的修复直接改"是高效的工作方式。bug 修复、import 调整、类型修复——这些小改动不需要大张旗鼓地检查影响范围。

这个原则在大多数情况下是对的。但前提是：你对代码的依赖关系有完整的了解。如果你不知道 restart.ps1 的日志重定向会影响子进程的 PATH，你就不会去检查。如果你不知道 `auto_send` 的布尔值有大小写变体，你就不会去测试。

**"明显的修复"之所以看起来明显，是因为你只看到了问题的一半。**

### 修复成本

这四次事故的修复成本都不高——每次都是几行代码的改动。但它们消耗了大量的调试时间。每次"改了 A 坏了 B"，你都需要：发现 B 坏了、定位是 A 导致的、理解 A 和 B 的关系、找到正确的修复方式。这个过程可能需要几分钟到几小时。

更重要的是，它们消耗了信任。用户——也就是我自己——开始对 AI 写的代码不信任。每次改动后都要手动验证所有相关功能，效率反而降低了。

### 教训

这条教训也被写进了开发纪律：

> **改动后检查影响范围。**每次修改或删除代码后，grep 被改/被删的标识符，确认没有副作用。改了 A 不能坏了 B。

具体来说：

- 改了函数/变量/参数的名字或签名 → grep 旧名字，确认没有残留引用。
- 删了文件或目录 → grep 文件名，确认没有其他代码依赖它。
- 改了启动方式或环境变量 → 想一遍所有子进程是否受影响。
- 改了类型校验逻辑 → 列举 LLM 可能传的所有变体（`true`/`false`/`"True"`/`"False"`/`"true"`/`"false"`/`1`/`0`），一并处理。

## 错误八：依赖 LLM 自觉的设计必定失败

### 发生了什么

AgentClaw 的任务系统有一个"待办提醒"功能。当 Agent 有多个子任务时，系统会注入一条提示词，提醒 Agent "别忘了更新待办列表"。

这个设计的初衷是：让 Agent 自己管理任务进度。每完成一个子任务，Agent 应该主动调用 `update_todo` 工具，标记任务完成。

结果：Agent 不听。

我们在提示词里写了"每完成一步请更新待办"。Agent 偶尔会遵守，大部分时候不会。它更倾向于直接做下一个任务，而不是先更新待办列表。

我们把提示词改得更强硬："**必须**在每步完成后更新待办！"Agent 还是不听。我们加了感叹号、加了粗体、加了"IMPORTANT"。Agent 偶尔遵守，大部分时候还是不会。

### 根因

大语言模型是概率性的。它不是在"遵守规则"，而是在"根据上下文生成最可能的下一个 token"。当上下文里有更重要的信息——比如用户的具体需求、工具的返回结果——待办提醒的优先级就降低了。

你不能通过加感叹号来提高一条指令的优先级。模型不看感叹号。它看的是这条指令在上下文中的位置、与其他信息的关系、以及它在训练数据中见过的类似模式。

**如果你的设计依赖 LLM "自觉"做某件事，这个设计就是不可靠的。**

### 误判

我们当时觉得，提示词是控制 LLM 行为的可靠方式。"写清楚了，LLM 就应该遵守。"

这在大多数情况下是对的——LLM 确实会遵守明确的指令。但在"优先级竞争"的场景下，提示词的效力会下降。当多条指令竞争 LLM 的注意力时，更具体、更直接、更贴近当前任务的指令会胜出。"更新待办列表"这种元任务（meta-task），在和"回答用户的问题"这种直接任务竞争时，几乎总是输。

### 修复成本

最终的修复方案是：不要依赖 LLM 自觉，框架直接做。当 Agent 完成一个子任务后，框架自动把对应待办标记为完成，不需要 Agent 主动调用工具。

这个方案更简单、更可靠、更省 token。Agent 不需要花精力在"管理待办"上，可以专注于"做任务"。

### 教训

- **需要确定性行为的地方，框架直接做，不要委托给 LLM。**LLM 擅长推理和判断，不擅长遵守琐碎的规则。把琐碎的规则交给框架，让 LLM 专注于它擅长的事。
- **提示词不是万能的。**它能引导 LLM 的行为，但不能保证 LLM 的行为。如果你需要 100% 确定某件事会发生，不要用提示词，用代码。
- **减少 LLM 的负担。**每多一条提示词指令，LLM 的注意力就分散一点。把不必要的指令删掉，把必要的指令交给框架执行，让 LLM 专注于核心任务。

## 错误九：信息不一致——提示词说有，工具列表说没有

### 发生了什么

有一次，Agent 反复调用一个叫 `execute_code` 的工具。每次都报"工具不存在"。然后它又试了一次。又报错。又试。

我们检查了工具列表——`execute_code` 确实存在。但这个 Agent 的工具白名单里没有它。

然后我们检查了系统提示词——系统提示词里提到了 `execute_code`，告诉 Agent "你可以用 execute_code 执行代码"。

问题找到了：系统提示词说 Agent 可以用 `execute_code`，但工具白名单里没有这个工具。Agent 从系统提示词"学到"了这个工具名，去调用，发现不存在，再试，再失败。

**框架给 LLM 的信息自相矛盾了。**

### 根因

AgentClaw 的系统提示词是外置的 Markdown 文件，和代码分开维护。工具列表是在代码里动态生成的。两者之间没有强制的一致性检查。

当我们在系统提示词里提到了某个工具，但工具列表里没有包含它时，信息就矛盾了。LLM 会相信系统提示词（因为它在上下文中的优先级最高），去调用那个不存在的工具。

### 误判

我们当时觉得，系统提示词和工具列表是两个独立的系统，各自维护就好。系统提示词是给 LLM 看的"说明书"，工具列表是给框架用的"配置"。两者不需要强关联。

但 LLM 不区分"说明书"和"配置"。它看到的所有信息都是平等的。如果说明书说"你可以用 A"，但配置里没有 A，LLM 会认为是配置错了，而不是说明书错了。它会坚持尝试调用 A。

### 修复成本

修复方案有两层：

第一层：加了一个 `pruneSystemPromptForTools` 函数，在构建系统提示词时自动扫描工具列表，删除提示词中提到但不在列表里的工具名。

第二层：在系统提示词里加了一段"工具边界声明"，告诉 Agent "你只能使用以下工具列表中的工具，其他工具不存在"。

两层防护，确保信息一致。

### 教训

- **框架给 LLM 的所有信息必须一致。**系统提示词、工具定义、历史对话——这些信息会被 LLM 综合考虑。如果有矛盾，LLM 的行为就是不可预测的。
- **不要假设 LLM 会忽略过时的信息。**LLM 不会"知道"哪些信息是最新的、哪些是过时的。它看到的所有信息都同等对待。
- **信息一致性检查应该是自动化的。**不要依赖人工检查——人会忘，会漏。用代码自动检查提示词和工具列表的一致性。

## 错误十：每轮累积的副作用

### 发生了什么

AgentClaw 的待办提醒系统有一个"nag 模式"。当 Agent 有未完成的待办任务时，系统会在每轮对话的上下文里注入一条提醒："你还有未完成的任务，请继续执行。"

这个提醒用 `runtimeHints.push()` 添加到上下文里。

然后我们发现，10 轮对话之后，上下文里有 6 条重复的提醒。

原因是：`runtimeHints.push()` 是在每轮循环开始时执行的，但它没有清除之前添加的提醒。每一轮都 push 一条新的，旧的还在。6 轮之后就有 6 条一模一样的提醒。

这些重复提醒不只是浪费 token。它们会误导 LLM——LLM 看到 6 条"你还有未完成的任务"，可能会认为有 6 个不同的任务，而不是 1 个任务被提醒了 6 次。

### 根因

这是一个典型的"循环累积副作用"问题。在循环中执行的操作，如果不做去重或清理，就会累积。

这个问题在传统编程里也存在，但在 Agent 系统里更隐蔽。因为 Agent 的上下文是"追加式"的——新信息不断加入，旧信息不会自动消失。你加了什么，就永远在那里，直到上下文被压缩或截断。

### 修复成本

修复很简单：用精确匹配代替范围匹配（`=== 4` 代替 `> 3`），加一个 flag 防止重复注入。

但发现这个问题花了很长时间。因为重复提醒不会导致错误——它只是让 LLM 的行为变得奇怪。你不会看到报错，只会觉得"Agent 今天怎么老是提待办的事"。

### 教训

- **在循环中执行的操作，问自己：这个操作在下一轮还会执行吗？会不会累积？**
- **Agent 的上下文是追加式的。**你加了什么，就永远在那里。每一条注入的信息都要考虑：它会在上下文里待多久？会不会重复？
- **副作用的累积比副作用本身更危险。**单条提醒是有用的，6 条重复提醒是有害的。

## 错误十一：硬编码应该动态获取的值

### 发生了什么

AgentClaw 的 agent-loop.ts 中，曾经有一行代码：

```typescript
const ALL_TOOL_NAMES = new Set([
  "shell", "file_read", "file_write", "file_edit",
  "glob", "grep", "ask_user", "web_fetch", "web_search",
  // ... 总共 20 个工具名
]);
```

这是一个硬编码的工具名列表。每次新增工具，都要手动在这个列表里加一行。

然后有人新增了一个工具，忘了加到这个列表里。Agent 不知道这个新工具的存在，即使工具注册表里有它。

### 根因

硬编码是编程中最常见的坏习惯之一。它的诱惑在于：写起来简单，看起来清晰。20 个工具名列在那里，一目了然。

但硬编码意味着**维护债务**。每新增一个工具，你都要记得同步更新这个列表。如果忘了，就会出 bug。而人总是会忘的。

更根本的问题是：这个信息已经存在于工具注册表（toolRegistry）里了。工具注册表知道所有已注册的工具。从注册表里获取工具名，比硬编码一个列表更可靠。

### 误判

我们当时觉得，硬编码的列表更"可控"。你能看到所有工具名，能确保没有意外的工具混进来。

但"可控"的代价是"容易出错"。两个事实源（硬编码列表和工具注册表）意味着它们可能会不一致。只要有两个事实源，就一定会出 bug。

### 修复成本

修复很简单：`new Set(this.toolRegistry.list().map(t => t.name))`，从工具注册表动态获取。一行代码。

但这个教训的价值在于：它揭示了一个更深层的原则——**唯一事实源**。同一件事只在一个地方定义。如果有两个地方定义了同一件事，它们就一定会不一致。

### 教训

- **如果数据源已经存在，就从源头获取。**不要复制一份出来。
- **硬编码 = 维护债务。**今天省的时间，明天会以 bug 的形式还回来。
- **唯一事实源。**同一件事只在一个地方定义。这是减少 bug 最有效的原则之一。

## 错误十二：web_fetch 不能下载二进制文件

### 发生了什么

用户让 Agent 下载一个 PDF 文件。Agent 用了 `web_fetch` 工具。

`web_fetch` 的设计目标是抓取网页内容，把 HTML 转成 Markdown。它不是文件下载器。

一个 22.6MB 的 PDF 经过 `web_fetch` 的 HTML→Markdown 转换后，变成了一个 41MB 的损坏文件。内容完全不可读。

### 根因

`web_fetch` 工具的描述里写的是"获取网页内容"。但"网页"这个词不够精确——模型可能认为 PDF 也是一个"网页"（毕竟它可以通过 HTTP 访问）。

而且模型在训练数据里见过大量"用 requests 库下载 PDF"的代码。它"知道" HTTP 可以获取 PDF 文件。所以它会用 `web_fetch` 来下载 PDF，即使这个工具不是为此设计的。

### 修复成本

修复方案是在系统提示词里加了一条规则："PDF 等二进制文件请用 `bash curl -o` 下载，不要用 web_fetch。"

同时在 `web_fetch` 的工具描述里加了更明确的说明："此工具用于获取网页文本内容，不支持二进制文件（PDF、图片、压缩包等）。"

两层防护：描述里写明限制，提示词里给出替代方案。

### 教训

- **工具描述的边界要明确。**不只说"它能做什么"，还要说"它不能做什么"。
- **给 Agent 提供替代方案。**告诉它"不能用 A"还不够，要告诉它"应该用 B"。
- **模型的先验知识是双刃剑。**它让模型能处理新场景，也让模型在不该用某个工具时用了那个工具。

## 总结：从错误中提取的原则

回头看这些错误，有几个反复出现的模式：

**原则一：框架对 LLM 的表现负全责。**LLM 会犯错，这是它的本性。框架的职责是减少 LLM 犯错的机会，在 LLM 犯错时兜底，而不是期望 LLM 自己做对。

**原则二：信息不对称是框架的责任。**框架知道的信息（工具列表、系统配置、边界条件），必须在架构层面传达给 LLM。不能依赖 LLM 自己去"发现"。

**原则三：简单是 Agent 框架的核心竞争力。**复杂的抽象层不会减少系统的复杂度，只会把复杂度从一个地方移到另一个地方。Agent 系统的真正复杂度在 LLM 的不确定性，不在代码结构。

**原则四：唯一事实源。**同一件事只在一个地方定义。两个事实源就一定会出 bug。

**原则五：一次做对，不要迭代试错。**在框架层面，"先试试看"的成本是应用层面的 N 倍。先研究清楚，再动手。

**原则六：改动后检查影响范围。**grep 被改/被删的标识符，确认没有副作用。改了 A 不能坏了 B。

这六条原则，不是从教科书里抄来的。它们是从真实的错误中提取的，每一条背后都有一个具体的 bug、一次痛苦的调试、一个被浪费的周末。

## 动手试一试

**任务 1：复盘你最近一次 AI 编程失败。**

如果你用过 Cursor、Claude Code 或其他 AI 编程工具，回忆一次 AI 写的代码有 bug 的经历。回答以下问题：

- 根因是什么？是 AI 理解错了需求，还是实现有 bug，还是设计本身有问题？
- 你当时的误判是什么？你为什么觉得 AI 的方案是对的？
- 修复花了多少时间？如果一开始就发现问题，能省多少时间？
- 下次怎么预防？

把这个复盘写下来。不需要很长，200 字就够。关键是诚实——对自己诚实比对自己苛刻更有用。

**任务 2：给你的项目做一次"两个事实源"检查。**

如果你有自己的项目（不管是代码项目还是其他类型的项目），检查一下：有没有同一件事在两个地方定义的情况？

比如：配置文件里的端口号和代码里的端口号是不是同一个值？API 文档里的参数名和代码里的参数名是不是一致？README 里的功能列表和实际实现的功能是不是匹配？

找到一个不一致的地方，修掉它。

**任务 3：设计一个"溢出防护"方案。**

如果你在造一个 Agent，想一想：当工具输出太大时，你的系统会怎么处理？

- 有没有输出上限？
- 超过上限后给模型看什么？
- 提示词是引导模型读完整内容，还是引导模型利用预览？

用三句话描述你的方案。

## 本章小结

1. **工具描述比工具实现更影响 Agent 行为。**描述的质量决定了 Agent 会不会正确使用工具。
2. **没有溢出处理的工具是炸弹。**一个 `cat` 命令就能炸掉整个上下文。
3. **子代理必须共享全局预算。**独立预算在递归场景下会无限膨胀。
4. **后台 LLM 调用是隐性成本。**记忆提取、摘要生成等操作的 token 消耗不可忽视，但常常被忽略。
5. **过度设计在 Agent 系统里是负担。**中间件链、能力协商这些传统模式，在 Agent 系统里增加了复杂度却没有带来价值。
6. **"先试试看"在框架层面代价很高。**先研究清楚再动手，一次做对。
7. **信息不一致是 bug 的温床。**唯一事实源是最有效的防 bug 原则。
8. **依赖 LLM 自觉的设计必定失败。**需要确定性行为的地方，框架直接做。
