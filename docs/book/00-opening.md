# 开篇：为什么现在可以从零造 Agent

---

## 我不是职业程序员，但我想把它造出来

2026 年春节，我决定造一个 AI Agent 框架。

这句话听起来有点轻巧。真实情况是，我当时并没有一个完整工程团队坐在旁边，也不是那种每天写八小时代码的职业程序员。

我从 2000 年开始做软件和网站，后来长期做互联网教育和医疗教育产品。过去这些年，我做过题库、App、小程序、后台系统，也做过医院培训考核这类需要长期运营和交付的业务。它们不总是看起来很酷，但它们训练了我一件事：一个系统最后不是给技术演示看的，而是给真实用户每天使用的。

所以我对软件并不陌生。我能看代码，能判断方案，能和工程师讨论架构，也写过不少脚本和小工具。但我也很清楚，自己不是从科班程序员路线成长起来的职业工程师。过去如果要从零搭一个完整、可维护、能上线的工程级项目，我通常会依赖团队。

但那一年，AI 编程工具开始变得不一样了。

Cursor、Claude Code、Codex——这些工具不再只是帮你补全几行代码。你用自然语言告诉它"我想做一个能搜索网页的工具"，它会帮你写出完整的实现。你告诉它"这段代码有 bug"，它会帮你定位问题、修复、测试。你告诉它"把这个功能拆成三个模块"，它会帮你规划架构、创建文件、写好接口。

我开始想：如果 AI 能承担大量实现工作，那"职业程序员级别的编码能力"还是不是造软件的唯一门槛？

答案是：**不再是唯一门槛了。**

门槛变成了另一件事——你能不能把目标定义清楚，把问题拆开，把边界想明白，把结果验收住。

这正是我过去二十多年做产品、内容、运营和交付时一直在练的能力。

## 这不是一本编程书

先说清楚这本书**不是**什么。

它不是一本编程教程。你不会在这本书里学到 TypeScript 的语法、React 的用法、或者怎么配置 Webpack。如果你想学编程，有无数更好的选择。

它不是一本 AI 科普书。你不会在这本书里看到 Transformer 的数学推导、注意力机制的原理、或者怎么微调一个大语言模型。如果你想了解 AI 的底层原理，有无数更好的选择。

它不是一本 prompt 技巧书。你不会在这本书里找到"100 个让 ChatGPT 更好用的 prompt 模板"。那些东西在网上搜一搜就有。

这本书是关于**造东西**的。

具体来说，是关于造一个 AI Agent——一个能自己做事、不只是回答问题的 AI 系统。

你会看到一个真实项目从零到一的完整过程：每个设计决策是怎么做的，每个方案是怎么选的，每个错误是怎么犯的、又是怎么修的。你不必是职业程序员，也能理解这些决策——因为很多关键问题本质上是判断题，不只是编程题。

如果你读完这本书想自己动手造一个 Agent，你有两个选择：

**选择一：用 AI 编程工具。**打开 Cursor、Claude Code 或 Codex，告诉它你想做什么，它帮你写代码。你负责想清楚需求、做出设计判断、验证结果对不对。代码不是完全不用管，而是不再需要你逐行从零写完。

**选择二：找一个程序员搭档。**你负责产品设计和架构决策，他负责实现。这本书让你成为一个更好的"甲方"——你知道该提什么需求、该问什么问题、该在哪些地方坚持。

两种方式都行。重点是：**你不需要先成为职业程序员才能造 Agent。你需要学会想清楚问题，并对结果负责。**

## AI 时代的新分工

造一个 Agent，你需要回答这些问题：

- Agent 应该能做什么？（工具设计）
- 什么时候该停下来？（循环控制）
- 什么信息值得保留？（上下文管理）
- 出错了怎么办？（错误处理）
- 怎么防止被恶意利用？（安全设计）
- 怎么让用户真的能用？（生产部署）

这些问题不一定要求你亲手写完每一行代码。但每一个都需要你做出判断。

"搜索工具应该有超时吗？"——需要你决定。超时多久？10 秒还是 30 秒？太短会打断正常搜索，太长会让用户等得不耐烦。这个判断不需要你写一行代码，但你需要理解它为什么重要。

"Agent 调用了同一个失败命令三次，应该怎么办？"——需要你决定。直接停掉？还是给它换个方式试试？停掉太激进，放任会浪费 token 和时间。你需要在两者之间找到平衡。

"Agent 的记忆应该保存在哪里？数据库？文件？内存？"——需要你决定。每种方案的代价不同。数据库可靠但需要维护，文件简单但容易损坏，内存快但重启就没了。你需要根据你的场景选择最合适的方案。

"如果有人在消息里藏了一段恶意指令，让 Agent 去读取服务器上的密钥文件，怎么防？"——需要你决定。安全不是事后补的装饰品，是设计时就要想清楚的架构决策。

这些判断加在一起，就是"造一个 Agent"的真正工作量。代码是重要的一步，但不再是唯一的一步；大量实现工作越来越可以交给 AI 辅助完成。

**AI 负责执行大量实现工作，你负责目标、拆解、边界和验收。**

这就是这本书要教你的事。不是 TypeScript 语法，而是怎么想清楚一个 Agent 系统该怎么做决策、怎么处理边界情况、怎么在"能用"和"好用"之间找到平衡。

## 我是怎么做到的

造 AgentClaw 的过程不是一帆风顺的。

第一个版本，我让 AI 帮我接上了一个大语言模型的 API，能聊天了。我觉得挺厉害。然后我给它加了一个搜索工具，它能帮我查东西了。我觉得更厉害了。

然后它开始失控。

有一次，我让它帮我"调查某个行业"。它开始搜索，搜索完觉得信息不够，又搜了一遍。搜完还是觉得不够，又搜了一遍。我看着它连续搜了十五次，消耗了大量 token，最后告诉我"信息太多了，无法整理"。

还有一次，它调用了一个不存在的工具名。报错。然后它又调了一次。又报错。又调了一次。连续调了八次同样的错误调用，每次都在浪费一次 LLM 调用。

还有一次更离谱的：它生成了一段坏掉的 JSON 作为工具参数，工具解析失败。这个失败的调用和错误信息留在了对话历史里。下一轮，它看到了这个错误，试图"修复"——但修出来还是坏 JSON。再下一轮，又修。三轮之后，整个对话上下文被垃圾信息填满了。

还有一次，我让它帮我重构一段代码。它改了一个文件，然后去改另一个文件。改到一半发现第一个文件改错了，又回去改第一个。改完发现第二个文件的改动跟第一个冲突了。来回改了六轮，两个文件都被改得面目全非，最后我不得不手动回滚。

还有一次更让人崩溃的：它在对话里记住了我的一个偏好——"用 Tab 缩进不用空格"。但这条记忆被写进了系统提示词，每次新对话都会加载。后来我改了主意，想用空格了。但 Agent 还是坚持用 Tab，因为那条记忆还在。我告诉它"改用空格"，它说好的，下次还是用 Tab。记忆比对话更持久——这本来是优点，但当记忆本身是错的时候，它就变成了一个很难清除的 bug。

这些经历让我意识到一件事：**让 AI 跑起来不难，让它稳定地做事才是真正的工程。**

循环需要刹车。工具需要护栏。上下文需要管理。错误需要分类。安全需要防线。记忆需要维护。成本需要控制。

这些才是 Agent 框架的核心问题，也是这本书要讲的内容。

## 我真正做的事

你可能会问：如果不是职业程序员，那我到底做了什么？

答案是：**技术理解 + 产品判断 + AI 编程，相当于一个小型团队。**

造 AgentClaw 的过程中，我的角色更像产品经理、架构判断者和验收负责人。我决定框架该有什么功能、优先做什么、不做什么、怎么设计接口、哪些风险不能接受。AI 承担了大量程序员工作：写代码、修 bug、做重构、补测试。

分工是这样的：

**我负责的事**：
- 确定需求：Agent 需要哪些工具？记忆怎么设计？安全怎么保障？
- 做架构判断：搜索超时设多少秒？上下文压缩到什么程度？错误重试几次？
- 验证结果：功能跑通了吗？符合预期吗？有没有副作用？
- 排优先级：先做记忆还是先做安全？先支持 Telegram 还是先支持钉钉？

**AI 负责的事**：
- 写代码：我说"给我实现一个工具注册表"，它写出完整的实现。
- 修 bug：我说"搜索工具超时了没有重试"，它定位问题并修复。
- 重构：我说"把渠道管理的重复代码提取出来"，它帮我整理。
- 写测试：我说"给记忆提取功能写测试"，它写出测试用例。

这个过程不是一帆风顺的。AI 写的代码经常有 bug。有时候它理解错了我的需求，写出来的东西完全不是我想要的。有时候它过度设计，把简单的事情搞复杂了。我需要不断地检查、纠正、引导。

但关键是：**我不需要自己从零写完所有代码。**我需要的是知道什么是对的、什么是错的，知道一个设计为什么成立，知道一个结果怎么验收。这是二十多年软件和互联网经验给我的判断力。

从第一个版本到能跑起来的骨架，只花了几天；从能跑到能上线，又经历了几周高强度迭代。如果我用过去的方式从零搭团队，周期和成本都会完全不同。

这不是因为我突然变成了职业程序员。是因为 AI 编程工具真的变了。它把大量实现工作的门槛拉低了，让"想清楚"变成了最重要的能力。

而"想清楚"不是玄学。它包括：知道用户要什么，知道一个功能的边界在哪里，知道什么结果算完成，知道什么时候该停下来重做。这些能力不是公司简介里的一串数字能说明的，只能在一次次真实交付里练出来。

## 一个具体的例子：搜索工具的诞生

让我用一个真实场景展示这种分工是怎么运作的。

2026 年春节，我在设计 AgentClaw 的搜索工具。Agent 需要能搜索网页，这是最基本的能力之一。听起来很简单——调一个搜索 API 不就行了？但实际操作起来，每一个细节都是判断题。

**第一步：我决定"要什么"，AI 决定"怎么要"。**

我告诉 AI："给 Agent 加一个搜索工具，能搜网页、返回结果。"AI 问我要用哪个搜索 API。我不知道有哪些选项，就让它列出来。它给了我三个：Google Custom Search、Bing Search API、SearXNG（一个开源自托管方案）。

这一步是**我做判断**：Google 和 Bing 需要付费、有配额限制，但结果质量好；SearXNG 免费、无限制，但需要自己部署。我选了 SearXNG 作为默认，同时支持 Bing 作为备选。理由不是技术判断——是成本判断。个人项目，能省则省。

**第二步：我决定"边界在哪"，AI 决定"怎么实现"。**

搜索工具有几个边界需要定义：

- 搜多少条结果？我决定默认 5 条，最多 20 条。理由：Agent 的上下文窗口有限，塞太多搜索结果会挤掉其他有用信息。
- 超时多久？我决定 10 秒。理由：用户等搜索结果的耐心大概就这么多，超过 10 秒体验就崩了。
- 搜不到结果怎么办？我决定返回一个明确的"无结果"消息，而不是空数组。理由：空数组会让 LLM 误以为工具调用失败，触发重试。
- 搜索结果太多怎么处理？我决定只返回标题、URL 和摘要，不返回完整网页内容。理由：完整内容太长，Agent 需要的时候可以再用另一个工具去读。

这些判断没有一个是"正确答案"。10 秒超时还是 15 秒？5 条结果还是 10 条？每一个都是权衡。但这些权衡不需要你写代码——你需要的是理解"为什么要这样权衡"。

**第三步：AI 实现，我验证。**

AI 写完了搜索工具的代码。我没有逐行读代码——我看的是行为。我让它搜"2026 年春节档电影票房"，看它返回什么。返回了 5 条结果，格式正确，数据看起来合理。我让它搜一个不存在的东西，看它怎么处理。它返回了"无结果"消息，没有报错。我让它在一个网络不通的环境里搜，看超时是否生效。10 秒后返回了超时错误。

这三个测试就够了。我不需要理解代码怎么写的，我只需要验证行为是否符合我的预期。

**第四步：出了 bug，我描述现象，AI 修。**

上线后发现一个问题：搜索中文关键词时，偶尔会返回乱码。我描述了现象（"搜中文有时候乱码"），AI 定位到是编码问题——SearXNG 返回的结果有时是 GBK 编码，代码默认按 UTF-8 解析。它加了编码检测逻辑，问题解决了。

整个过程，我做了四个判断（选哪个 API、设什么边界、怎么测试、bug 的现象是什么），AI 做了所有实现（写代码、调 API、修 bug）。这就是"产品经理 + AI"的分工模式。你不需要知道怎么写代码，但你需要知道怎么定义"好"和"不好"。

## 你将造出什么

读完这本书，你会理解一个 Agent 框架的完整结构：

它的结构可以拆成四层：

1. 用户消息进入系统。
2. Agent Loop 接管任务，反复执行“思考（LLM）→ 行动（工具）→ 观察（结果）”。
3. 安全层、记忆层和容错层在循环外提供保护：沙箱、黑名单、上下文压缩、长期记忆、重试、降级和熔断。
4. 渠道层把结果发回 Telegram、微信、钉钉或 Web，用户收到回复。

**大脑**（第 1-2 章）：Agent 的核心循环，思考 → 行动 → 观察。你会理解循环怎么跑，以及为什么"控制循环"比"启动循环"更重要。

**手脚**（第 3-5 章）：工具系统。怎么给 Agent 装上能力，怎么设计工具的接口，怎么处理工具出错，怎么让 Agent 同时做几件事而不互相破坏。

**工作记忆**（第 6 章）：上下文窗口管理。AI 的"工作台"就那么大，怎么在有限空间里塞进最多有用信息？什么时候该压缩，什么时候该保留？

**长期记忆**（第 7 章）：跨会话的记忆存储和检索。怎么让 Agent 记住你是谁、你喜欢什么、上次发生了什么？记忆放哪里、怎么找、怎么防过时？

**成本控制**（第 8 章）：token 经济学。工具输出、系统提示词、后台 LLM 调用会怎样膨胀？怎样用压缩、缓存和审计把浪费降下来？最贵的浪费往往是你看不见的。

**容错**（第 9 章）：LLM 犯错时的处理策略。API 返回错误怎么办？AI 卡住了怎么办？AI 输出了垃圾怎么办？七种错误，七种应对。

**安全**（第 10 章）：当 Agent 有了 shell 和文件系统，怎么防止被滥用？六类威胁，逐层防御，以及诚实的安全评估——哪些风险能降低，哪些风险仍然存在。

**感知**（第 11 章）：浏览器自动化。怎么让 AI 能"看"网页、"点"按钮？三种方式，一个答案。

**触达**（第 12 章）：多渠道接入。怎么让同一个 Agent 同时在 Telegram、微信、钉钉上运行？一个大脑，七张嘴。

**上线**（第 13 章）：从"能跑"到"能用"的生产化。容器化、备份、优雅关闭、健康检查、日志监控——七件无聊但救命的事。

你不需要一口气读完。每一章都可以独立读。但按顺序读效果最好，因为后面的章节会引用前面的概念。

全书九个部分、十五章，大致按这个顺序展开：

| 部分 | 章节 | 核心问题 |
|------|------|----------|
| 一、Agent 到底是什么 | 第 1-2 章 | 什么是 Agent？核心循环怎么跑？ |
| 二、给 Agent 装上手脚 | 第 3-5 章 | 工具怎么设计？出错怎么办？能并行吗？ |
| 三、教 Agent 记住事情 | 第 6-8 章 | 上下文怎么管？记忆怎么存？成本怎么控？ |
| 四、当事情出错 | 第 9 章 | LLM 犯错时系统怎么办？ |
| 五、安全 | 第 10 章 | 怎么防止 Agent 被滥用？ |
| 六、让 Agent 看见世界 | 第 11 章 | 怎么让 Agent 用浏览器？ |
| 七、走进用户的生活 | 第 12 章 | 怎么在多个平台同时运行？ |
| 八、上线给别人用 | 第 13 章 | 从"能跑"到"能用"要做什么？ |
| 九、复盘与未来 | 第 14-15 章 | 踩了哪些坑？未来会怎样？ |

每个部分解决一类问题。前两个部分是基础（Agent 是什么、怎么给它能力），中间四个部分是核心工程（记忆、容错、安全、感知），最后三个部分是落地（多渠道、生产化、反思）。

## 这本书怎么读

**如果你是产品经理或创业者**：重点读第 1 章（Agent 是什么）、第 7 章（记忆）、第 12 章（多渠道）、第 13 章（上线）。理解 Agent 能做什么、不能做什么、上线要花多少精力，比理解技术细节更重要。你可以跳过代码片段，只读文字解释和图表。

**如果你是独立开发者**：全书都读。你可能是最接近"一个人造一个完整 Agent"的读者。每一章的"动手试一试"都值得做一遍。

**如果你是工程师**：重点读第 3-5 章（工具系统）、第 6 章（上下文）、第 9-10 章（容错和安全）。这些章节的工程深度最高，有很多你在其他地方看不到的实战经验。你可以对比 AgentClaw 的源码来读。

**如果你只是好奇**：读第 1 章和第 15 章就够了。第 1 章让你理解 Agent 是什么，第 15 章让你看到 Agent 的未来。两章加起来不到一小时。

每一章末尾都有一个"动手试一试"栏目。你可以用 Cursor、Claude Code 或 Codex，把你想做的事情用自然语言告诉它，让它帮你实现。重点不是炫技式地手写代码，而是你能不能把问题描述清楚，并验证结果是否正确。

如果你手边没有 AI 编程工具，也没关系。每一章的"动手试一试"都有替代方案：用纸笔画流程图、用表格做对比分析、用自己的话复述核心概念。重点是动脑，不是动手。

## 关于 AgentClaw

这本书以 AgentClaw（爬爬虾）为案例。它是一个开源的 AI Agent 框架，用 TypeScript 写成，部署在生产环境，服务于 Telegram、WhatsApp、钉钉、飞书、企业微信和 Web 等多个渠道。

但 AgentClaw 不是这本书的主角。**你才是。**

AgentClaw 是一个真实的、有缺陷的、在不断改进的框架。它踩过的坑、删掉的设计、保留的妥协，比任何"最佳实践"都更有参考价值。你从它的经历中学到的判断力，可以用在任何你自己的 Agent 项目上——不管你用什么语言、什么框架、什么模型。

代码是开源的：`github.com/vorojar/AgentClaw`。

每一章会标注对应的源码文件。如果你想深入某个实现细节，可以直接去看代码。看书本身不要求你逐行读懂所有实现，但你应该愿意理解代码背后的设计判断。

## 一个请求

如果你读完这本书，造出了自己的 Agent——不管多简单、多粗糙——请告诉我。

不是因为我需要知道，而是因为这件事本身就意味着：**AI 时代，造东西的门槛真的变了。**

十年前，造一个软件通常需要一个团队。五年前，一个强全栈开发者可以搞定很多事。现在，一个半技术出身、懂产品和业务的人，用 AI 帮忙，也能把一个完整框架搭起来。

这不是因为这个人突然变成了工程专家。是因为工具变好了。AI 编程工具把大量实现难度降了下来，让"想清楚"的能力浮到了水面上。而"想清楚"这件事，依赖的是经验、判断力和对用户的理解。
