# 第 8 章 Token 经济学：别把同一段上下文发两遍

---

## 第二天就看到了用量曲线

AgentClaw 的第二天——2026 年 2 月 21 日——我就开始优化 token 了。

不是因为我有远见。是因为我打开 LLM 用量面板，看到了前一天的输入 token 曲线。2 月 20 日，项目启动的第一天，我一个人在测试，跑了大概二十几个对话，每个对话 5-10 轮。对话不多，但输入 token 增长得非常快。

真正吓人的不是某一笔成本，而是增长方式：每多一轮对话，前面所有工具输出都要再次发送。一个人测试时已经明显膨胀，如果上线后有更多用户同时使用，这个曲线会失控。

这个用量曲线逼着我从第二天就开始关注 token。不是作为一个"优化项目"，而是作为一个上线前必须解决的工程问题：**如果 token 膨胀控制不住，这个产品就没法稳定运行。**

后续迭代里，我做了几十次优化。有些有用，有些没用，有些反而让事情变糟了。这一章讲的是这些尝试——不是教科书式的"四层压缩管道"，而是一个真实的、混乱的、反复试错的过程。

---

## 先搞清楚 token 花在哪

在动手优化之前，我需要知道 token 花在哪。AgentClaw 很早就加了 token 统计功能（`2026-02-21 14:51` commit: `feat: Implement token usage logging`），因为不统计就只能瞎猜。

一次 Agent 的 LLM 调用，token 来自这几个部分：

- 系统提示词：约 350-700 token
- 工具定义（20 个工具的 JSON Schema）：约 1,500-2,000 token
- 对话历史（之前所有轮次的输入输出）：**变量，但增长很快**
- 当前用户输入：约 50-200 token

第一次调用，总共 2,000-3,000 token，很合理。问题出在第五次、第十次调用——对话历史在累积，每一轮的工具输出（可能有几千 token）都被完整保留在历史里，每次调用都重新发送。

一个 `file_read` 读了 500 行源文件，输出约 8,000 token。这 8,000 token 在后续每次调用中都会被重新发送。三次文件读取就是 24,000 token 的"历史包袱"。10 轮下来，同样的内容被发了 10 遍。

**你最贵的 token，是你发了两遍的 token。**

## 第一次尝试：截断工具输出

最早的一批 commit 里就包含了 shell 输出截断。思路很简单：shell 命令的输出可能很长（比如 `ls -la` 列了 200 个文件，或者 `cat` 了一个大文件），超过一定长度就截断。

最初的实现很粗暴——直接砍到前 N 个字符。效果立竿见影，但马上出了问题。

**问题一：砍掉了错误信息。** Agent 执行 `npm run build`，编译输出有 200 行，真正的错误在第 180 行。截断后 Agent 只看到前面的正常日志，不知道出了什么错。它又执行了一遍 `npm run build`——同样的输出，同样的截断，同样的看不到错误。循环了三次。

这个问题花了我一天才意识到根因：**截断不是压缩，截断是丢信息。** 关键信息不一定在开头。

**问题二：砍得太短导致 Agent 困惑。** 截断后只剩前 50 个字符，Agent 看到的是 `total 48\ndrwxr-xr-x  12 user  staff   384`——它不知道这是 `ls` 的输出还是什么别的东西，开始瞎猜。

修复方案迭代了好几次：先改成长度 + 保留首尾行，后来改成提取错误行和状态行，再后来加了 JSON 关键字段提取。每一次修改都对应一个或多个 commit。不是"设计好了四层管道"，而是"出了问题就改，改了又出新问题"。

## 第二天的另一个发现：系统提示词太长

2 月 22 日，commit `a75bd7a`：`perf: compress system prompt (~400 tokens/round saved)`。

当时 AgentClaw 的系统提示词包含了完整的技能目录——每个技能的名称、描述、触发条件，加起来好几百字。这些信息每轮调用都发一遍，但从第二轮开始，Agent 已经知道有哪些技能了，不需要重复看。

我把技能目录从完整描述改成了只列名字。单这一项，每轮省了约 400 token。

但这只是个开始。后来我发现系统提示词里还有更多可以精简的地方——冗余的安全规则、重复的格式说明、一些"以防万一"加的长段指令。`2026-02-28` 的 commit `e2bc223`：`perf: 系统提示词精简 + 记忆去重双层防线`，又砍了一刀。

## 那个让我兴奋了一整天的发现

2026 年 2 月 26 日，commit `8ccb0e5`：`perf: KV-Cache 上下文优化，input token 成本降低 50-60%`。

这是整个 token 优化过程中最大的单次胜利。

Anthropic 的 API 有一个叫"前缀缓存"（Prompt Prefix Caching）的功能：如果你的系统提示词在多次调用之间完全一致（字节级相同），那些 token 的成本降到约 10%。

但"完全一致"是关键。我之前的代码里，系统提示词包含动态内容——当前时间、会话 ID、最新的记忆列表。每来一条新消息，这些内容就变了，缓存就失效了。

修复方案：**把动态内容从系统提示词里移出去。** 当前时间放到用户消息里，会话 ID 放到 metadata 里，记忆在第一轮构建后冻结（新记忆写数据库，但不更新当前会话的系统提示词）。

这个改动的效果是立竿见影的——前缀缓存开始命中，系统提示词的 token 成本从每轮全价降到每轮 10%。commit 消息里写的"50-60%"是保守估计。

但这个优化有一个代价：**会话期间写入的新记忆，当前会话看不到。** 用户在第 5 轮告诉 Agent"我喜欢用 Tab 缩进"，Agent 在第 6 轮不会自动知道这条记忆——因为它要到下个会话才会被注入系统提示词。

这个取舍在当时让我纠结了很久。最后决定接受它，因为 Agent 从对话历史里已经知道了这条信息——它不需要记忆系统重复告诉它。用户感知不到延迟，但用量曲线能体现节省。

## Skill 预注入：一个看起来聪明但被撤回的方案

2 月 25 日，commit `5215624`：`feat: Skill 预注入——Embedding 相似度匹配，省掉 use_skill 轮`。

当时的想法是：Agent 每次需要使用技能，都要先调 `use_skill` 工具拿指令，再执行。能不能在系统提示词里预注入最相关的技能指令，省掉 `use_skill` 那一轮？

用 embedding 计算用户消息和技能描述的相似度，把最相关的技能指令直接塞进系统提示词。

14 分钟后，commit `388d1f9`：`Revert "feat: Skill 预注入——Embedding 相似度匹配，省掉 use_skill 轮"`。

撤回了。原因：预注入的技能指令不一定对——embedding 相似度匹配不如 LLM 自己判断准确。而且注入了错误的指令比不注入更糟糕，Agent 会按照错误的指令行事。更重要的是，预注入增加了系统提示词的长度，破坏了前缀缓存。

这个方案从提出到撤回只用了 14 分钟。但我记得当时花了好几个小时在犹豫——它看起来真的很合理。最后让我下决心撤回的是一个简单的测试：我故意发了一条跟技能无关的消息，系统还是注入了"最相关"的技能指令。Agent 看到那段指令，差点开始执行一个完全不需要的操作。

**省掉一轮工具调用的收益，不如避免错误注入的风险。**

## 邮件发送：一个 62K token 的惊喜

2 月 27 日，我让 Agent 帮我发一封邮件。

看起来是个简单的任务。但打开 token 统计，我傻了：**单次调用 62,000 token。** 六万二。

排查后发现原因：邮件 skill 的指令太长了——完整的 SMTP 配置说明、各种参数的详细解释、十几个示例。这些指令每次调用都发，而且 Agent 在执行邮件发送时需要读取邮件 skill 的完整内容。

commit `809d3cf`：`perf: 邮件发送 token 优化（62k→预期18k）`。把 skill 指令精简，去掉冗余说明。

然后 commit `58f835f`：`perf: email skill 改为预置脚本模式（20k→预期7k）`。更进一步——不给 Agent 一大段指令让它自己写代码发邮件，而是提供一个写好的脚本，Agent 只需要填参数调用。指令从 20K 降到了 7K。

这两次优化教会我一件事：**工具的 token 成本不只是工具定义，还有工具使用时附带的指令。** 一个技能可能只有 50 token 的定义，但使用时需要 20,000 token 的指令。

## 截断旧工具结果：一个后来才想到的点子

3 月 2 日，commit `583e767`：`perf: web_fetch 自动 scroll 重试 + 截断旧 tool result 控制上下文膨胀`。

这个优化的思路是：工具输出超过 3 轮后，模型大概率已经从中提取了关键信息，再看到完整输出对决策没有帮助。把旧的工具结果替换成占位符（`[之前的 file_read 结果]`），模型知道"发生过什么"但不需要看具体内容。

为什么是 3 轮？不是算出来的，是试出来的。先试了 1 轮（太激进——Agent 经常需要回顾上一轮的结果），又试了 5 轮（太保守——节省不够），最后 3 轮是平衡点。

占位符的设计也迭代了两次。第一版只写了 `[tool result]`，Agent 不知道是什么工具的结果，有时候会重新调一次来"确认"。第二版改成了 `[之前的 file_read 结果（第 2 轮）]`，包含了工具名和轮次，Agent 就不再重新调用了。

## 冻结快照：一个影响了好几个设计的决定

3 月 11 日，commit `541a651`：`feat: frozen snapshot 系统提示词——session 内冻结 dynamic context 提高 prompt cache 命中率`。

这个改动把"冻结系统提示词"从一个临时方案变成了一个正式的设计。"动态上下文"（记忆、技能目录、平台提示）在会话第一轮构建后就冻结，整个会话复用同一份系统提示词。

这个决定后来影响了好几个其他设计：

- 记忆系统必须在写入时就持久化到数据库，而不是依赖系统提示词实时更新
- 技能目录必须在会话开始时就确定，会话期间安装的新技能当前会话看不到
- 平台提示必须在会话创建时注入，不能动态切换

每一个都是一个小取舍。但加在一起，它们让前缀缓存的命中率接近 100%。

## 从 MiroThinker 借来的想法

3 月 18 日，commit `70911a6`：`feat: MiroThinker 三项改进 — 格式错误回滚 / 失败摘要 / 预测性 token 管理`。

这是从开源项目 MiroThinker 借鉴的三个想法，其中跟 token 最相关的是"格式错误回滚"：当 Agent 的输出是坏 JSON 时，不要把错误记录留在对话历史里——直接删掉这一轮，回滚迭代计数器，让 Agent 重新来。

为什么要删掉？因为坏 JSON + 错误信息会占据对话历史的空间，后续每轮调用都要重新发送这些垃圾。而且 LLM 看到一堆错误记录，输出质量会下降——它开始"焦虑"，试图修复之前的错误，而不是继续做正事。

回滚之后，这些垃圾从未存在过。Agent 重新来过，对话历史干干净净。

## 让 Agent 自己决定什么时候压缩

3 月 20 日，commit `7396f87`：`feat: compact 工具 — LLM 主动触发上下文压缩`。

之前所有压缩都是框架自动做的——达到阈值就压缩。但有时候 Agent 自己知道"这段历史不需要了"。比如它刚完成了一个子任务，之前的搜索结果和文件内容都不再相关。

给 Agent 一个 `compact` 工具，它可以在觉得合适的时候主动触发压缩。这比框架盲目按阈值压缩更精准——Agent 知道哪些信息还有用，哪些已经过时。

这个工具的使用率不高——大部分时候框架的自动压缩就够了。但在少数场景下（长对话、多子任务），它帮了大忙。

---

## 渐进截断：最后一次大改

3 月 31 日，commit `686db0d`：`feat: microCompact 改为渐进截断，保留前 200 字符 + 长度标记`。

之前的截断是"要么全留，要么砍到 50 字符"。问题是：50 字符有时候不够 Agent 回忆起它做了什么，200 字符有时候又太多。

改成渐进截断：先保留前 200 字符 + 总长度标记（比如 `[truncated, original: 3200 chars]`）。如果 Agent 需要看更多，它可以自己用工具去读完整内容。

这个改动看起来很小，但它是前面积累的所有经验的总结。从最初的"直接砍"，到"保留首尾行"，到"提取关键信息"，到"渐进截断"——每一次迭代都在解决上一次遗留的问题。

---

## 优化轨迹

让我用 commit 时间线来展示优化的真实过程，而不是教科书里那种"设计好了四层管道一步到位"的叙事：

| 时间 | 做了什么 | 效果 |
|------|---------|------|
| 2/22 | Shell 输出截断 + 系统提示词压缩 | 省了，但砍掉了错误信息 |
| 2/25 | Skill 预注入 | 14 分钟后撤回 |
| 2/26 | 前缀缓存（冻结系统提示词） | **最大的单次胜利** |
| 2/27 | 邮件 skill 优化 | 62K → 7K |
| 2/28 | 系统提示词精简 + 记忆去重 | 继续砍 |
| 3/2 | 截断旧工具结果（3 轮前） | 有效的增量优化 |
| 3/11 | 冻结快照正式化 + tool pair 保护 | 基础设施级改动 |
| 3/18 | 格式错误回滚（借 MiroThinker） | 减少垃圾历史 |
| 3/20 | compact 工具（Agent 主动压缩） | 精准但使用率低 |
| 3/31 | 渐进截断 | 最终形态 |

**不是四层管道，是十几次迭代。** 每次都是"发现一个问题 → 试一个方案 → 方案引入新问题 → 修新问题"。有些方案被撤回了（Skill 预注入），有些方案改了好几版（截断策略从"直接砍"改到"渐进截断"），有些方案看起来很小但影响深远（冻结快照）。

## 看不见的 token：后台调用

还有一个不太容易发现的成本来源：后台 LLM 调用。

Agent 的主循环在 trace 日志里有完整记录——每轮调了多少 token，一目了然。但系统里还有一些后台进程也在调 LLM，它们的 token 不出现在任何仪表盘上。

**记忆提取** 是最大的隐形杀手。最初设的是每 3 轮对话提取一次记忆。30 轮对话 = 10 次提取调用 = 约 30,000 token 的隐性消耗。

改成每 8 轮提取一次后，调用次数减少 60%。为什么 8 轮够？因为值得记住的信息通常不会每 3 轮就变一次——用户的姓名、技术栈、项目结构在一段对话中只会出现一次。

**上下文摘要** 是另一个。当对话太长时，用 LLM 对早期对话做摘要。最初的缓存键包含了最新消息的 ID——每来一条新消息，缓存键就变了，即使要摘要的旧对话完全没变，也会重新调 LLM。

改成按轮次计数做缓存键后，同样的旧对话 = 同样的摘要 = 缓存命中 = 零 LLM 调用。

**审计的方法：** 在代码里搜索所有调用 LLM API 的地方——不只是主循环，还有后台任务、定时任务、错误处理。每个调用点都是一个潜在的成本来源。

## 优化的本质

这轮优化做下来，我发现所有优化都是同一个原则的变体：

**不要发送模型已经知道的信息。**

工具输出——模型已经处理过了。写操作参数——模型自己写的。工具定义——静态的，不需要每次都发。系统提示词——固定的，缓存起来。

这跟人类的沟通是一样的。你跟同事开会，不会每次都从项目背景重新讲一遍。你们共享一个"工作记忆"。Agent 框架就是在 LLM API 的无状态限制下，模拟这个"共享记忆"——但用最经济的方式。

还有一个经常被忽略的维度：**token 不只是成本问题，还是时间问题。** 更多的 token = 更长的首 token 延迟（TTFT）= 用户觉得 Agent 变慢了。压缩到更少的 token 后，响应速度明显改善。控制成本和提升速度是同一件事的两面。

---

## 那些被删掉的方案

**字典压缩。** 把常见字符串替换成短码（`"error"` → `$E1`）。试了，LLM 处理编码的能力不稳定，4-8% 的节省不值得冒降低输出质量的风险。删了。

**激进历史截断。** 只保留最近 N 条消息，其他全部丢弃。试了，多步骤任务被打断——Agent 在第 3 步读了文件，第 7 步需要引用那个内容，但历史已经被截断了。Agent 只好重新读一遍，反而多花了 token。删了。

**模型降级。** 把简单查询路由到更便宜的模型。建了基础设施——一个 14 维加权评分的路由器。但 Agent 任务对模型质量敏感，工具选择错误直接导致任务失败。减少每次调用的 token（通过压缩）比降低每个 token 的价格（通过换模型）效果更好。路由器保留了，但主循环不用。

这些被删掉的方案有一个共同点：**它们都是"看起来最直接"的方案。** 换便宜模型、砍掉历史、压缩编码——直觉上应该有用，但在 Agent 场景下都有副作用。真正的优化往往不是"盲目降低单价"，而是"不要重复发送同一批信息"。

---

## 本章小结

1. **优化从第二天就开始了，不是等出了问题才做。** 快速膨胀的 token 用量曲线就是最好的驱动力。

2. **不是四层管道，是十几次迭代。** 真实的优化过程是"发现问题 → 试方案 → 方案引入新问题 → 修新问题"的循环。

3. **最大的单次胜利来自前缀缓存。** 冻结系统提示词，让 Anthropic 的 KV-Cache 命中，效果立竿见影。

4. **有些方案 14 分钟就被撤回了。** Skill 预注入看起来很合理，但测试后发现风险大于收益。

5. **后台 LLM 调用是隐形消耗。** 记忆提取、上下文摘要——不在 trace 里，但累加很快。

6. **Token 优化的本质是"不发重复信息"。** 所有优化都是这一个原则的变体。

## 动手试一试

**任务 1：审计你的第一条 token 曲线。**

如果你有在用 LLM API（不管是直接调用还是通过 Cursor、Claude Code），查一下你的用量面板。一天用了多少输入 token？一个对话用了多少？最大的消耗来源是什么？

**任务 2：找到你的"62K token 的邮件"。**

在你的 Agent 或 AI 工具里，找一个 token 异常高的对话或调用。是什么导致了这么高的消耗？是工具输出太长？是系统提示词太长？是对话历史太长？
