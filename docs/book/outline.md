# 《造一个真能用的 AI Agent》大纲定稿

副标题：从循环、工具、记忆到上线，一本书讲透智能体工程

作者署名建议：Rosibo

配套项目：AgentClaw（爬爬虾）

GitHub：https://github.com/vorojar/AgentClaw

---

## 出版定位

### 一句话

这本书写给想亲手造 AI Agent 的产品人、创业者、独立开发者和半技术出身的互联网人：你不必先成为职业程序员，但必须学会把目标、边界、流程和验收标准想清楚，指挥 AI 一步步造出一个能工作、能记忆、能接入真实用户的智能体。

### 核心承诺

读完这本书，读者不只是知道 Agent 是什么，而是能理解一个真实 Agent 框架从 0 到 1 的关键决策：循环怎么控制，工具怎么设计，记忆怎么保存，成本怎么压低，错误怎么处理，安全怎么兜住，最终怎么上线给别人用。

### 核心判断

造 Agent 的难点不在“接一个 LLM API”，而在让它稳定地做事。真正的工程问题是控制循环、约束工具、管理上下文、处理失败、保护安全和承受真实用户。

### 目标读者

**核心读者**：
- 想用 AI 做产品、工具、自动化系统的产品经理、创业者、独立开发者、运营者、内容创作者。
- 不以职业程序员自居，但有技术理解力，愿意用 Cursor、Claude Code、Codex 等 AI 编程工具动手的人。
- 已经会写一点代码，但还没有系统理解 Agent 工程的人。

**次级读者**：
- 想把聊天机器人升级为可执行任务系统的工程师。
- 想理解 AI Agent 产品化路径的技术管理者和投资人。

**不服务的读者**：
- 只想学 prompt 小技巧的人。
- 只想看模型数学、论文推导或底层训练原理的人。
- 期待复制粘贴代码、不理解设计取舍的人。

### 市场位置

这不是一本传统编程教程，也不是一本 AI 科普书。它的位置更接近“AI Agent 工程方法论”：用一个真实开源项目做案例，把产品判断、系统拆解和必要工程深度放在一起。

### 写作基调

- 故事驱动：每章从一个真实困境开始，不从概念定义开始。
- 类比先行：先用生活场景解释，再进入技术结构。
- 少代码、多决策：代码只展示关键骨架，重点解释为什么这样设计。
- 数据说话：涉及效率、成本、稳定性时，尽量给出 before/after。
- 诚实写失败：保留踩坑、删掉的设计和没有解决的问题。
- 章末可行动：每章都有“动手试一试”，读者可以用 AI 编程工具完成最小实验。

## 封面文案

2026 年春节，一个从 2000 年开始做软件和网站、长期经营互联网教育和医疗教育产品的创始人，用 AI 编程工具在几天内搭出了一个 AI Agent 框架的可运行骨架，随后持续迭代，把它扩展成一个能上网搜索、能记住用户、能调用工具、能在多个聊天平台运行的开源框架。

他是半技术出身的创始人：懂软件，懂网站，能看代码、做判断、和工程师讨论架构，但主战场更偏产品、运营和商业。他长期经营的易哈佛/知学科技，公开介绍显示已服务大量医疗从业者和基层医疗机构。这段经历让他习惯从真实用户、真实交付和真实运营成本出发看技术。AI 编程工具让他把“明确目标、拆解问题、验证结果”变成了造软件的核心能力，把大量实现工作交给 AI 协助完成。

本书从一个最简单的问题开始：怎样让 AI 不只回答问题，而是自己把事情做完？

你会跟着作者拆开一个真实 Agent 框架：它的大脑如何循环思考，它的手脚如何调用工具，它如何记住长期信息，它如何处理失败、减少重复 token、降低安全风险，并最终部署到真实用户面前。

你不需要先成为职业程序员。你需要学会提出清楚的问题、做出清楚的取舍、验收清楚的结果，然后让 AI 帮你把系统搭起来。

## 全书结构

全书分为九个部分、十五章。前两章建立读者信心和 Agent 的最小模型；中间十章拆解真实框架的核心系统；最后三章讨论生产化、复盘和未来。

### 开篇：为什么现在可以从零造 Agent

**作用**：建立读者代入感，解释“半技术出身的产品型创始人造 Agent”为什么不是噱头，而是 AI 编程工具带来的新分工。

**主要内容**：
- 一个产品人为什么要造 Agent。
- AI 时代的新分工：人负责目标、边界和判断，AI 负责实现和试错。
- 本书的读法：先理解系统，再动手实验；不要求手写所有代码。
- 配套项目 AgentClaw 是案例，不是唯一答案。

## 第一部分：Agent 到底是什么

### 第 1 章 AI 不只是聊天的对象

**核心问题**：ChatGPT 和 Agent 到底有什么区别？

**本章主旨**：聊天机器人回答问题，Agent 试图完成任务。区别不在模型，而在循环、工具和外部世界。

**章节要点**：
- 从“问一句答一句”到“自己把事情做完”。
- Agent 的最小定义：感知（Perceive）→ 推理（Reason）→ 行动（Act）→ 观察（Observe）。
- 用“做饭”解释 Agent 循环：看冰箱、想菜单、动手做、尝味道、再调整。
- 为什么现在是造 Agent 的时机：模型能力、工具生态、成本下降、AI 编程工具成熟。
- 本书最终要造的东西：能聊天、能搜索、能记忆、能调用工具、能多渠道运行、能上线给真实用户使用的 Agent。

**动手试一试**：让 AI 编程工具画出一个最小 Agent 流程图，并用自己的语言解释每个环节。

### 第 2 章 最简 Agent：20 行代码背后的循环

**核心问题**：Agent 的核心循环到底长什么样？

**本章主旨**：最简 Agent 很容易跑起来，但真正的难点是让它知道什么时候停、什么时候换方法、什么时候向用户交代。

**章节要点**：
- 接上 LLM API，得到一个只会说话的 Agent。
- 给它一个搜索工具，对话变成任务。
- 核心循环：LLM 调用 → 工具执行 → 结果反馈 → 再次调用。
- 第一次失控：无限调用、重复失败、重复消耗 token。
- 控制循环比启动循环更重要。
- 对应 AgentClaw：`agent-loop.ts` 的骨架。

**动手试一试**：用 AI 编程工具写一个最小 Think-Act-Observe 伪代码，并加上最大迭代次数限制。

## 第二部分：给 Agent 装上手脚

### 第 3 章 工具：Agent 的手和脚

**核心问题**：怎么让 AI 读文件、搜网页、执行命令？

**本章主旨**：工具不是给人看的按钮，而是写给 AI 看的能力说明。工具描述质量决定了 AI 会不会正确使用它。

**章节要点**：
- 工具的本质：一个有名字、参数、返回值和执行逻辑的函数。
- 六个字段就够了：`name` / `description` / `parameters` / `category` / `pure` / `execute`。
- `description` 比 `execute` 更影响 AI 行为。
- 永远加载的核心工具 vs 按需加载的条件工具。
- 过度设计的教训：中间件、能力协商、自定义校验器为什么被删掉。
- 少给选择，反而能减少选错工具。

**动手试一试**：设计 3 个工具的说明，让 AI 判断什么时候该用哪个工具。

### 第 4 章 当工具出错的时候

**核心问题**：AI 调用工具失败了怎么办？

**本章主旨**：工具错误不能只返回“失败了”。每个错误都必须变成下一步行动的清晰指令。

**章节要点**：
- AI 调了不存在的工具：返回完整可用工具列表。
- 工具输出太大：写入文件，只给预览。
- AI 重试同一个失败命令：重复检测、失败上限、全局安全网。
- 格式错误和工具不存在错误如何回滚。
- 核心原则：限制消息是指令，不是通知。

**动手试一试**：写 5 条工具错误提示，把“报错信息”改成“下一步指令”。

### 第 5 章 并行：让 Agent 同时做几件事

**核心问题**：AI 同时要做三件事，怎么让它们并行而不互相破坏？

**本章主旨**：并行不是“越多越快”，而是区分哪些操作可以同时发生，哪些必须按顺序发生。

**章节要点**：
- 读操作可以并行，写操作必须串行。
- `pure` 标记：告诉框架哪些工具没有副作用。
- 批量执行多个工具调用，减少等待时间。
- 并行的边界：文件写入、数据库写入、外部状态变更不能随意并发。
- AgentClaw 的批处理逻辑如何节省 200-800ms。

**动手试一试**：把一组工具分成“可并行”和“必须串行”两类，并解释理由。

## 第三部分：教 Agent 记住事情

### 第 6 章 上下文窗口：Agent 的工作记忆

**核心问题**：AI 的“记忆”只有这么大，怎么塞进最多有用信息？

**本章主旨**：上下文不是容器，而是预算。你不是决定什么能放进去，而是决定什么值得留下。

**章节要点**：
- 128K token 看起来很大，实际可用空间很快被系统提示词、工具定义和工具输出吃掉。
- 上下文预算的三类开销：固定开销、历史对话、工具结果。
- 压缩的三层瀑布：正常摘要、激进摘要、无 LLM 的确定性截断。
- 新鲜尾部保护：最近消息不能压缩。
- 冻结快照：系统提示词只构建一次，提高 prompt cache 命中率。
- 对应 AgentClaw：`context-manager.ts`。

**动手试一试**：给一段长对话做三种压缩版本：5 句话、200 字、50 字。

### 第 7 章 长期记忆：跨越会话的智能

**核心问题**：对话关了，AI 就忘了你。怎么让它记住？

**本章主旨**：记忆不是保存聊天记录，而是从聊天中提取值得长期保留的事实。

**章节要点**：
- 默认状态：每次对话都从零开始。
- 五类记忆：身份、偏好、事实、关系、经历。
- 记忆的衰减：你的名字不该忘，上周随口提的文章可以淡忘。
- 怎么存：SQLite + 全文搜索 + 向量嵌入。
- 怎么找：关键词、语义相似、时间衰减、重要性权重。
- 自动提取：不要求用户说“记住这个”。
- 记忆也是攻击面：写入前必须扫描 prompt injection。

**动手试一试**：从 10 轮对话中提取 5 条长期记忆，并标注类型和重要性。

### 第 8 章 Token 经济学：别把同一段上下文发两遍

**核心问题**：Agent 为什么会迅速消耗大量 token，怎么把重复输入压下来？

**本章主旨**：最贵的 token 往往不是模型输出，而是你反复发送给模型的重复上下文。

**章节要点**：
- 工具输出为什么会成为 token 膨胀源。
- 大输出存磁盘，只给模型看预览。
- 过时输出替换成占位符。
- 智能提取错误行、状态行和关键字段。
- JSON 压缩和空白标准化。
- 后台 LLM 调用的隐性成本：记忆提取、摘要、标题生成。
- 实测数据：平均 token、响应耗时和缓存命中变化。

**动手试一试**：把一段 5000 字工具输出压缩成 300 字摘要，保留可行动信息。

## 第四部分：当事情出错

### 第 9 章 LLM 会犯错，而且经常犯

**核心问题**：AI 调用失败、卡住、输出垃圾时，系统应该怎么办？

**本章主旨**：成熟的 Agent 框架不是假设 LLM 会稳定工作，而是假设每一次调用都可能失败。

**章节要点**：
- 七类 LLM 错误：密钥、配额、限速、服务器忙、服务器挂、请求太大、网络抖动。
- 200 OK 的谎言：状态码成功，body 里仍然可能是错误。
- 卡住但没报错：用相似度检测重复输出。
- 格式错误回滚：坏 JSON 不该永久污染上下文。
- 降级链：首选模型、备选模型、任意可用模型。
- 流式输出的约束：首 token 之后不能切模型。

**动手试一试**：为 7 类错误各写一条处理策略，区分“重试”“等待”“换模型”“停止”。

## 第五部分：安全

### 第 10 章 你的 Agent 有 root 权限，谁没有？

**核心问题**：给 AI 装了 shell、文件系统和网络访问，安全风险怎么降下来？

**本章主旨**：Agent 安全的核心不是防人进来，而是防进来的人控制你的 AI 去做坏事。

**章节要点**：
- Agent 的威胁模型和传统 Web 应用不同。
- 六类威胁：Prompt injection、路径遍历、SSRF、Shell 注入、子代理逃逸、信息泄露。
- Shell 沙箱：防意外，不宣称能防住蓄意攻击。
- 文件读取黑名单：`.env`、`.pem`、`id_rsa` 等不能读。
- SSRF 防护：拦截私有 IP 和云元数据地址。
- 记忆写入扫描：防止长期记忆被污染。
- 子代理隔离：禁用危险工具和回调能力。
- 容器是最后防线，软件层防御不是银弹。

**动手试一试**：列出一个 Agent 的 10 条危险输入，并设计对应拦截点。

## 第六部分：让 Agent 看见世界

### 第 11 章 浏览器：Agent 的眼睛

**核心问题**：怎么让 AI 能“看”网页、“点”按钮？

**本章主旨**：让 Agent 使用浏览器，不是把整张网页塞给模型，而是把网页转成它能理解、能行动的结构。

**章节要点**：
- 三种看网页的方式：截图、原始 HTML、无障碍快照。
- 截图直观但贵，坐标不可靠。
- 原始 HTML 完整但 token 灾难。
- 无障碍快照借鉴屏幕阅读器，只保留可读、可点、可填的结构。
- 给元素编号，让 AI 用 `e1`、`e2` 这类引用行动。
- 登录态持久化：保存 cookie，下次直接复用。
- 无头部署：服务器没有屏幕时如何跑浏览器。

**动手试一试**：把一个登录页手工抽象成无障碍快照格式，让 AI 指出下一步点击哪里。

## 第七部分：让 Agent 走进用户的生活

### 第 12 章 多渠道：一个大脑，七张嘴

**核心问题**：怎么让同一个 AI 同时在 Telegram、微信、钉钉等平台运行？

**本章主旨**：多渠道的关键不是为每个平台复制一套逻辑，而是让不同入口共享同一个大脑和生命周期。

**章节要点**：
- 一个平台写一套代码会很快失控。
- `ChannelManager`：每个渠道只需要暴露 `stop` 和 `broadcast`。
- 共享 session 管理、超时处理、消息分割。
- 平台提示：让 AI 知道自己正在跟哪个平台的用户说话。
- 5 分钟超时：用户走了，Agent 不能无限等待。
- 新渠道代码从 120 行降到 30 行的原因。

**动手试一试**：设计一个新渠道接入清单，标出哪些逻辑应该共享，哪些必须平台特定。

## 第八部分：上线给别人用

### 第 13 章 从“能跑”到“能用”

**核心问题**：本地跑得好好的，部署到服务器为什么会崩？

**本章主旨**：生产可用不是一个功能，而是一组无聊但救命的系统：容器、备份、迁移、关闭、健康检查、日志和监控。

**章节要点**：
- 容器化：打包环境，而不是依赖服务器。
- 非 root 用户：最高 ROI 的安全措施。
- 自动备份：朴素的每日备份比没有备份强无穷倍。
- 优雅关闭：不要杀死正在对话的用户。
- 数据库迁移：`addColumnIfMissing` 的幂等模式和边界。
- 健康检查：让 AI 提前知道哪些工具坏了。
- 上线检查清单：少于 12 项就是 demo，不是生产系统。

**动手试一试**：给自己的 Agent 项目打生产可用分数，列出上线前必须补齐的 5 项。

## 第九部分：复盘与未来

### 第 14 章 我们犯过的错，以及你不必再犯

**核心问题**：造这个框架的过程中，我们踩了哪些坑？

**本章主旨**：真正有价值的经验不只是“最后怎么做”，还包括“哪些方案看起来聪明，最后被删掉”。

**章节要点**：
- 工具描述比工具实现更影响效果。
- 没有输出溢出处理时，一个 `cat` 就能炸掉上下文。
- 子代理不共享预算会导致迭代爆炸。
- 记忆提取频率太高，会变成隐性 token 消耗。
- 过度设计：中间件链、能力协商、自定义校验器为什么被删。
- “先试试看”的代价：没有研究清楚就动手，会连续制造 fix。

**动手试一试**：复盘自己最近一次 AI 编程失败，写出根因、误判和下次预防策略。

### 第 15 章 Agent 的未来：更强的模型不会消灭工程

**核心问题**：模型越来越强，Agent 框架会不会变得不重要？

**本章主旨**：模型越强，能做的事越多，越需要清晰的边界、工具、预算和安全控制。

**章节要点**：
- Prompt injection 仍然是本质难题。
- 模型变强会减少某些胶水代码，但不会消灭系统设计。
- 从单 Agent 到多 Agent 协作。
- Agent 时代的新分工：你不必会写所有代码，但必须会定义问题。
- “造什么”比“怎么造”更重要。
- 本书的出口：读者可以从 AgentClaw 开始，也可以造自己的第一个 Agent。

**动手试一试**：写一页“我的第一个 Agent 需求说明”，包括目标用户、核心任务、工具、记忆、安全边界和上线方式。

## 章节统一模板

每章写作时固定使用以下结构，保证全书节奏一致：

1. **真实场景**：用一个具体失败、困惑或需求开场。
2. **本章核心判断**：一句话说清本章真正要解决什么。
3. **大白话类比**：先把问题讲清楚，再进入技术。
4. **系统拆解**：讲结构、数据流、边界和取舍。
5. **AgentClaw 案例**：展示真实实现，但不变成源码逐行讲解。
6. **我们踩过的坑**：写清失败方案和删除理由。
7. **动手试一试**：给一个最小实验，读者可用 AI 编程工具完成。
8. **本章小结**：用 3-5 条可复述结论收尾。

## 术语规范

- 全书统一使用“Agent”作为核心称呼，首次出现时写作“AI Agent（智能体）”。
- “上下文窗口（Context Window）”“工具调用（Tool Calling）”“长期记忆（Long-term Memory）”“向量嵌入（Embedding）”“提示词注入（Prompt Injection）”首次出现时给出中文解释和英文原词。
- “AI”“LLM”“token”“API”“prompt”保留英文，不强行翻译。
- “用户”指使用 Agent 的人；“读者”指读这本书的人；“开发者”只在确实指写系统的人时使用。
- 不使用“所有人都能”这类过度承诺，改为“愿意动手的人可以开始”。

## 出版修改原则

1. **书名定为《造一个真能用的 AI Agent》**：比“Agent 时代”更具体，比“你的第一个 AI Agent”更有结果感。
2. **读者不再写“所有人”**：出版定位必须有核心读者，否则封面、目录和营销都会失焦。
3. **删除内部备忘语气**：目录里不写“从 CLAUDE.md 和 memory 中提炼”等项目内部痕迹，改成读者可理解的价值表达。
4. **章节数保持 15 章**：内容足够厚，但不显得百科化；每章只解决一个核心问题。
5. **每章必须有读者收益**：读者看完这一章，应该能说出自己学会了什么、能做什么。
6. **技术深度服务叙事主线**：不是把英文系列文章翻译成中文，而是改写成一本中文书。
7. **没有出处的精确数字比没有数字更糟糕**：无法从代码/commit/trace 核实的百分比和金额一律改为定性描述（"大幅下降""明显减少"）。能核实的保留（如 Anthropic 前缀缓存 90% 折扣、代码行数 37,000 行）。
