# 第 3 章 工具：Agent 的手和脚

---

## 大脑在罐子里

上一章我们造了一个能循环思考的 Agent。它能接上 LLM API，能自己决定下一步做什么，能在失败时重试，能在卡住时换策略。

但它什么都做不了。

它能告诉你"我应该搜索一下这个关键词"，但它不能真的去搜索。它能说"我应该读一下这个文件"，但它不能真的打开文件。它能说"我应该执行这段代码"，但它不能真的运行代码。

它是一个大脑，装在罐子里。能想，不能动。

我第一次让 AgentClaw 跑起来的时候，就是这种感觉。我输入"帮我查一下今天的天气"，它回复了一大段关于"我应该如何帮你查询天气"的文字，然后告诉我"但我没有访问天气数据的能力"。

它说得对。它确实没有。

这让我意识到一件事：上一章我们花了那么多精力在循环控制上——迭代上限、失败追踪、格式回滚、相似度检测。这些都很重要。但如果 Agent 连一个工具都没有，这些防护代码一行都不会执行。循环需要有东西可循环——工具就是那个"东西"。

没有工具的 Agent 是一个很聪明的废物。它能分析问题、规划步骤、评估方案——但它不能做任何事。就像一个棋手能看到十步之后的棋局，但没有手去落子。

于是下一个问题来了：**怎么给这个大脑装上手和脚？**

这个问题的答案看起来很简单：给它一些函数，让它能调用。但"给它函数"这件事本身，比你想象的复杂得多。函数叫什么名字？怎么描述它的功能？参数怎么定义？执行失败了怎么办？输出太大了怎么处理？哪些函数可以同时调用？哪些必须按顺序来？

这些问题的答案，决定了你的 Agent 是一个"能做事的助手"还是一个"经常犯蠢的聊天机器人"。

---

## 工具不是给人看的按钮

如果你用过 ChatGPT 的插件功能，你可能觉得"工具"就是一个按钮——你点一下，它帮你做一件事。

但 Agent 的工具不是给人点的。它是写给 AI 看的。

这句话值得展开说，因为它决定了你后面所有关于工具设计的思考方向。

当你设计一个 App 的按钮时，你在考虑的是：用户看到这个按钮会怎么理解？它的颜色、位置、文字能让人一眼看出它是干什么的吗？你设计的是**人机界面**。

当你设计一个 Agent 的工具时，你在考虑的是：LLM 看到这个工具的描述，会正确地使用它吗？它会在该用这个工具的时候用它，还是会在不该用的时候乱用？你设计的是**AI 机界面**。

LLM 不看颜色、不看位置、不看图标。它只看三样东西：**工具名、描述、参数**。这三样东西写得好不好，直接决定了 Agent 的行为质量。

打一个比方：你在教一个新员工做事。你不能指着桌子说"做那个"——他不知道"那个"是什么。你需要告诉他："桌上有三个文件夹，红色的要签字，蓝色的要归档，绿色的要发给财务。"你描述得越清楚，他犯的错越少。

LLM 就是那个新员工。工具描述就是你的"交代"。你交代得越清楚，它的表现越好。

我们踩过的最深的一个坑就是这个：**工具的"说明书"比工具的"实现"更重要。**

## 六个字段就够了

AgentClaw 的工具接口（Tool Interface）长这样：

```
工具 = {
  name:          "工具的名字"
  description:   "告诉 LLM 这个工具是干什么的、什么时候该用"
  parameters:    "工具接受什么参数，每个参数是什么类型"
  category:      "这个工具属于哪个类别"
  pure:          "这个工具是只读的吗？有没有副作用？"
  execute:       "实际执行工具的函数"
}
```

六个字段。没有更多了。

你可能觉得这太简单了。不应该有中间件（Middleware）吗？不应该有能力协商（Capability Negotiation）吗？不应该有自定义校验器（Custom Validator）吗？不应该有前置钩子（Before Hook）和后置钩子（After Hook）吗？

我们试过。

AgentClaw 的早期版本有中间件链——每个工具调用会经过一系列中间件，每个中间件可以检查输入、修改输出、记录日志、做权限校验。听起来很专业。

然后我们发现：中间件让调试变得地狱般困难。一个工具调用失败了，你不知道是工具本身的问题、还是某个中间件修改了输入、还是另一个中间件拦截了输出。错误信息被中间件层层包装，到了你手里已经面目全非。

我们删掉了中间件。能力协商也被删掉了——模型不需要"协商"，它需要的是清晰的指令。自定义校验器也被删掉了——JSON Schema 本身就是校验器，再加一层只是重复。

删掉之后，代码量明显减少，bug 率下降了，调试时间缩短了。不是因为这些功能"没用"，而是因为它们带来的复杂性超过了它们解决的问题。

这给了我们一个重要的教训：**过度设计（Over-engineering）不是多做了工作，是做了不该做的工作。** 每一个额外的抽象层、每一个"以防万一"的扩展点、每一个"将来可能会需要"的接口，都在增加系统的复杂性。复杂性是 bug 的温床。你加的每一层，都需要被理解、被测试、被维护。如果它没有解决一个当前存在的真实问题，那它就是在制造未来的问题。

**六个字段就够了。当你觉得需要第七个的时候，先问自己：这个字段解决的问题，能不能用更好的描述来解决？**

让我们逐个看这六个字段，理解每个字段为什么存在：

**`name`** —— 工具的唯一标识符。LLM 用它来指定"我要调用哪个工具"。命名规则：用下划线连接的英文小写，比如 `file_read`、`web_search`、`grep`。不要用驼峰命名（`fileRead`），LLM 在生成 JSON 时更容易犯拼写错误。不要用太长的名字（`read_file_contents_from_path`），浪费 token。

**`description`** —— 告诉 LLM 这个工具是干什么的、什么时候该用、什么时候不该用。这是最重要的字段，后面会单独展开。

**`parameters`** —— 工具接受什么参数，每个参数是什么类型。用标准的 JSON Schema 格式，因为 LLM 原生就理解 JSON Schema——任何额外的翻译层都是 bug 的来源。

**`category`** —— 标记工具属于哪个类别：`builtin`（内置）、`external`（外部）、`mcp`（MCP 连接）。这不影响 LLM 的决策，但影响框架的内部管理——比如 MCP 工具在断开连接后需要自动清理。

**`pure`** —— 布尔值，标记工具是否"纯净"——只读取信息、不修改任何东西。这决定了工具能否被并行执行。默认 `false`（不纯），只有明确标记为 `true` 的工具才会被并行处理。

**`execute`** —— 实际执行工具的函数。接受参数和执行上下文，返回工具结果。这是你唯一需要写代码的地方——其他五个字段都是声明式的。`execute` 的设计有一个重要约束：它接受的 `context` 参数是可选的，包含 `promptUser`（问用户问题）、`sendFile`（发文件）、`saveMemory`（存记忆）等回调。不同的运行环境（CLI、Telegram、Web）会注入不同的回调——同一个工具实现，在 CLI 里问用户问题是打印到终端，在 Telegram 里是发一条消息。这让工具代码可以跟渠道无关。

## description 比 execute 更重要

这六个字段里，最重要的不是 `execute`（执行逻辑），而是 `description`（描述）。

听起来反直觉。工具不就是用来"执行"的吗？执行逻辑才是核心吧？

不是。

对于一个 Agent 来说，工具的价值 = 描述质量 × 执行正确性。如果描述写得不好，LLM 不会在正确的场景调用它，再正确的执行逻辑也没用。如果描述写得好，LLM 能准确判断什么时候该用这个工具，即使执行逻辑偶尔出错，LLM 也能看到错误信息并调整。

AgentClaw 的 `shell` 工具最初的描述是：

> "执行 shell 命令。"

六个字。简单明了。

然后我们发现：LLM 会用 shell 来做所有事——读文件、搜索文本、下载网页。明明有更好的专用工具（`file_read`、`grep`、`web_fetch`），但 LLM 觉得 shell 能做所有事，就什么都用 shell。

问题出在哪？描述太宽泛了。"执行 shell 命令"听起来确实能做所有事。LLM 不会去想"我应该用更专用的工具"——它只看到"能做所有事"就用了。

我们把描述改成了：

> "执行 shell 命令。仅当没有更专用的工具可用时使用：读文件用 file_read，搜索用 grep，下载网页用 web_fetch。适用于系统管理、包安装、编译等其他工具无法完成的操作。"

改完之后，LLM 用 shell 的频率大幅下降。它开始优先使用专用工具，只在确实需要 shell 的时候才调用。

这不是因为模型变聪明了——同一个模型，同一个版本。只是描述变好了，模型的行为就变了。

再举一个例子。AgentClaw 的 `file_write` 工具最初只有三个参数：`path`（文件路径）、`content`（文件内容）、`encoding`（编码）。LLM 经常忘记给 `encoding` 参数，导致写入的中文文件变成乱码。

我们没有加任何代码逻辑——只是在 `encoding` 参数的描述里加了一句："如果不指定，默认使用 UTF-8。中文文件请确保使用 UTF-8 编码。"然后在 `content` 参数的描述里加了一句："写入前请确保内容编码正确。"

乱码问题从此消失了。又是描述的功劳。

**你不是在写文档给人看。你是在写指令给 AI 看。** 每一个字都影响 AI 的决策。

## 工具描述的四个原则

从 AgentClaw 的几十次描述重写中，我总结出了四个原则：

**原则一：说清楚"什么时候用"。** 不要只说工具能做什么，要说清楚什么时候该用它。"执行 shell 命令"太宽泛，"执行 shell 命令，仅当没有更专用的工具可用时使用"就精确多了。

**原则二：说清楚"什么时候不用"。** 这比"什么时候用"更重要。LLM 天然倾向于使用它"理解"的工具——如果一个工具描述太通用，LLM 会把它当成万能工具。你需要明确告诉它：在哪些情况下，你应该用别的工具。

**原则三：给参数起好名字。** 参数名是描述的一部分。`file_path` 比 `input` 好，`search_pattern` 比 `query` 好。LLM 看到 `file_path` 就知道应该传文件路径，看到 `input` 可能传任何东西。

**原则四：给参数加上描述。** 每个参数都应该有 `description` 字段。不要假设 LLM 会从参数名推断出参数的含义。`"type": "string"` 不够，`"type": "string", "description": "要搜索的正则表达式模式"` 才够。

这四个原则听起来像废话。但 AgentClaw 的 bug 报告里，有大量问题最终追溯到工具描述写得不好。"Agent 用了错误的工具"这个问题，十次里有八次不是模型的错，是描述的错。

一个反直觉的事实是：工具描述是 Agent 框架里"迭代最频繁"的部分之一。我们改描述的次数比改执行逻辑的次数多得多。因为描述直接影响行为，而行为问题是最容易被用户感知到的——"Agent 做了蠢事"比"Agent 的代码有 bug"更常被反馈。每次用户反馈"Agent 不应该用 shell 来做这个"，我们首先检查的就是工具描述。

## 分层加载：少给选择，反而更好

AgentClaw 有 22 个工具。如果每次调用 LLM 都把 22 个工具的定义全发过去，有两个问题：

**第一个问题是 token 消耗。** 每个工具定义大约 100-150 token（名字、描述、参数 schema）。22 个工具就是 2,000-3,000 token。每一轮迭代都要发送一遍，一个 10 轮对话就是 20,000-30,000 token——光是工具定义就会形成明显的重复输入。

**第二个问题更严重：决策质量下降。** 给 LLM 看 22 个工具和给它看 11 个工具，它选错工具的概率明显不同。工具越多，LLM 越容易选到一个"差不多能用"但不是最合适的工具。

这不是 LLM 的问题。你去一家餐厅，菜单上有 200 道菜，你很难选到最想吃的。菜单上有 20 道菜，你反而能更快做出好选择。选择太多会降低决策质量——这个道理对人和对 AI 都适用。

于是我们把工具分成了两层：

| 层级 | 包含哪些工具 | 何时加载 | 数量 |
|------|-------------|----------|------|
| 核心层（永远加载） | shell, file_read, file_write, file_edit, glob, grep, ask_user, web_fetch, web_search | 每个会话 | 9 |
| 条件层（按需加载） | send_file, schedule, update_todo, sandbox, subagent, browser_cdp, remember, use_skill 等 | 仅在特定模式下 | +13 |

一个 CLI 会话只看到 9 个核心工具。一个完整的网关会话看到 22 个。

这不是一个复杂的改动。核心代码大概 30 行——就是一个 `if` 判断：

```
function createBuiltinTools(options):
    tools = [shell, file_read, file_write, file_edit, glob, grep, ask_user, web_fetch, web_search]

    if options.gateway:
        tools.push(send_file, schedule, update_todo, sandbox, subagent, browser_cdp, ...)

    if options.memory:
        tools.push(remember)

    if options.skills:
        tools.push(use_skill)

    return tools
```

但效果立竿见影。"选错工具"的反馈少了，Token 消耗也降了——每个会话的工具定义从 22 个降到 9 个，省下的 token 是白捡的。

还有一个附带好处：系统提示词里提到的工具也需要跟着清理。如果系统提示词里写着"你可以使用 send_file 工具发送文件"，但 send_file 并没有注册（因为不在网关模式下），LLM 会困惑——"为什么提示词说我能用这个工具，但我找不到它？"分层加载后，系统提示词会自动裁剪掉未加载工具的引用。这让 LLM 看到的世界是一致的：提示词里提到的每个工具都能找到，没提到的就不存在。

## 并行执行：纯的和不纯的

现代 LLM 一次可以请求多个工具调用。比如 Claude 可能在一次响应里同时要求：用 `glob` 搜索 Python 文件，用 `grep` 搜索 import 语句。

如果你按顺序执行——先 `glob`，等完了再 `grep`——你浪费了时间。`glob` 和 `grep` 之间没有依赖关系，完全可以同时跑。

但不是所有工具都能并行。如果 LLM 要求先用 `file_write` 写入一个文件，再用 `shell` 执行这个文件，那顺序就很重要了。你不能让它们同时跑——`shell` 可能在 `file_write` 完成之前就开始执行了。

怎么区分哪些能并行、哪些不能？

答案是：看工具有没有"副作用"（Side Effect）。

- **纯工具（Pure）**：只读取信息，不修改任何东西。`glob`（搜索文件名）、`grep`（搜索文件内容）、`file_read`（读文件）、`web_search`（搜索网页）。这些工具之间没有依赖，可以同时跑。
- **不纯工具（Impure）**：会修改状态。`shell`（执行命令）、`file_write`（写文件）、`file_edit`（编辑文件）。这些工具可能影响其他工具的输入，必须按顺序执行。

AgentClaw 的每个工具都有一个 `pure` 标记。默认是 `false`（不纯），只有明确标记为 `true` 的工具才会被并行处理。这是一个"安全默认值"——如果你不确定一个工具能不能并行，就让它串行。宁可慢一点，也不要出错。

执行时，连续的纯工具会被打包成一个并行批次，用 `Promise.all` 同时执行。遇到不纯工具就暂停，等它完成后再继续。

如果 LLM 请求的工具序列是 `[glob, grep, file_write, grep, grep]`，执行计划是：

- 第一批：`glob` + `grep`，都是纯工具，用 `Promise.all` 并行执行。
- 第二批：`file_write`，会修改文件，单独串行执行。
- 第三批：`grep` + `grep`，再次并行执行。

这个优化能省几百毫秒。听起来不多，但累积下来用户能感觉到"快了一点"。

实现只有大约 30 行代码——就是把工具列表按 `pure` 标记分成批次，然后交替执行并行批次和串行批次。

这个优化的价值在不同场景下差异很大。对于简单的单工具调用（一问一答），没有任何区别。但对于复杂的多工具调用（Agent 同时需要搜索、读文件、查 API），效果很明显。

有一点需要注意：并行执行的前提是工具之间没有数据依赖。如果 LLM 请求的第一个工具的输出是第二个工具的输入（比如先 `file_read` 读一个路径，再用读到的内容做 `grep`），那它们不能并行。AgentClaw 的实现是按顺序检查——连续的纯工具并行，遇到不纯工具就暂停。这确保了有依赖关系的工具不会被错误地并行执行。

## 当工具输出太大

这是一个我们没有预料到的问题，但它在生产中出现得很早。

Agent 调用了 `shell` 执行 `find . -name "*.ts" -exec wc -l {} +`。这是一个合理的命令——统计项目中所有 TypeScript 文件的行数。但它在一个有 500 个文件的项目上执行，返回了 47,000 个字符的结果。

这些字符全部被塞进了对话历史。下一轮调用 LLM 时，上下文窗口被撑满了。Agent 开始"忘事"——它记不住你最早说的需求了。更糟的是，它可能会从这 47,000 个字符里"挑"一些它觉得重要的信息，但挑出来的往往是错的。

解决方案是**溢出模式（Overflow Mode）**：当工具输出超过 8,000 个字符时，把完整内容保存到一个临时文件里，只给 LLM 看前 1,500 个字符的预览，再加上一行提示："完整内容已保存到 data/tmp/overflow_shell_170234.txt，可以用 file_read 或 grep 查看。"

溢出模式会把一次 47,312 字符的工具输出拆成两部分：

- 给 LLM 的内容：前约 1,500 字符的预览，并在最后一个完整换行处截断。
- 存到磁盘的内容：完整输出，例如 `data/tmp/overflow_shell_170234.txt`。
- 附加提示：如果需要完整内容，可以用 `file_read` 或 `grep` 按需查看。

这个设计有一个巧妙的地方：LLM 通常不会去读完整的溢出文件。它会用 `grep` 搜索它需要的信息——比如 "grep 500 行以上的文件"。这比把 47,000 个字符全部读进去高效得多。**溢出模式不仅节省了 token，还引导 LLM 用更高效的方式获取信息。**

但有两个边界情况是我们踩坑后才发现的：

**不要溢出错误信息。** 错误信息通常很短但信号很强。如果一个工具失败了，LLM 需要看完整的错误栈才能理解问题在哪。溢出检查会跳过 `isError: true` 的结果。

**不要溢出溢出文件的读取结果。** 如果 LLM 读了一个溢出文件，读取结果又溢出了，新的溢出又保存到文件里……无限循环。我们检测到溢出文件路径后，对这些读取直接跳过溢出逻辑。

这两个边界情况不是"想到了就加的"——是线上出了问题后才补的。第一次遇到错误信息被溢出时，LLM 只看到了前 1,500 个字符的错误栈，丢失了最关键的信息（实际出错的那一行）。它基于不完整的信息做了一个错误的"修复"，结果引入了新的 bug。第二次遇到溢出文件循环时，对话直接卡死——Agent 在读文件和生成溢出文件之间无限循环，直到迭代上限耗尽。

溢出处理看起来是一个简单的"截断"逻辑，但边界情况比你想象的多。这就是为什么我们说"实现 40 行代码，调试 2 天"。

## 外部工具：MCP

你建了一个干净的工具系统。然后有人问："我能接入我们公司内部的工具吗？"

你可以让他们实现你的 `Tool` 接口，然后重新编译。但这样每次加新工具都要改代码、重新部署。

或者你可以支持 MCP（Model Context Protocol，模型上下文协议）——一个让第三方工具即插即用的标准。MCP 服务器暴露自己的工具定义，你的 Agent 框架自动识别并注册。

AgentClaw 的 MCP 集成把外部工具的 schema 转换成内部的 `Tool` 接口。工具名用双下划线做命名空间隔离（比如 `playwright__browser_click`、`slack__send_message`），避免不同服务器之间的名称冲突。外部工具走的是跟内置工具完全一样的执行管道——同样的失败计数、同样的溢出处理、同样的去重检测。LLM 不知道也不在乎一个工具是内置的还是 MCP 连接的。

但有一个限制是故意的：MCP 服务器只能在启动时加载，不支持运行时动态增删。

为什么？因为你不知道运行时加减工具会出什么问题。一个工具正在被 LLM 调用，你突然把它卸载了——调用怎么办？LLM 已经根据某个工具规划了下一步，你突然把它删了——规划怎么办？启动时加载是"无聊但正确"的做法。

MCP 的价值在于：它让 Agent 的能力可以无限扩展，而不需要修改 Agent 框架本身的代码。你想接 Slack？装一个 MCP 服务器。你想接 Jira？装一个 MCP 服务器。你想接公司内部的审批系统？写一个 MCP 服务器。Agent 框架不需要知道这些系统的存在——它只需要知道"我有一些工具可以用"。

这是一种非常干净的扩展方式：框架提供核心能力（循环、工具执行、错误处理），MCP 服务器提供具体工具。两者通过一个标准协议通信，互不依赖。

## 注册表：优雅降级

所有工具都注册在一个叫 `ToolRegistry`（工具注册表）的地方。它本质上就是一个 `Map<string, Tool>`——名字到工具的映射。

注册表的有趣行为在 `execute()` 方法里。

当 LLM 调用了一个不存在的工具名时，注册表不会直接返回错误。它会先检查这个名字是否匹配一个已注册的技能（Skill）。LLM 有时候会把技能和工具搞混——比如调用 `agent-browser`（一个技能）而不是 `use_skill`（工具）。注册表检测到这种混淆后，会自动重定向到正确的工具。

这一个检查消除了整整一类浪费的迭代。没有这个检查，LLM 会：调用错误的名字 → 收到错误 → 道歉 → 再调用正确的名字。三轮消息，浪费了时间、token 和迭代次数。有了这个检查，LLM 的"错误"调用被自动修正，零浪费。

对于真正不存在的工具名，错误信息会附带完整的可用工具列表："工具 'X' 不存在。你只能使用：shell, file_read, file_write, ..."。**给模型纠正信息，而不只是失败信号。**

## 子代理的工具隔离

当主代理启动子代理时，子代理不应该拥有跟主代理一样的全部工具。

原因有三：

**防止递归。** 如果子代理有 `subagent` 工具，它可以启动自己的子代理，子代理又可以启动子代理。无限嵌套。所以 `subagent` 工具必须从子代理的工具列表里移除。

**防止记忆污染。** 如果子代理有 `remember` 工具，它可能会写入一些主代理不想要的记忆。子代理是临时工，不应该有权限修改长期记忆。

**防止越权。** 如果子代理有 `send_file` 工具，它可能会在主代理不知情的情况下给用户发送文件。子代理的文件操作应该通过主代理来协调。

AgentClaw 的子代理工具黑名单包含 6 个工具：`subagent`、`ask_user`、`remember`、`schedule`、`send_file`、`social_post`。子代理拿到的是一个过滤后的注册表——它不知道这些工具的存在。

这个设计是"最小权限原则"的体现：每个组件只拥有它需要的最小权限集。子代理只需要读写文件、搜索信息、执行命令——这些是核心工具。其余的，主代理来做。

工具黑名单不是一成不变的。随着 AgentClaw 的功能增加，黑名单也在调整。但原则始终不变：**子代理不应该有能力影响它控制范围之外的任何东西。** 它是临时工，不是合伙人。

## 我们犯过的错

**低估了描述质量。** 前面说过，不再赘述。核心教训：写描述的时间应该超过写执行逻辑的时间。一个 20 行的工具实现，值得配一段 200 字的描述。

**过度信任 LLM 的重试判断。** 没有框架级重试时，LLM 会"微调"参数重试——换一个引号、改一个路径、加一个参数。把一个简单的"服务不可用"变成了 10 轮调试会话。自动重试（带指数退避）在 LLM 开始"创新"之前就解决了瞬时故障。我们在生产数据里发现：网络类工具（`web_search`、`web_fetch`）的瞬时失败率大约 5%。没有自动重试时，这些 5% 的失败会变成 LLM 的 10 轮"调试"——每次调试消耗一次迭代、一次 LLM 调用、几百个 token。有了自动重试（2 秒、4 秒退避），这些失败在框架层就被消化了，LLM 根本看不到。

**没有预设输出大小上限。** 第一个版本没有溢出处理。一次 `cat` 一个大文件就往上下文里塞了 200,000 个字符，撑爆了窗口，对话崩溃了。8,000 字符的阈值是通过生产数据校准的：我们统计了几千次工具调用的输出长度，发现 95% 的输出在 5,000 字符以内，99% 在 8,000 以内。超过 8,000 的几乎都是异常情况——大文件的全文输出、大型命令的执行结果。这个阈值足够大，正常的输出不会触发；足够小，异常值不会破坏上下文预算。

**给了太多工具。** 22 个工具全给 LLM，选错率明显高于只给 11 个核心工具时。少给选择，反而更好。这个教训不只适用于工具——在 Agent 设计的很多地方，"少即是多"都是一个可靠的指导原则。

**没有处理"工具名接近但不对"的情况。** LLM 有时候会调用一个很像但不完全对的工具名——比如 `read_file` 而不是 `file_read`，或者 `search` 而不是 `grep`。直接返回"工具不存在"太生硬了。后来我们加了模糊匹配：如果调用的工具名跟某个已注册工具的名字很接近（编辑距离 <= 2），错误信息会提示"你是不是想用 file_read？"。这一个改动减少了一类"LLM 调错名字 → 报错 → 道歉 → 调对名字"的浪费。

## 从工具到能力

六个字段、分层加载、并行执行、溢出处理、MCP 接入、子代理隔离——这些加在一起，就是一个完整的工具系统。

但工具系统解决的只是"Agent 能做什么"。它没有回答另一个同样重要的问题："Agent 做的事对不对？"

工具返回了一个结果，Agent 怎么判断这个结果是正确的？工具执行失败了，Agent 怎么决定是重试、换方法、还是放弃？工具输出了一个意外的结果，Agent 怎么调整计划？

还有一个更深层的问题：**工具之间的协作**。单个工具的使用很简单——调用、看结果、决定下一步。但当 Agent 需要协调多个工具完成一个复杂任务时，事情就变难了。它需要知道：哪些工具的输出可以被其他工具使用？哪些操作必须在其他操作之前完成？哪些工具的结果是相互矛盾的？

这些问题，我们会在下一章"当工具出错的时候"和后面的章节里详细讨论。工具系统是基础——有了基础，才能往上建。

---

## 本章小结

1. **工具是写给 AI 看的能力说明**，不是给人看的按钮。工具描述的质量直接决定了 Agent 的行为质量。

2. **六个字段就够了**：name, description, parameters, category, pure, execute。多余的复杂性（中间件、能力协商、自定义校验器）被证明弊大于利。

3. **description 比 execute 更重要**。LLM 根据描述决定"什么时候用"和"什么时候不用"。描述不好，再好的执行逻辑也没用。

4. **分层加载减少选择错误**。核心工具永远加载，条件工具按需加载。工具越多，LLM 选错的概率越高。

5. **纯工具可以并行，不纯工具必须串行**。一个 `pure` 标记 + 30 行批处理逻辑，多工具迭代明显更快。

6. **输出太大必须溢出**。8,000 字符阈值，完整内容存磁盘，LLM 只看预览。不要溢出错误信息，不要溢出溢出文件。

7. **子代理工具隔离**：最小权限原则，6 个工具从子代理列表里移除，防止递归、记忆污染和越权。
