# 第 2 章 最简 Agent：20 行代码背后的循环

---

## 十五次搜索之后

2026 年春节，AgentClaw 的第一个版本跑起来了。

那天晚上我特别兴奋。接上了一个大语言模型的 API，给它配了一个搜索工具，然后在聊天框里输入了第一句话："帮我调查一下 2025 年中国新能源汽车的出口情况。"

它开始搜索了。

搜完一次，觉得信息不够，又搜了一次。换了个关键词，又搜了一次。再换一个，又搜了一次。我看着屏幕上一行一行的搜索结果往上滚，心里觉得挺厉害——它在自己干活呢。像一个实习生，虽然笨拙但很努力。

十五次搜索之后，它停下来了。不是因为搜够了，是因为达到了我设置的迭代上限。它给我的回复是："信息太多了，无法整理成报告。"

十五次搜索。消耗了大量 token。最后告诉我它做不了。

我复盘了一下：问题出在哪？它每次搜索都"成功"了——搜索工具返回了结果，没有报错。但它不知道什么叫"信息够了"。它没有一个内部标准来判断"我已经搜集到了足够的数据，可以开始整理了"。它只知道"我还能再搜一次"，于是它就搜了。十五次。

这就像一个人去超市买东西，清单上写着"买牛奶"。他走到奶制品区，看到了牛奶，但觉得"再看看有没有更好的"。于是走到了另一个货架，又看到了牛奶。"再看看下一个。"逛了十五圈之后，他还是站在奶制品区，手里一瓶牛奶都没有。

这不是最糟的。最糟的是第二次测试：我让它"帮我读一下项目里的 README 文件"。它调了一个不存在的工具名——大概是 `read_file` 而不是 `file_read`。报错了。它又调了一次，同样的工具名。又报错了。又调了一次。连续调了八次，每次都是同样的错误，每次都在浪费一次 LLM 调用。

我盯着屏幕看了两分钟，等它自己停下来。它没有。最后我手动关掉了对话。

第三次更离谱。我让它"帮我改一下配置文件"。它生成了一段格式错误的 JSON 作为工具参数——少了一个引号。工具解析失败，报错信息被塞进了对话历史。下一轮，它看到了错误信息，试图"修复"——但修出来还是坏 JSON，这次是多了一个逗号。再下一轮，又修，这次是括号不匹配。三轮之后，整个对话上下文被垃圾信息填满了，Agent 开始胡说八道——它已经不知道我最初要它做什么了。

这三个故事有一个共同点：**Agent 跑起来了，但不知道什么时候该停。**

## 一个循环的诞生

上一章我们说过，Agent 的核心是一个循环：思考 → 行动 → 观察 → 再思考。20 行伪代码就能写出来。

但"写出来"和"能用"之间，隔着一道巨大的鸿沟。

先看这个循环到底长什么样。把 Agent 的一次完整工作流程拆开：

```
用户说："帮我查一下今天的天气"

第 1 轮：
  思考：用户想知道天气，我应该调用搜索工具
  行动：调用 web_search("今天北京天气")
  观察：搜索结果显示"晴，25°C"

第 2 轮：
  思考：我已经拿到了天气信息，可以直接回答了
  行动：无（不再调用工具）
  回复："今天北京天气晴朗，气温 25°C。"
```

两轮就结束了。简单任务，循环跑得干净利落。

再看一个稍微复杂一点的：

```
用户说："帮我读一下 package.json，看看用了哪些依赖"

第 1 轮：
  思考：用户想看 package.json 的内容，我应该读文件
  行动：调用 file_read("package.json")
  观察：返回了文件内容，包含 dependencies 和 devDependencies

第 2 轮：
  思考：我已经拿到了文件内容，可以整理成列表回复用户了
  行动：无
  回复："你的项目有 12 个运行时依赖和 8 个开发依赖……"
```

也是两轮。工具调用一次，拿到信息，整理回复。

但复杂任务不一样。同样是搜索，"帮我调查一个行业"可能需要：搜行业概况 → 读几个网页 → 搜竞争对手 → 读更多网页 → 搜政策变化 → 整理信息 → 写报告。六七轮下来，Agent 确实在一步步推进。每一轮都在前一轮的基础上做新的事情。

问题在于：**谁来判断它是在"推进"还是在"原地打转"？**

从外面看，"搜了行业概况再搜竞争对手"和"搜了行业概况再搜行业概况"长得差不多——都是在调搜索工具。但前者在推进，后者在打转。Agent 自己分不清这两者。LLM 每一轮都是独立的决策，它没有"我上一轮已经搜过这个了"的内在感知。

这个判断，就是循环控制（Loop Control）。它是 Agent 框架里最关键、也最容易被忽视的部分。

## 为什么"控制循环"比"启动循环"更重要

在第 1 章里，我们说过 Agent 和聊天机器人的区别在于"循环"。聊天机器人是一问一答，Agent 是接到目标后自己规划、执行、调整、再执行。

但有了循环不等于有了 Agent。就像有了发动机不等于有了一辆能上路的车。发动机让你能跑，刹车和方向盘让你能控制往哪跑、什么时候停。

一个最简 Agent 循环的核心逻辑，用伪代码写出来大概是这样：

```
while 迭代次数 < 最大限制:
    回复 = 调用LLM(系统提示词, 对话历史, 工具列表)

    if 回复里没有工具调用:
        输出最终回答
        结束

    for 每个工具调用:
        结果 = 执行工具(工具名, 参数)
        把结果加入对话历史
```

六行核心逻辑。加上变量声明和错误处理，也就二十行上下。

这二十行代码能让一个 Agent 跑起来——接上任何 LLM API，配上任何工具，它就能"做事"了。你可以用 Python 写，可以用 JavaScript 写，甚至可以用伪代码让 AI 编程工具帮你实现。核心逻辑是一样的。

但跑起来之后，你会遇到五种失控场景。每一种都是 AgentClaw 在真实使用中踩出来的，每一种都烧过真金白银。

---

## 失控一：不知道什么时候停

这是最常见也最浪费 token 的问题。

你让 Agent"优化这段代码"。它优化了一次，觉得可以再优化一次。优化完觉得还能再改改。改完觉得有个地方可以更好。十五轮之后，你的代码被改得面目全非，Agent 还在继续。

你让 Agent"搜索最新的 AI 新闻"。它搜了一次，觉得结果不够多。又搜了一次，用不同的关键词。又搜了一次。十次搜索之后，它消耗了大量 token，给你返回了一大堆它自己都整理不完的信息。

Agent 没有"够了"的概念。它不知道什么叫"差不多了"。它只知道"我还能再做一步"。

这个问题的根源不在模型——LLM 足够聪明，能做复杂推理。问题在于**循环没有刹车**。你给了它油门（工具），但没给它刹车（终止条件）。

打一个比方：一辆车有发动机但没有刹车。发动机很强大，能跑很快。但你不敢让它上路——不是因为发动机不行，是因为没有刹车。

Agent 的循环就是发动机。循环控制就是刹车。

---

## 失控二：重复同一个失败

你让 Agent 修一个 bug。它调了一个工具，工具报错。它又调了一次，同样的错误。又调了一次。又调了一次。八次同样的失败调用，每次都在浪费一次 LLM 调用。

为什么它不知道换个方法？

因为 LLM 的工作方式是"根据上下文决定下一步"。它看到了错误信息，但它没有"这个方法已经试过三次了"的记忆。每一次调用对它来说都是独立的决策。它可能会想："上次可能是网络抖动，再试一次就好了。"或者："可能是参数格式不对，我微调一下再试。"微调之后还是错的，它又想："再微调一下。"

人会从失败中学习——同一个错误犯两次就会烦。LLM 不会——除非你帮它记住。

AgentClaw 的经验：在没有失败追踪的早期版本中，重复失败调用是常态——Agent 反复尝试同一个注定失败的操作，浪费迭代预算和 token。加上失败追踪后，重复失败调用几乎降到了零。

更微妙的是，LLM 不只是重复完全相同的调用。它可能会"微调"参数——改一个引号、换一个路径、加一个参数。每次都不完全一样，但本质上在做同一件事。简单的"检查是否完全相同"抓不住这种模式。

---

## 失控三：格式错误污染上下文

这是最隐蔽的一种失控。

Agent 生成了一段格式错误的 JSON 作为工具参数。工具解析失败，返回了一个错误信息。这个错误信息被塞进了对话历史。下一轮，LLM 看到了这个错误，试图修复——但修出来的 JSON 可能还是错的。再下一轮，又修。三轮之后，对话历史里充满了坏 JSON 和错误信息。

这些垃圾信息占用了上下文窗口的空间，让 Agent 更难记住你最初的需求。更糟的是，LLM 可能会从这些错误中"学习"到错误的模式，开始生成更多格式错误的输出。

我见过最极端的情况：五轮格式错误之后，LLM 开始把"生成错误 JSON"当成一种正常的输出模式。它不再尝试生成正确的 JSON，而是直接生成一段带有 `Error:` 前缀的文字——因为它在历史里看到的全是这种格式。

---

## 失控四：子代理爆炸

这一种失控发生在更复杂的场景里——当你的 Agent 不只是一个人干活，而是能指挥别人干活的时候。

如果 Agent 能启动子代理（Sub-Agent）来处理子任务，事情会变得更复杂。

假设主代理有 15 次迭代的预算。它启动了一个子代理，子代理也有 15 次。子代理又启动了一个子代理，又是 15 次。三层嵌套，你实际上有 45 次迭代的预算——远超你的预期。

如果子代理之间还能并行，事情就更失控了。三个子代理同时跑，每个 15 次，总共 45 次。五个子代理，75 次。你根本不知道 Agent 到底烧了多少钱。

在 AgentClaw 的早期版本中，有一次子代理启动了三个子任务，每个子任务又各自启动了两个更小的子任务。整个调用树最终消耗了 67 次迭代。对一次普通对话来说，这已经是明显失控的 token 和时间消耗。

更要命的是，你很难提前预估一次对话会消耗多少迭代。用户说"帮我分析一下这个项目"，你不知道 Agent 会启动几个子代理、每个子代理会做几轮。没有共享预算，成本就是不可控的。

---

## 失控五：卡住了但没报错

这是最难检测的一种失控。

Agent 没有报错。它在正常执行工具。但它做的事情几乎没有进展。它读了一个文件，没找到想要的信息。换了一个文件，还是没找到。又读了第一个文件。又换了一个。

从技术角度看，每次工具调用都是成功的——没有错误码，没有超时。但从人的角度看，它明显卡住了。就像一个人在图书馆里来回走，每本书都翻了一下，但什么都没读进去。

传统的错误检测机制（检查错误码、检查超时）对这种"软卡住"无能为力。你需要一种更聪明的方式来识别它。

---

## 五层防护

这五种失控场景，不是理论推演，是 AgentClaw 在生产环境中一个个踩出来的。每一种失控都烧过钱、浪费过时间、让用户体验变差过。

最终，我们建了五层防护来解决它们。不是一次性设计出来的——是先被问题打脸，再一层层加上去的。

### 第一层：迭代上限 + 共享预算

最简单也最有效的一层：给循环设一个硬上限。

AgentClaw 的默认迭代上限是 15 次。这意味着不管任务多复杂，Agent 最多只能做 15 轮思考-行动-观察。简单的问答（1 轮，不需要工具）到复杂的多步骤操作（8-12 轮），15 次都够用了。

为什么是 15？不是拍脑袋决定的。我们观察了大量对话，发现绝大多数正常对话在十轮以内就能结束。超过 15 轮的对话，几乎都是出了问题——要么在循环，要么在失败重试。15 轮是"正常对话"和"失控对话"的分界线。

但单纯的上限在有子代理时不够用。如果主代理和子代理各有 15 次上限，嵌套起来就是 30 次、45 次。解决方案是**共享预算（Shared Budget）**。

打个比方：一个团队的任务预算是 100 个单位。父任务用了 30 个，子任务用了 20 个，另一个子任务用了 10 个——都从同一个预算里扣。不管谁用，总额不变。

迭代预算也是同一个道理。主代理创建一个预算对象，子代理拿到的是同一个对象的引用。子代理消耗一次迭代，主代理的预算也跟着减少。不管嵌套几层，总共就那么多。

举个例子：主代理创建预算 `max = 15`。

- 子代理 A 消耗 5 次，预算剩余 10。
- 子代理 B 消耗 3 次，预算剩余 7。
- 主代理自己消耗 6 次，预算剩余 1。
- 总共消耗 14 次，永远不会超过 15。

这个设计简单到只有一个计数器，但它彻底解决了子代理爆炸的问题。

还有一个细节值得说：加载技能定义（Skill Definition）不应该消耗预算。当 Agent 调用 `use_skill` 工具时，它只是在读一个指令文件，没有做任何"真正的"工作。AgentClaw 会把这种迭代"退还"给预算——最多退还 3 次。这样 Agent 不会因为加载技能而白白浪费迭代次数。

### 第二层：单工具失败追踪

迭代上限是"一刀切"——不管什么工具，到次数就停。但更精细的控制需要在工具层面做。

AgentClaw 为每个工具调用生成一个"去重键（Dedup Key）"——由工具名和参数的特征值（Fingerprint）组成。当同一个去重键失败两次，循环直接跳过这个调用，返回一条明确的指令："这个操作已经失败两次了，停止重试，换个方法。"

关键细节：**参数变了，计数器重置。** `ls -la` 失败两次后被拦截，但 `ls -la /tmp` 是一个新的调用，不受影响。这确保了 Agent 修正参数后还能继续尝试。Agent 不会因为一次失败就被永久禁止使用某个工具——它只是不能用同样的方式再试了。

另外还有两个全局计数器：

- **单工具总调用次数上限**：比如 `web_search` 最多 8 次，`web_fetch` 最多 8 次。这是整个会话的上限，不管参数是否相同。搜了 8 次之后，不管搜什么关键词都不让搜了——你已经搜够了，该用手头的信息了。

- **全局工具调用次数上限**：40 次。不管什么工具，不管成功失败，整个会话最多 40 次。这是最后的安全网。触发后，消息非常直接："你必须立即停止使用工具，用已有信息回复用户。"

这里有一个设计原则值得强调：**每条限制消息都必须是指令，而不是通知。** 不要告诉模型"达到限制了"——告诉它该做什么。"达到限制了"是一个状态描述，模型不知道该怎么办。"立即停止使用工具，用已有信息回复用户"是一个明确的指令，模型会照做。

### 第三层：格式错误回滚

当一轮迭代里所有的工具调用都因为格式错误失败时（JSON 解析失败、工具名不存在），循环会做一件事：**把这轮从对话历史里删掉。**

LLM 看到的就好像这轮从未发生过。迭代计数器也减回去。然后进入下一轮，LLM 重新来过。

这就像打字时按了退格键——你不会惩罚一个人打错了一个字，你让他重新打就行了。格式错误是 LLM 的"打字错误"——给它一个干净的状态，它大概率会自我纠正。

但回滚不能无限进行。连续回滚三次后，错误信息就保留在历史里，LLM 必须自己面对它们。三次都纠正不了，说明问题不只是"打错了"，而是"不会打"——这时候需要 LLM 看到错误才能理解问题在哪。

回滚计数器会在任何一次成功的工具调用后重置。只有连续的格式错误才会触发回滚。

### 第四层：三击升级

这是最精巧的一层防护，专门对付"卡住了但没报错"的情况。

检测方法是这样的：每一轮迭代结束后，把 LLM 的输出做一个"指纹"——取前 300 个字符加上调用的工具名列表。最近三轮的指纹放在一个滑动窗口里。如果三轮的指纹相似度超过 80%，就认为 Agent 卡住了。

为什么用指纹而不是完整比较？因为完整的 LLM 输出可能有几百个字，逐字比较成本高而且容易被无关细节干扰。前 300 个字符加上工具名，已经足够捕捉"这轮做了什么"的本质特征。如果三轮的本质特征都差不多，那 Agent 一定是在重复。

触发后，循环会注入一条策略切换指令：

> "你似乎陷入了循环——最近三次响应非常相似。停止重复同样的方法，尝试完全不同的策略：使用不同的工具、改变参数、或者向用户说明你遇到了什么障碍。"

这条指令只触发一次——用一个布尔标志位防止重复触发。因为卡住的 Agent 不需要更多迭代——它需要的是换一个视角。给它一次机会就够了。

### 第五层：连续全错自动停止

最后一道防线：如果连续三轮迭代里，每个工具调用都失败了（不是格式错误，是真的执行失败），循环立即停止。不等 LLM 再想一个方案，不等它再试一次。

这抓住了单工具失败追踪漏掉的情况：Agent 在三个不同的工具之间轮换，每个工具都失败，但没有任何一个达到自己的失败上限。整体来看是绝望的，但局部看每个工具都"还有机会"。

连续全错检测跳过局部，看全局：三轮全是错的，那就别再试了。

## 五层防护是怎么长出来的

这五层不是一次性设计出来的。回头看 AgentClaw 的提交记录，每层都是被一个具体的失控场景"打脸"后加上去的。

**第一层（迭代上限）** 是第一天就加的。因为第一次测试就遇到了无限搜索的问题。这是最基本的防护，没有它一切无从谈起。

**第二层（单工具失败追踪）** 是第三周加的。当时发现 Agent 会反复调用同一个失败的命令——`ls /not/exist` 调了八次。单纯靠迭代上限不够，因为它可能调了八次才到上限，每次都浪费一次 LLM 调用。

**第三层（格式错误回滚）** 是第二个月加的。之前遇到格式错误就让它留在历史里，结果发现 LLM 会试图"修复"错误，但修出来还是错的，陷入"生成坏 JSON → 看到错误 → 修复 → 还是坏 JSON"的死循环。回滚让它从干净的状态重新开始。

**第四层（三击升级）** 是后来加的。有一次用户反馈说 Agent "很努力但没有进展"——它在读文件、搜索、再读文件，每个操作都成功了，但三轮下来做的事情几乎一样。传统的错误检测抓不住这种情况，需要比较输出的相似度。这个方案的灵感来自人类的"重复检测"直觉——你看到一个人反复做同一件事，不需要任何技术手段就知道他卡住了。

**第五层（连续全错）** 是第四个月加的。有一次 Agent 在三个工具之间轮换，每个工具都失败，但没有任何一个达到自己的失败上限。三轮下来全是错的，但系统还以为"每个工具都还有机会"。这个场景很微妙——从单个工具的角度看，每个都"还有希望"；从全局的角度看，已经没救了。

每加一层，都减少了一类用户投诉。这就是真实工程的样子——你不可能一开始就想到所有问题。但你可以设计一个可以层层加防护的架构。每一层解决一类问题，层与层之间互不干扰。新加一层不需要改旧的层。这就是好的架构应该有的样子。

## 循环的可见性

一个看不见的循环是没法调试的。

AgentClaw 在循环的每个状态变化时都会发出结构化事件：

比如一次两轮任务，会产生这样的事件序列：

- 第 1 轮迭代：
  - `thinking`：LLM 调用开始。
  - `response_chunk`：流式输出的文本片段。
  - `tool_call`：调用 `grep` 查找 auth 中间件。
  - `tool_result`：找到 3 个匹配，耗时 45ms。
  - `tool_call`：调用 `file_read` 读取匹配文件。
  - `tool_result`：返回 120 行，耗时 12ms。
- 第 2 轮迭代：
  - `thinking`：LLM 再次调用。
  - `response_chunk`：继续输出。
  - `response_complete`：最终回答，2 轮迭代，1,847 tokens。

每个事件都带着时间戳。网关通过 WebSocket 把这些事件转发给前端，用户可以实时看到 Agent 在做什么——调了什么工具、返回了什么结果、花了多长时间。

同样的事件也喂给了追踪系统（Trace System），方便事后复盘。当 Agent 做出奇怪的决策时，你可以在追踪系统里回放整个过程，看它是在第几轮开始偏离正轨的。

**循环的可见性不是锦上添花，是生存必需。** 你看不见循环，就看不见问题。看不见问题，就无法改进。

没有追踪系统的时候，我们只能靠用户反馈来发现问题。用户说"Agent 不好用"，我们不知道具体是哪里不好用。有了追踪系统之后，我们可以看到每一次迭代的细节——调了什么工具、花了多少时间、返回了什么结果、LLM 做了什么决策。这让调试从"猜"变成了"看"。

有一次，一个用户反馈说 Agent "回复很慢"。我们查了追踪系统，发现 Agent 在第 1 轮就搜到了足够的信息，但在第 2 轮又搜了一遍（用略有不同的关键词），第 3 轮又搜了一遍。三轮搜索花了 40 秒。如果它在第 1 轮就停下来整理结果，只需要 15 秒。

没有追踪系统，我们只会听到"回复很慢"。有了追踪系统，我们看到的是"第 2、3 轮搜索是多余的"。这两种信息的质量天差地别。

## 我们试过但删掉的方案

在建这五层防护的过程中，我们也走了一些弯路。

**让 LLM 自己管理迭代次数。** 我们试过在系统提示词里告诉模型"你还有 10 次迭代"。结果：模型完全忽略了它。有时候它会在第 3 次迭代就告诉用户"我的迭代次数快用完了"，即使实际上还有 12 次。有时候它会一直跑到上限都不提迭代次数。LLM 不擅长管理自己的资源——这件事必须由系统层面来做。

**每轮迭代都问 LLM "你卡住了吗？"。** 多一轮 LLM 调用就多一份成本和延迟。而且 LLM 判断"是否卡住"的准确率远不如简单的输出相似度比较。指纹方案零成本，且抓住了同样的问题。

**激进的全局超时。** 给每个工具调用设 10 秒超时。结果杀死了合理的长时间操作——读一个大文件可能需要 15 秒，等一个 API 响应可能需要 20 秒。后来改成了工具级别的差异化超时：shell 命令 120 秒，web 请求 30 秒，文件操作 15 秒。

这些被删掉的方案不是"错的"——它们在特定场景下也能用。但它们要么成本太高，要么不够可靠，要么被更简单的方案替代了。**好的工程不是找到最聪明的方案，是找到最简单且够用的方案。**

## AgentClaw 的循环骨架

如果你去看 AgentClaw 的源码（`packages/core/src/agent-loop.ts`），核心循环的骨架大概是这样的：

```typescript
while (iterations < config.maxIterations && !aborted) {
  // 检查共享预算
  if (iterationBudget?.exhausted) break;
  iterations++;
  iterationBudget?.consume();

  // THINK：问 LLM 下一步做什么
  const response = await provider.stream(systemPrompt, messages, tools);

  // 如果没有工具调用，任务完成
  if (response.toolCalls.length === 0) {
    yield createEvent("response_complete", { text: response.text });
    return;
  }

  // ACT：执行每个工具调用
  for (const toolCall of response.toolCalls) {
    const result = await toolRegistry.execute(toolCall.name, toolCall.input);
    // OBSERVE：把结果加入对话历史
    messages.push({ role: "tool", content: result });
  }
}
```

骨架就这么多。但围绕这个骨架，有大约 500 行的防护代码——迭代预算、失败追踪、格式回滚、相似度检测、溢出处理、事件发射。这些"外围代码"才是 Agent 能在生产环境稳定运行的原因。

**启动一个循环需要 20 行代码。让这个循环可靠运行需要 500 行。**

这就是为什么"控制循环"比"启动循环"更重要。

---

## 从一个循环到一个系统

20 行代码让 AI 能做事。500 行防护代码让 AI 做事不出大问题。

但"不出大问题"和"做得好"之间还有距离。500 行防护代码解决的是"别崩溃、别浪费 token、别死循环"——这是底线，不是天花板。

一个真正好用的 Agent 还需要解决更多问题。比如：

- Agent 做完一件事后，怎么把结果告诉用户？直接输出文字？还是保存成文件？还是发一条消息？
- Agent 需要做一件它现在不能做的事（比如读数据库），怎么办？放弃？还是问用户要一个新工具？
- 用户给了一个很长的任务，Agent 的上下文窗口装不下了，怎么办？
- Agent 做了三轮之后，它之前读的文件内容还在不在？还在的话，上下文会不会被撑满？

这些问题，我们会在后面的章节一一解答。下一章，我们先解决最基本的问题：**Agent 有哪些"工具"可以用？** 给 AI 装上手和脚，是让它从"能说话"变成"能做事"的第一步。

---

## 本章小结

1. **最简 Agent 循环只有三步**：调用 LLM → 执行工具 → 把结果反馈给 LLM。20 行代码就能跑起来。

2. **启动循环容易，控制循环难。** 五种失控场景（无限调用、重复失败、格式污染、子代理爆炸、软卡住）是真实生产中一个个踩出来的。

3. **五层防护是递进的**：迭代上限兜底，单工具追踪精细控制，格式回滚保持干净，相似度检测识别卡住，连续全错及时止损。

4. **共享预算解决子代理爆炸**：父代理和子代理共享同一个计数器，不管嵌套几层，总数不变。

5. **循环必须可见**：结构化事件让你能实时看到 Agent 在做什么，事后能回放复盘。

6. **每条限制消息都必须是指令**：告诉模型"该做什么"，而不是"发生了什么"。LLM 遵循指令——模糊的消息产生模糊的行为。
