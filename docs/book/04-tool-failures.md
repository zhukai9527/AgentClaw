# 第 4 章 当工具出错的时候

---

## 一个死循环的故事

2026 年初的一个晚上，我盯着 AgentClaw 的日志，看着一个数字缓慢但坚定地往上跳。

1，2，3，4，5……

事情是这样的。我让 Agent 帮我调查某个行业的市场情况。它开始工作了——搜索网页、读取结果、再搜索、再读取。看起来很能干。

然后搜索引擎返回了一个错误：请求太频繁，被限速了（Rate Limit）。

Agent 收到了这个错误。它的反应是什么？**再试一次。**

同样的关键词，同样的搜索工具，同样的错误。

**再试一次。**

1，2，3，4，5……一直试到了第 15 次。

每一次调用都在消耗 token。15 次无效调用，加上 Agent 每次收到错误后的"思考"过程，token 消耗远超预期。更糟糕的是，这些失败的调用和错误信息全部留在了对话历史里。下一轮，Agent 看到了一大堆错误记录，开始"分析"为什么失败——但它的分析是错的。它以为是关键词不对，于是换了关键词，但问题其实是限速。换关键词没用，继续失败，继续浪费 token。

**这不是模型的问题。这是系统设计的问题。**

一个没有错误处理的 Agent，就像一辆没有刹车的汽车。平路上跑得挺快，一遇到下坡就失控。而"工具出错"是 Agent 世界里最常见的下坡。

---

## 工具会出错，而且经常出错

AgentClaw 运行一段时间后，我翻了一遍工具调用的日志，发现失败比想象中频繁得多。

失败大致可以分成五类：

| 失败类型 | 典型场景 |
|---------|---------|
| 工具不存在 | Agent 调用了一个没注册的工具名 |
| 输出太大 | `find` 命令返回了 50,000 字符 |
| 网络失败 | 搜索引擎超时、API 返回 503 |
| 参数错误 | Agent 传了错误的参数格式 |
| 重复调用 | Agent 反复调用同一个失败的工具 |

每一类都需要不同的处理策略。但 AgentClaw 的第一个版本只有一种处理方式：把错误信息原样返回给 Agent，让它自己决定怎么办。

结果？Agent 的"决定"通常是：**再试一次。**

---

## 一个没有错误处理的世界

让我先描述一下"没有错误处理"的世界是什么样的，这样你才能理解为什么需要错误处理。

**场景一：工具不存在。**

Agent 决定调用一个叫 `web_scraper` 的工具。但系统里注册的工具叫 `web_fetch`，不叫 `web_scraper`。工具注册表找不到这个名字，返回错误：`Tool not found`。

Agent 收到错误，想了想，决定再试一次：`web_scraper`。同样的错误。再试：`web_scraper`。又错了。它可能试了 5 次才"想起来"应该用 `web_fetch`。也可能试了 10 次都没想起来，最后放弃，告诉用户"我做不到"。

**场景二：输出太大。**

Agent 执行 `find . -name "*.ts"` 命令，搜索项目里所有 TypeScript 文件。项目很大，返回了 50,000 个字符的文件列表。这些字符全部被塞进了对话历史。

下一轮调用 LLM 时，对话历史已经膨胀到了几十 KB。LLM 需要处理这么多信息，响应变慢，token 消耗暴增。更糟糕的是，LLM 可能会从这个巨大的列表中"幻觉"出一些不存在的文件名。

**场景三：网络失败。**

Agent 调用 `web_search` 搜索最新新闻。搜索引擎返回 503 Service Unavailable。Agent 收到错误，决定再试一次。搜索引擎还是 503。再试。还是 503。15 次之后，Agent 终于放弃，告诉用户"搜索失败了"。

这 15 次尝试等了 3 分钟，烧了一堆 token，最后什么都没做。

**场景四：参数错误。**

Agent 调用 `file_read` 工具，但传的参数是 `{"file": "data.txt"}`，而工具期望的参数名是 `{"path": "data.txt"}`。工具返回错误：`Missing required parameter: path`。

Agent 收到错误，想了想，决定再试一次。这次它可能传对了，也可能换了一个错误的参数名。如果它传对了，浪费了一轮。如果它又传错了，再浪费一轮。

**场景五：重复调用。**

Agent 调用 `web_search("2025年新能源汽车销量")`，得到了一个结果。然后它又调用了一次 `web_search("2025年新能源汽车销量")`——完全相同的关键词，完全相同的结果。然后又调用了一次。为什么要调三次？因为 Agent 的推理过程是概率性的，它可能在每一轮都"决定"这个搜索是有用的。

这五种场景加在一起，就是一个没有错误处理的 Agent 的日常：**它会浪费大量时间和金钱，做大量无效的工作，最后给用户一个不好的结果。**

---

## 错误不是"失败了"，是"下一步该做什么"

这是本章最重要的一个判断。

传统的错误处理思路是：出错了 → 返回错误信息 → 让调用方决定怎么办。

在 Agent 的世界里，这个思路是错的。

因为 Agent 的"调用方"是 LLM。LLM 看到一个错误信息，它的反应不是"分析错误原因，制定修复方案"，而是"根据上下文生成一段看起来合理的文字"。这段文字可能是"让我再试一次"，也可能是"让我换个方式试"，也可能是"看起来有问题，我放弃"。

你没法控制 LLM 会怎么"解读"一个错误。但你可以控制 LLM 看到什么。

**错误信息不是"通知"，是"指令"。**

一个好的错误信息不应该只说"失败了"。它应该告诉 Agent：
- 为什么失败了
- 下一步应该做什么
- 不应该做什么

比如，当 Agent 调用一个不存在的工具时，错误信息不应该是：

```
Tool "web_scraper" not found.
```

而应该是：

```
Tool "web_scraper" does not exist. You can ONLY use these tools:
shell, file_read, file_write, file_edit, glob, grep, ask_user,
web_fetch, web_search. Use web_fetch for scraping web pages.
```

第一种错误信息让 Agent 自己猜"我应该用什么工具"。第二种直接告诉它"你只能用这些工具，如果你想抓网页，用 web_fetch"。

区别不大？在 LLM 的世界里，区别巨大。第一种信息给了 LLM 自由发挥的空间——它可能猜对，也可能猜错，也可能反复猜错。第二种信息把自由发挥的空间压缩到了最小——它只能从给定的列表里选。

**约束 LLM 的选择空间，比提升 LLM 的能力更有效。**

这不是我的观点，这是实践告诉我们的。AgentClaw 把错误信息从"通知"改成"指令"之后，无效重试的现象明显减少了。

## AgentClaw 的四层错误防护

现在让我介绍 AgentClaw 是怎么处理工具错误的。不是一层，是四层。每一层都在拦截前一层漏掉的问题。

### 第一层：自动重试

有些失败是暂时的。搜索引擎偶尔会超时，API 偶尔会返回 503，DNS 偶尔会抖动。这些失败如果等 Agent 自己决定重试，它可能会换关键词、换工具、或者干脆放弃——都不是正确的做法。

正确的做法是：**在 Agent 看到错误之前，框架自动重试。**

AgentClaw 对网络类工具（`web_search`、`web_fetch`）实现了自动重试机制。失败后等 2 秒重试一次，如果还失败就等 4 秒再试一次。两次都失败了，才把错误交给 Agent 处理。

这叫"指数退避"（Exponential Backoff）——每次重试的等待时间翻倍。这个策略在分布式系统里用了几十年，因为它是处理暂时性网络问题最有效的方式。

关键是：**Agent 完全不知道这次重试发生了。**从它的视角看，工具就是正常返回了结果（或者在两次重试后返回了真正的错误）。这避免了 Agent 看到错误后"过度反应"。

### 第二层：重复检测

有些失败不是暂时的。搜索引擎不是偶尔超时，而是被限速了。Agent 自动重试两次后，把错误交给了 Agent。Agent 的反应？换个说法再试一次。

`web_search("2025年新能源汽车销量")` → 失败
`web_search("新能源汽车 2025 销量数据")` → 失败
`web_search("2025 new energy vehicle sales")` → 失败

关键词变了，但本质是一样的：搜索引擎被限速了，不管用什么关键词都搜不了。

AgentClaw 的第二层防护是"重复检测"（Duplicate Detection）。它会记住每次工具调用的"签名"——工具名加上参数的哈希值。如果同一个签名在短时间内出现了多次（默认 2 次），框架会拦截后续调用，返回一条"指令"：

```
You have already called web_search with similar parameters 2 times
and it failed both times. DO NOT retry this tool. Instead, tell the
user that the search service is temporarily unavailable and suggest
trying again later.
```

这条信息做了三件事：
1. 告诉 Agent 它已经试过了（不是新信息）
2. 明确说"不要再试"（约束行为）
3. 给出下一步建议（告诉用户情况）

重复检测的签名比你想的要复杂一些。对于 `shell` 工具，它不是简单地对整个命令做哈希——因为不同的命令是不同的操作，不应该被去重。它只取命令的前 80 个字符作为签名的一部分。这样 `ls -la` 和 `ls -la /tmp` 会被识别为不同的调用，但 `ls -la` 和 `ls -la` 会被识别为重复。

对于 `file_read` 工具，有一个特殊的边界情况：当文件内容超过溢出阈值时，框架会把内容存到一个临时文件里，然后返回预览。如果 Agent 读了这个临时文件，结果又超过了阈值……就会产生无限循环。所以 `file_read` 对溢出文件的读取做了归一化——不管读哪个溢出文件，签名都是一样的。

### 第三层：单工具失败上限

有些失败不是重复调用造成的，而是 Agent 每次都换一种方式尝试。

比如，Agent 想写入一个文件，但路径不存在。它试了 `file_write("/a/b/c.txt", ...)`，失败。它想了想，改成 `file_write("/a/b/d.txt", ...)`，还是失败。又改成 `file_write("/a/b/e.txt", ...)`，还是失败。

参数变了，所以重复检测不会拦截。但问题的根源是一样的：Agent 不知道 `/a/b/` 目录不存在。

AgentClaw 的第三层防护是"单工具失败上限"（Per-Tool Failure Cap）。它为每个唯一的"工具+输入"组合维护一个失败计数器。不是简单的签名匹配，而是更宽泛的匹配——同一个工具，不管参数怎么变，如果连续失败了 2 次，就触发拦截。

触发后返回的信息：

```
file_write has failed 2 times. You are NOT allowed to retry this
tool. The likely issue is that the target directory does not exist.
Use shell("mkdir -p /a/b") first, then retry file_write.
```

注意这条信息不只是说"别试了"，它还**猜了失败的原因**（目录不存在），并**给出了修复方案**（先创建目录）。

这个猜测不总是对的。但即使是错的，它也给了 Agent 一个明确的方向去尝试，而不是让它在黑暗中摸索。

### 第四层：全局安全网

前三层已经拦截了大部分问题。但总有一些模式是它们抓不住的。

比如，Agent 在一次对话中调用了 10 种不同的工具，每种都失败了 1 次。重复检测没触发（没有重复），单工具失败上限也没触发（每种只失败了 1 次）。但总失败次数已经很高了。

或者，Agent 没有失败，但它在做一件"不应该做的事"——比如反复搜索同一个主题，每次都用不同的关键词，每次都成功了，但信息已经足够了。

AgentClaw 的第四层防护是"全局安全网"（Global Safety Net）。它对每次用户消息触发的所有工具调用设置了一个硬上限：**40 次。**

不管是什么工具，不管成功还是失败，不管参数怎么变——一次用户消息最多触发 40 次工具调用。达到上限后，Agent 会被强制要求立即回复用户，不能再调用任何工具。

40 次是怎么来的？这是一个经验值：大多数正常对话在十几次工具调用内就能完成，但有些复杂任务（搜索 + 读取多个网页 + 写报告）需要更多空间。40 次既能给复杂任务留余地，又能拦截失控的循环。

这是一个"熔断器"（Circuit Breaker）——平时不起作用，但当其他机制都失效时，它是最后一道防线。

## 输出太大：当工具说太多了

错误处理不只是"工具失败了怎么办"。还有一种情况是"工具成功了，但结果太大了"。

一个真实的例子：Agent 执行 `find . -name "*.ts" -exec wc -l {} +`，统计项目里每个 TypeScript 文件的行数。项目有 200 多个文件，命令返回了 47,000 个字符的输出。

如果把这 47,000 个字符全部塞进对话历史，会发生什么？

**第一，token 暴增。**47,000 个字符大约是 10,000-15,000 个 token。一次工具调用就吃掉了上下文窗口的 10-15%。

**第二，LLM 处理质量下降。**LLM 对长文本的注意力是不均匀的。它对开头和结尾的内容记得比较清楚，中间的内容容易"遗忘"或"幻觉"。47,000 字符的输出，LLM 大概率只会看前几百行和后几百行，中间的内容基本被忽略。

**第三，后续对话被污染。**这 47,000 字符会一直留在对话历史里，每一轮 LLM 调用都要重新处理它。即使 Agent 已经不需要这个文件列表了，它还是会占用上下文空间。

AgentClaw 的解决方案叫"溢出模式"（Overflow Mode）。规则很简单：

**当工具输出超过 8,000 个字符时，框架自动做三件事：**
1. 把完整输出保存到一个临时文件
2. 把工具结果替换成 1,500 字符的预览（取前 1,500 个字符，在最后一个换行符处截断）
3. 告诉 Agent 文件路径，建议它用 `file_read` 或 `grep` 按需查看

以 47,312 字符、2,847 行的工具输出为例，溢出后的结果是：

- LLM 收到：前约 1,500 字符的预览，在换行处截断。
- 磁盘保存：完整内容写入 `data/tmp/overflow_170234.txt`。
- 提示信息：如果需要完整内容，可以用 `file_read` 或 `grep` 查看。

这个设计有一个关键判断：**Agent 通常不需要看完整的输出。**它需要的是"这个输出大概是什么样的"加上"能按需深入查看"的能力。1,500 字符的预览给了它足够的上下文来理解输出的结构，而文件路径让它能在需要时精确地查看某个部分。

在实践中，Agent 的行为验证了这个判断：它通常会用 `grep` 在溢出文件里搜索它需要的信息，而不是把整个文件读出来。这正是我们想要的行为。

但有两个边界情况是我们"踩坑"之后才学到的：

**坑一：永远不要溢出错误信息。**

错误信息通常很短，但信号密度很高。如果一个工具失败了，Agent 需要看到完整的错误信息（包括堆栈跟踪）才能判断问题出在哪。我们一开始把溢出逻辑写成了"所有输出都检查"，结果有一次工具报了一个很长的错误（包含完整的依赖树冲突信息），被截断了，Agent 看到截断的错误信息后做出了完全错误的判断。

修复很简单：如果 `isError` 为 `true`，跳过溢出检查。

**坑二：永远不要溢出溢出文件的读取。**

想象这个循环：Agent 读了一个大文件 → 输出溢出 → 内容存到了临时文件 → Agent 读这个临时文件 → 输出又溢出 → 又存到一个新的临时文件 → Agent 又读 → 又溢出……

无限循环。

修复也很简单：如果被读取的文件路径包含 `overflow_` 前缀，跳过溢出逻辑，直接返回完整内容。

## 工具不存在：当 Agent 说了你听不懂的话

还有一种常见的错误：Agent 调用了一个不存在的工具。

这比你想的更常见。LLM 是概率模型，它生成工具名的方式和生成自然语言一样——根据上下文猜测最可能的下一个 token。有时候它会"猜"出一个听起来合理但实际上不存在的工具名。

比如，系统里有一个 skill（技能）叫 `agent-browser`，它是一个描述文件，告诉 Agent 怎么使用浏览器。但 skill 不是 tool——你不能直接调用它，你需要先用 `use_skill` 工具加载它。

Agent 分不清这个区别。它直接调用 `agent-browser` 这个"工具"。工具注册表找不到这个名字，返回错误。

AgentClaw 的处理方式不是简单地返回"工具不存在"。在返回错误之前，它会做一个额外的检查：**这个名字是不是一个已注册的 skill？**

如果是，错误信息会自动重定向：

```
"agent-browser" is not a tool, but it is a registered skill.
Use use_skill("agent-browser") to load it, then follow the
instructions in the next turn.
```

这一条检查消除了整整一类浪费：Agent 调用 skill 名 → 失败 → 道歉 → 调用 `use_skill` → 成功。四轮对话变成了零轮——因为 Agent 在第一次就得到了正确的指令。

如果名字既不是工具也不是 skill，错误信息会包含完整的可用工具列表：

```
Tool "xxx" does not exist. You can ONLY use: shell, file_read,
file_write, file_edit, glob, grep, ask_user, web_fetch, web_search,
send_file, schedule, remember, use_skill, subagent, ...
```

为什么要给完整列表？因为 LLM 的"记忆"是上下文驱动的。它不是真的"记住"了有哪些工具——它是在每一轮根据上下文重新"推断"。如果上下文里没有工具列表，它就可能推断出一个不存在的工具名。把列表放在错误信息里，等于把"事实"重新注入了上下文。

## 格式错误：当 Agent 说错了话

还有一种错误更微妙：Agent 调用工具的方式本身是对的（工具名正确，参数名也正确），但参数的格式是错的。

比如，`file_read` 期望的参数是 `{"path": "data.txt"}`，但 Agent 传的是 `{"file": "data.txt"}`。参数名错了。

或者，`shell` 期望的参数是 `{"command": "ls -la"}`，但 Agent 传的是 `{"command": ["ls", "-la"]}`。参数类型错了（应该是字符串，不是数组）。

这些错误通常发生在 Agent 第一次使用某个工具的时候。它"猜"了一个参数格式，猜错了。

AgentClaw 的处理方式是：**在错误信息里给出正确的参数格式。**

```
Invalid parameter for file_read: "file" is not a valid parameter.
Expected: {"path": "<string>"}
Example: file_read({"path": "data.txt"})
```

这比"Invalid parameter"有用得多。Agent 看到这个信息后，通常能在下一轮正确地调用工具。

---

## 错误信息设计的三个原则

总结一下，AgentClaw 的错误信息设计遵循三个原则：

**原则一：告诉 Agent "不要做什么"。**

LLM 的默认行为是"再试一次"。错误信息必须明确打破这个默认。"DO NOT retry"、"You are NOT allowed"、"Stop trying"——这些否定词是必要的。不要委婉，不要客气。LLM 不会因为你说了"please"就更听话。

**原则二：告诉 Agent "应该做什么"。**

只说"不要做什么"是不够的。Agent 需要一个明确的替代方案。"Use web_fetch instead"、"Create the directory first"、"Tell the user the service is unavailable"——具体的下一步指令。

**原则三：提供纠正信息。**

错误信息不只是"你错了"，还应该是"正确的长什么样"。可用工具列表、正确参数格式、示例调用——这些信息帮助 Agent 在下一轮做出正确的选择。

这三个原则加在一起，把错误从"Agent 的困惑"变成了"Agent 的指引"。

---

## 一个被删掉的设计：中间件链

在 AgentClaw 的早期版本里，工具系统有一个"中间件链"（Middleware Chain）的设计。

想法是这样的：每个工具调用在执行之前，要经过一系列中间件——日志记录、权限检查、参数校验、错误处理、重试逻辑……每个中间件做一件事，串成一条链。

```
工具调用 → 日志中间件 → 权限中间件 → 校验中间件 → 执行 → 错误处理中间件 → 重试中间件 → 返回
```

这个设计在传统软件工程里很常见。Express.js、Koa、Django 都用中间件。看起来很优雅：每个中间件职责单一，可插拔，可组合。

但在 Agent 的世界里，它失败了。

原因有几个：

**第一，调试困难。**当一个工具调用失败时，你需要搞清楚是哪个中间件出了问题。是日志中间件吞了异常？是权限中间件误判了？是校验中间件的规则写错了？链越长，排查越难。

**第二，性能开销。**每个中间件都要执行一遍，即使它什么都没做。Agent 的工具调用频率很高（一次对话可能有几十次调用），中间件的累积开销不可忽视。

**第三，也是最重要的：Agent 的错误处理不需要那么灵活。**AgentClaw 的四层错误防护是固定的——自动重试、重复检测、失败上限、全局安全网。这四层逻辑不会因为不同的工具而改变。把它们写成中间件，只是增加了一层抽象，没有任何实际好处。

我们删掉了中间件链，把错误处理逻辑直接写在了工具执行流程里。代码更直接，更容易理解，更容易调试。

**教训：不要把传统软件工程的模式直接搬到 Agent 系统里。Agent 的约束和传统软件不同，最优解也不同。**

---

## 错误处理的成本

错误处理不是免费的。每一层防护都有成本。

**自动重试的成本：**延迟。每次重试等 2-4 秒，两次重试最多等 6 秒。对于网络类工具，这个延迟是可以接受的——搜索引擎偶尔超时，等几秒再试通常能成功。但如果重试逻辑被用在了非网络工具上（比如 `file_read`），6 秒的等待就是浪费——文件不存在，等多久都不存在。

所以 AgentClaw 只对网络类工具启用自动重试。`file_read`、`shell`、`file_write` 这些本地工具不重试。

**重复检测的成本：**内存。框架需要维护一个哈希表，存储最近的工具调用签名。这个哈希表很小（一次对话最多几十个条目），但在高并发场景下需要考虑清理策略。

**失败上限的成本：**灵活性。有时候 Agent 的重试是合理的——比如它第一次传了错误的参数，第二次修正了。失败上限可能会误杀这种"有价值的重试"。

AgentClaw 把失败上限设为 2 次，就是为了在"拦截无效重试"和"允许有价值的重试"之间找到平衡。2 次是一个经验值——第一次失败可能是偶然，第二次失败说明问题比较顽固，第三次尝试大概率不会成功。

**全局安全网的成本：**中断复杂任务。40 次工具调用的上限，对大多数对话足够了。但对于某些特别复杂的任务（比如"帮我分析这个项目的代码质量，然后写一份详细报告"），可能需要更多调用。

这是一个两难选择：上限太低会误杀复杂任务，上限太高会让失控循环烧更多钱。AgentClaw 选择了 40 次作为默认值，并且在触发上限时给 Agent 一条"温和"的指令：

```
You have reached the maximum of 40 tool calls for this message.
Please provide your best response to the user based on what you
have gathered so far.
```

不是"停下来"，而是"基于你已经收集到的信息，给出你能给的最好回答"。这让 Agent 能"体面地"结束任务，而不是突然中断。

---

## 错误处理的效果

AgentClaw 从"没有错误处理"到"四层错误防护"的改造，效果是肉眼可见的：

- 无效重试大幅减少——Agent 不再像以前那样反复撞墙
- 每次对话的工具调用数降了下来——因为不再浪费调用在注定失败的重试上
- "Agent 卡住"的投诉几乎消失了——用户不再看到 Agent 转了好几圈最后说"我做不到"

错误处理看起来是一个"无聊"的工程问题，但它的影响是实实在在的——降低 token 消耗、节省时间、提升用户体验。

---

## 本章小结

1. **工具错误是 Agent 的常态，不是异常。**没有错误处理机制的 Agent，会在失败上浪费大量时间和 token。

2. **错误信息不是"通知"，是"指令"。**告诉 Agent "不要做什么"和"应该做什么"，比只说"失败了"有效得多。

3. **四层防护：自动重试 → 重复检测 → 失败上限 → 全局安全网。**每一层拦截前一层漏掉的问题。

4. **输出太大也是问题。**溢出模式把大输出存到磁盘，只给 Agent 看预览。

5. **约束 LLM 的选择空间，比提升 LLM 的能力更有效。**给它完整工具列表比让它猜更可靠。

6. **不要照搬传统软件的模式。**中间件链在 Agent 系统里不一定是好设计。

## 动手试一试

**任务 1：设计三条错误信息。**

想象你有一个 Agent，它有三个工具：`search`（搜索网页）、`read_file`（读取文件）、`run_code`（执行代码）。

为以下三种错误场景各写一条错误信息，遵循"告诉 Agent 不要做什么 + 应该做什么 + 提供纠正信息"的原则：

- 场景 A：Agent 调用了一个叫 `web_crawler` 的工具，但这个工具不存在。
- 场景 B：Agent 调用 `search("某个关键词")` 连续失败了 3 次。
- 场景 C：Agent 执行 `run_code` 后，输出了 50,000 个字符。

写完后，让 AI 编程工具评价你的错误信息是否足够清晰、是否包含了"不要做什么"和"应该做什么"两个要素。

**任务 2：用 AI 编程工具实现一个简单的重复检测器。**

提示词参考：

> "写一个 TypeScript 函数 `isDuplicate(toolName: string, params: object, recentCalls: Map<string, number>): boolean`。功能：检查最近是否已经有相同的工具调用。如果同一个 toolName + params 的组合在 recentCalls 中出现过，返回 true。用 JSON.stringify 做签名。加上注释解释每一步。"

不需要集成到任何框架里，只要能跑起来、能理解逻辑就行。

**任务 3：统计你自己的"重复行为"。**

回想你最近一次用 AI 工具（ChatGPT、Cursor 等）的经历。你有没有重复问过同一个问题？有没有在 AI 回答不满意时反复用类似的方式追问？

记录下来：你重复了几次？每次的措辞有变化吗？如果 AI 在第一次就明确告诉你"我已经回答过了，你需要换个角度"，你还会继续重复吗？

这个练习帮你理解：**Agent 的重复行为不是因为它"笨"，而是因为它缺少"知道自己已经试过了"的机制。**
