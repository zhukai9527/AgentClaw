# 第 6 章 上下文窗口：Agent 的工作记忆

---

## 聊到第 12 轮，它忘了自己在干什么

那是一个深夜，我在让 AgentClaw 帮我重构一个中等规模的项目。

前三轮很顺利。Agent 读了项目结构，看了配置文件，列出了需要改动的文件清单。第四轮开始改第一个文件。第五轮改第二个。第六轮改第三个。

第七轮，它回去改第一个文件。但改法跟第四轮完全不一样——它好像不记得自己已经改过了。

第九轮，它开始读一个已经读过的配置文件，花了 4000 token 的工具调用，得到了跟第四轮一模一样的结果。

第十二轮，我问它"进度怎么样了"，它回答："我没有看到之前关于重构的讨论。你想做什么重构？"

它彻底忘了。

不是模型不够聪明。不是 prompt 写得不好。是上下文窗口被撑爆了。

我打开 trace 日志，看到了真相：前三轮的文件列表输出了 18,000 token。第四到六轮读了三个源文件，每个文件几千行，又是 22,000 token。第七轮开始，上下文已经接近 80K——触发了压缩。系统把最早的对话（包括"帮我重构项目"的目标和最初的文件清单）压缩成了一个 200 token 的摘要。

摘要写的是："用户要求重构项目，已读取项目结构和三个源文件。"

这个摘要对不对？对。有没有用？有用——但不够。它没有告诉 Agent：哪些文件需要改、每个文件改什么、优先级是什么。这些细节在原始对话里都有，但压缩后只剩下了宏观脉络。

Agent 看到这个摘要，就像一个项目经理看了一眼三周前的会议纪要——知道"有个项目在做"，但不记得具体做到哪了。于是它开始"重新调查"：重新读文件、重新列清单、重新制定计划。用户看到的就是"它忘了"。

这个经历揭示了一个重要事实：**上下文窗口不只是"装东西的容器"，它是 Agent 的工作记忆。** 就像人类的工作记忆一样——你能同时处理 7±2 个信息块。如果有人往你脑子里塞了太多旧信息，新信息就进不来了。你不是变笨了，是工作记忆被占满了。

Agent 也一样。当上下文窗口被工具输出塞满时，LLM 不是变笨了——它是看不到最新、最重要的信息了。它还在"思考"，但它思考的素材被噪音淹没了。

前三轮读了项目结构，输出了大量文件列表。第四到六轮读了三个文件，每个文件几千行。这些工具输出全部堆在对话历史里。到第七轮的时候，上下文已经接近极限。系统不得不丢掉最早的消息来腾出空间——而那些被丢掉的，恰好是 Agent 最初制定的计划。

**Agent 的"工作记忆"就那么大，堆满了旧文件就没地方放新东西。**

这不是 AgentClaw 的个案。任何用 LLM 做 Agent 的人都会撞上这堵墙。128K 的上下文窗口听起来很大——128,000 个 token，差不多能装下一整本小说——但当你的 Agent 开始调用工具、读文件、执行命令，那些工具输出会像洪水一样涌进来，五轮之内就能把窗口填满。

还有一次更诡异的经历。我让 Agent 帮我调试一个 bug。它读了日志文件（2,000 行），执行了几个 shell 命令来排查，然后开始改代码。改到一半，它突然问我："你之前说的 bug 是什么症状来着？"

它不是不记得 bug——它是不记得日志文件里的关键报错行了。那 2,000 行日志已经被压缩了。压缩摘要写的是"执行了日志分析，发现了若干错误"，但具体是什么错误、哪一行报的、错误堆栈是什么——这些细节在压缩中丢失了。

Agent 只好重新读一遍日志文件。又是 8,000 个 token。

这次经历让我意识到：**压缩不只是"变短"的问题，是"保留什么"的问题。** 摘要如果丢了关键细节，Agent 会回头去补——这比不压缩还费钱。

本章要解决的问题：**怎么在有限的上下文空间里，塞进最多有用的信息？**

答案不是"怎么塞更多"，而是"怎么扔掉该扔的"。

这个认知转变很重要。很多人的第一反应是"怎么优化 prompt 让它更短"或者"怎么找到更大的上下文窗口"。但这些都不是根本解法。prompt 可以优化，但系统提示词和工具定义是有最低限度的——你不能把"你是一个有帮助的助手"压缩成"帮"。更大的上下文窗口可以等，但 256K、1M 的窗口意味着更高的输入成本，而且工具输出的增长速度永远比窗口增长速度快。

真正的解法是：**把上下文当成预算来管理，而不是当成容器来填充。**

这一章会带你走一遍上下文管理的完整工程：先搞清楚 128K 窗口的真实开销，然后看怎么触发压缩、怎么压缩、怎么保护最新消息不被误伤、怎么处理工具调用的配对问题、怎么用缓存降低重复输入成本。最后你会看到 AgentClaw 的九步管线——每一步都对应一个真实的生产事故。

读完这一章，你会理解一件事：**上下文窗口不是技术问题，是信息经济学问题。**

## 128K 真实开销：先扣掉固定预算

在谈怎么管理上下文之前，先搞清楚一个基本事实：你拿到的 128K 上下文，真正能用来装对话历史的空间，远没有 128K。

想象你租了一个 128 平米的办公室。听起来很宽敞。但你一进门就发现：会议室占了 4 平米，文件柜占了 8 平米，工位占了 4 平米，你还要留 4 平米过道。真正能用来摊开图纸干活的，只有 108 平米——而且随着一天天过去，桌上会越堆越满。

LLM 的上下文窗口也是这样。每次调用 API，你必须把以下内容一起发过去：

| 常驻开销 | 大约占用 | 说明 |
|---|---|---|
| 系统提示词（指令、人设、规则） | 1,500–4,000 token | 每次调用都要发 |
| 工具定义（10–20 个工具的 JSON 描述） | 3,000–8,000 token | 静态内容，每次调用都要发 |
| 动态上下文（记忆、技能目录） | 500–2,000 token | 每个会话冻结一次 |
| 模型回复的安全余量 | 4,096 token | 不可谈判 |
| **固定开销合计** | ** 10,000–18,000 token** | ** 还没开始对话就没了** |

算一下：128K 的上下文窗口，减去 18K 固定开销，再减去安全余量，剩下的大约 106K 才是真正能用来装对话历史和工具结果的空间。

106K 听起来还是很多？来看一个真实的场景。

Agent 调用了 `file_read` 读一个 500 行的源文件——大约 8,000 token。执行了三次 shell 命令，每次输出几百行——又是 10,000 token。读了一个 JSON 配置文件——3,000 token。

五个工具调用，21,000 token 就没了。

而这只是第五轮。如果你的 Agent 做的是一个复杂任务（比如重构代码、调查一个行业、整理一堆文档），十五轮下来，工具输出轻松超过 60K token。再加上对话历史本身，上下文窗口在第十轮左右就会开始报警。

让我们拆解一次真实的 LLM 调用，看看 token 到底花在哪：

| 内容 | 典型规模 | 说明 |
|------|----------|------|
| 系统提示词 | 约 350 token | 每次调用都要带上 |
| 动态上下文（记忆） | 约 200-800 token | 随用户和会话变化 |
| 工具定义（20 个工具） | 约 1,500-2,000 token | 每次调用都重复发送 |
| 对话历史 | 可变 | 每轮都在增长 |
| 当前用户输入 | 约 50-200 token | 通常不是主要成本 |
| 第一次调用合计 | 约 2,100-3,400 token | 看起来很合理 |
| 第五次调用合计 | 约 40,000-68,000 token | 工具输出开始成为核心问题 |

第五次调用比第一次多了 10-20 倍。不是因为用户说了更多话，而是因为工具输出在累积。每一次工具调用的结果都会留在对话历史里，下一轮被完整地重新发送给 LLM。

一个 `file_read` 调用 = 约 8,000 token。三次文件读取 = 24,000 token 的历史输出，模型已经处理过了，但每轮还在重复付费。这是 Agent 框架里最大的 token 浪费源。

**AgentClaw 的默认策略是：上下文预算设为 80,000 token，大约是 128K 窗口的 60%。**

为什么不设 90%？核心原因有两个。

第一，token 估算不精确。中文混合英文的内容，大约每 3 个字符算 1 个 token，但这是粗估。如果按 90% 的容量运行，估算偏差 20% 就会导致溢出——API 直接报错，Agent 崩溃。

第二，Anthropic 提供了提示词缓存（Prompt Caching）机制：如果你的系统提示词在多次调用之间保持不变，这些 token 的成本可以享受大幅折扣。但这个折扣的前提是"前缀不变"。如果你压到 90% 才开始压缩，系统提示词可能因为内存更新而频繁变化，缓存命中率暴跌，前面省下来的 token 又被浪费回去了。

这里有一个容易被忽视的连锁反应：压缩太晚 → 系统提示词频繁变化 → 缓存失效 → 每轮系统提示词都按完整输入重新计算 → 前面省下来的 token 又被浪费回去。所以 60% 的预算阈值不是保守，是留出估算误差、回复空间和缓存稳定性的工程余量。

> **核心判断：上下文窗口不是容器，是预算。你要决定的不是"还能塞多少"，而是"什么值得留下"。**

## 为什么上下文管理是最高杠杆的工程问题

在 Agent 框架的所有工程问题中——循环控制、工具设计、错误处理、安全防护——上下文管理是投入产出比最高的一个。

原因很简单：**上下文管理影响每一轮 LLM 调用，而 LLM 调用是 Agent 最大的成本中心。**

一次 15 轮的 Agent 对话，LLM 调用通常是最大的成本中心。工具执行（shell 命令、文件读写）本身很便宜，真正的消耗来自每一轮都要发送的大量输入 token。

如果你能把每轮的输入 token 砍掉一半以上（AgentClaw 做到了），累积效果是非常可观的。一个对话省一点，乘以每天的对话量，再乘以持续运行的时间——总体用量曲线上的差距是显而易见的。

更重要的是，上下文管理直接影响 Agent 的**任务完成率**。上下文被撑满 → Agent 失忆 → 重新执行操作 → 浪费更多 token → 用户体验差 → 用户放弃。这是一个恶性循环。反过来，良好的上下文管理 → Agent 记得住 → 任务完成率高 → 用户满意 → 更多使用 → 更多数据 → 更好的 Agent。这是一个良性循环。

**上下文管理不是"锦上添花"的优化，是决定 Agent 能不能用的基础工程。**

## Token 估算：一门不精确的科学

在讨论压缩之前，还有一个基础问题需要解决：**你怎么知道当前的对话用了多少 token？**

答案是：你不能精确知道。

不同的 LLM 使用不同的 tokenizer（分词器）。GPT-4 用的 cl100k_base 和 Claude 用的 tokenizer 对同一段文字的切分方式不同。中文尤其麻烦——一个汉字在某些 tokenizer 里算 1 个 token，在另一些里算 2-3 个。混合中英文的内容更复杂。

AgentClaw 使用一个简单的启发式规则：**大约每 3 个字符算 1 个 token。** 这对中英混合内容来说是一个合理的平均值——中文大约 1-2 个字符/token，英文大约 4 个字符/token。

但"合理"不等于"精确"。在实际测试中，我们的估算偏差在 ±15% 之间。这意味着如果估算值是 50,000 token，实际值可能在 42,500 到 57,500 之间。

这就是为什么预算阈值设在 70% 而不是 90%——你需要给估算误差留空间。在 90% 的容量下，15% 的估算偏差意味着你可能已经溢出了还不知道。

有些框架选择在每次调用后用 API 返回的 `usage` 字段来获取精确的 token 数。这确实更准确，但有一个问题：你只有在调用之后才知道用了多少 token——来不及在调用之前做压缩决策。所以估算虽然不精确，但它是唯一能在"调用之前"做出判断的方式。

**不精确的估算 + 足够的安全余量 > 精确的事后统计。** 你需要的是预警机制，不是事后报告。

## 压缩的触发：什么时候该开始清理

"对话变长了就压缩"——听起来简单，但"变长"有两个维度。

一个对话可能有 25 条短消息，总共才 5,000 token。另一个对话可能只有 8 轮，但 Agent 读了三个大文件，已经用了 60,000 token。你需要两种触发条件：

```
压缩触发条件（任一满足即启动）：

  轮数：  对话轮数 > 20         ← 消息多但单条短
  预算：  估算 token > 56,000    ← 轮数少但工具输出大
```

56,000 是 80,000 预算的 70%。为什么是 70% 而不是 100%？这是你车里的油灯——你想在油箱见底之前收到警告，而不是在高速公路上熄火之后。70% 的阈值留了 30% 的缓冲空间，足够在下一次 LLM 调用之前完成压缩。

两种触发条件解决的是两种完全不同的场景：

**场景一：话多但单条短。** 用户在跟 Agent 聊天，每条消息几十个字，但聊了很多轮。这种对话的 token 增长是线性的，缓慢但持续。轮数触发条件（>20 轮）负责处理这种情况。

**场景二：轮数少但工具输出大。** Agent 在做编程任务，只跑了 8 轮，但每轮都读了大文件或执行了输出很多的命令。这种对话的 token 增长是阶梯式的——某一轮突然暴涨。token 预算触发条件（>56K）负责处理这种情况。

在生产环境中，第二种场景更危险。因为它来得突然——可能上一轮还是 30K，这一轮一个 `file_read` 就跳到了 60K。如果没有预算触发条件，等到轮数触发时，上下文可能已经溢出了。

## 三层压缩瀑布：失败时仍有结果

压缩不是一步到位的操作。它是一个三层升级机制，设计目标是：**即使最智能的那一层失败了，系统仍然能产出结果。**

想象一个快递出了问题。第一选择是联系发件人重新发货（最精准，但需要发件人配合）。发件人联系不上，第二选择是从最近的仓库调货（快，但可能不是原装的）。仓库也没库存，第三选择是给客户退款（不完美，但至少不会让客户一直等）。

Agent 的压缩也是这个逻辑：

**第一层：LLM 摘要（正常模式）。**

把旧对话发给 LLM，让它生成 3-5 个要点的摘要，不超过 500 个字符。30 轮对话、几万 token 的历史，压缩成几百 token 的摘要。大多数时候能成功。

这是最理想的路径。摘要保留了语义——用户想要什么、做了什么决定、当前状态是什么——但丢弃了原始输出。就像你开完会写的会议纪要：记录了结论和待办，但不会记录每个人说的每一句话。

**第二层：LLM 摘要（激进模式）。**

如果第一层失败了（LLM 超时、API 报错、输出格式不对），降级到更简单的指令："2-3 个关键事实，最多 200 字。"温度参数调到 0.05（接近零），因为这时候你更需要准确性，不需要创意。

为什么低温度重要？因为用于上下文管理的摘要里如果出现一个幻觉细节——比如把"用户要重构登录模块"幻觉成"用户要重构支付模块"——Agent 会沿着错误的方向跑完整个会话。越压缩，越不能出错。

**第三层：确定性截断（兜底）。**

如果 LLM 两层都失败了，最后一道防线是纯字符串操作：取对话历史的前 2,048 个字符，加上一个说明头。没有 LLM 参与，没有网络调用，没有概率性输出——直接切割。

这一层永远不会失败。它不完美——你可能会丢掉一些重要信息——但一个能在 99% 的情况下给出完美摘要、在 1% 的情况下硬崩溃的系统，不如一个在 95% 的情况下给出好摘要、在 5% 的情况下给出粗糙但可用结果的系统。

**生产系统的底线是：永远有结果。优雅降级好过完美但脆弱。**

这个设计思路来自一个更广泛的工程原则：**任何依赖外部服务的组件，都必须有不依赖外部服务的降级方案。** LLM 是一个外部服务——它可能超时、可能限速、可能返回格式错误的输出。如果你的上下文管理完全依赖 LLM 来做摘要，那 LLM 一挂，整个 Agent 就瘫痪了。

三层瀑布的本质是：用 LLM 的智能换压缩质量，同时保留一个不依赖 LLM 的确定性退路。这样即使摘要调用失败，Agent 也不会因为"压缩失败"直接中断——它会损失一些历史细节，但仍能继续处理当前任务。

三层瀑布的具体策略是：

1. **第一层：LLM 摘要（正常）**
   - 摘要要求：3-5 个要点，不超过 500 字。
   - `maxTokens`: 300。
   - 成功率约 95%。
   - 失败后进入第二层。
2. **第二层：LLM 摘要（激进）**
   - 摘要要求：2-3 个关键事实，最多 200 字。
   - `maxTokens`: 150，`temperature`: 0.05。
   - 低温度代表准确优先于创意。
   - 失败后进入第三层。
3. **第三层：确定性截断**
   - 取前 2,048 个字符，加一段说明头。
   - 不调用 LLM，只做字符串切割。
   - 永远成功，是最后一道防线。

三层之间还有一个缓存机制：用 `conversationId + 轮数` 作为缓存键。同一个压缩边界，不会重复调用 LLM。该压缩的段落已经压缩过了，直接用缓存结果，零开销。

来看一个真实的第一层摘要长什么样。一段 30 轮的编程对话，压缩后变成了：

> 用户要求将 Express 项目的端口从 3000 改为 8080，并添加 CORS 支持。已修改 config.json（端口）和 server.ts（CORS 中间件）。package.json 中新增了 cors 依赖。所有修改已完成，未运行测试。用户最后提到还需要添加 rate limiting。

200 个 token。它保留了什么？**目标**（改端口、加 CORS）、** 决策**（改了哪些文件、怎么改的）、** 当前状态**（改完了但没测试）、** 待办**（rate limiting）。它丢弃了什么？每个文件的完整内容、每条 shell 命令的输出、每个中间步骤的细节。

这个摘要够用吗？对大多数情况来说够了。Agent 看到这个摘要，知道项目做了什么改动，可以继续做 rate limiting 而不需要重新读那些文件。

但如果你需要回溯"config.json 改了哪些具体字段"呢？这就是磁盘备份的用武之地——调一次 `file_read`，完整记录就在那里。

## 新鲜尾部保护：最近的消息优先保留

这是一个会浪费你好几天时间的错误：你压缩了对话历史，Agent 立刻忘了它**正在做什么**。

摘要记录了宏观脉络——"用户要重构登录模块"——但没有记录 Agent 刚才打开的那个文件、上一条命令的具体报错、或者用户三分钟前刚补充的那句话。

Agent 会怎么做？它会重新读一遍那个文件。重新跑一遍那个命令。重新问一遍那个问题。

你花在压缩上省下来的 token，全被重新执行的操作吃回去了。而且更糟——Agent 的行为会变得莫名其妙，因为它失去的是**工作记忆**：它正在处理的具体上下文。

解决方案是**新鲜尾部保护（Fresh Tail Protection）**：最近 N 条消息永远不被压缩。在 AgentClaw 里，N 是 32。

32 条消息看起来很多？故意的。尾部数量（32）比压缩触发阈值（20）还大。这意味着：

- 按轮数触发的压缩（超过 20 轮），主要针对话多但内容短的对话。压缩掉的是更早的闲聊或重复操作，最近 32 条完整保留。
- 按 token 预算触发的压缩（工具输出撑爆了窗口），才是处理"少数几轮包含大量内容"的场景。

还有一个关键细节：如果压缩的切割点刚好落在一条工具结果消息上，切割点会自动往前推，直到落在一条用户消息或助手消息上。

为什么要这样做？因为 LLM API 有一个铁律：**一条包含工具调用的助手消息，后面必须跟着对应的工具结果。** 如果压缩把工具结果丢掉了，但工具调用还在，API 会直接返回 400 错误——不是降级输出，是彻底拒绝。

## 工具对安全：最容易忽视的炸弹

这是上下文管理中最容易被忽视、也最容易炸的问题。

LLM API 对消息序列有一个严格的要求：助手发出的每个 `tool_use`（工具调用）块，后面必须有对应的 `tool_result`（工具结果）块。这是 API 层面的硬性校验，不是建议。

压缩操作会删掉旧消息。但删消息可能造成两种"孤儿"：

**孤儿工具结果**：工具结果还在，但它引用的那个工具调用被压缩掉了。结果无家可归。

**孤儿工具调用**：助手的工具调用还在，但对应的工具结果被压缩掉了。调用没有回应。

任何一种都会让 API 返回 400 错误。你的 Agent 直接崩溃，用户看到的是"请求失败"。

AgentClaw 的解决方案叫 `sanitizeToolPairs`（清理工具对）。它在所有消息修改操作完成之后运行，扫描整条消息链，找到所有不匹配的工具调用和结果：

- 孤儿工具结果 → 移除这个结果块（注意是块级别，不是消息级别。一条消息可能包含多个工具结果，有些匹配、有些不匹配）
- 孤儿工具调用 → 插入一个占位结果："*来自早期对话的结果——见上方摘要*"

这里有一个实现细节很重要：操作是在**内容块（block）级别** 进行的，不是消息级别。一条工具消息可能包含多个工具调用的结果——有些结果的调用还在，有些已经不在了。你不能整条删掉，只能精确地移除不匹配的块。

**如果没有 `sanitizeToolPairs`，任何涉及历史修改的上下文管理策略都会在某个时刻产生无效消息序列。**"某个时刻"在生产环境里的意思是：第一次有重度用户跑了 30 轮编程对话的时候。

这个 bug 在 AgentClaw 的早期版本里存在了两周。它不是每次都出现——只有当对话足够长、触发了压缩、而且切割点恰好落在工具调用附近时才会触发。前 20 次测试都没问题，第 21 次突然报 400 错误。排查了整整一天才定位到根因：压缩制造了孤儿工具调用。

教训是：**上下文管理的每一步操作都可能破坏消息序列的完整性。** 你不能假设"压缩只是删消息"——删消息会改变消息之间的引用关系。工具对清理不是可选的附加功能，是必须存在的安全保障。

## 冻结快照：让系统提示词命中缓存

Anthropic 的 API 有一个叫"提示词缓存"的机制。如果你发给 API 的系统提示词在多次调用之间保持字节级一致，那些 token 的成本可以享受大幅折扣。

这意味着：系统提示词越稳定，重复输入成本越低。

但问题是，系统提示词里经常需要注入动态内容——用户记忆、技能目录、平台提示。如果每次调用都重新构建这些动态部分，系统提示词每轮都在变，缓存永远命中不了。

AgentClaw 的解决方案叫**冻结快照（Frozen Snapshot）**。

逻辑很简单：每个对话（conversationId）第一次调用 `buildContext` 时，构建动态上下文（记忆 + 技能目录），缓存起来。这个对话后续的所有轮次，直接复用缓存版本。

```
第一轮：构建动态上下文 → 缓存
第二轮：读缓存 → 直接用
第三轮：读缓存 → 直接用
...（直到会话结束）
```

代价是什么？如果用户在会话中途存储了一条新记忆，这条记忆会写入数据库，但不会出现在当前会话的系统提示词里。它要到下一次新会话才会被加载。

这是一个刻意的权衡。Agent 在当前对话中刚刚学到的信息，可以通过对话历史来回顾——它就在最近的消息里。而系统提示词的稳定性带来的缓存收益，远大于会话内记忆更新的价值。

用 token 用量来说明：假设你的系统提示词（含工具定义和动态上下文）有 15,000 token。如果每轮都变化，每次调用都要重新计算这 15,000 token。如果冻结了，从第二轮开始，这些 token 可以命中缓存。一个 15 轮的对话，光系统提示词部分就能避免大量重复输入。

**冻结快照不是优化，是基础设施。**

但这里有一个需要诚实面对的权衡：冻结意味着"不实时"。如果用户在对话中途告诉 Agent "我改主意了，以后叫我老王"，这条记忆会被提取并写入数据库，但当前会话的系统提示词不会更新。Agent 在接下来的对话中可能还是会叫用户原来的名字。

这会让用户困惑："我刚说了改叫老王，你怎么还在叫旧名字？"

AgentClaw 的处理方式是：Agent 可以通过对话历史看到用户刚说的话，所以它在当前轮次的回复中会用新名字——但下一轮的系统提示词里仍然没有更新。如果用户特别在意这一点，可以在新会话开始时确认。

这是一个典型的工程权衡：完美的一致性 vs 可接受的性能。我们选择了性能，因为缓存折扣带来的成本节省是确定的、可衡量的，而"会话内记忆不实时"是一个小概率、低影响的不便。

## 工具定义：你忘了算的税

每次调用 LLM API，你必须把所有已注册工具的完整 JSON Schema 一起发过去。AgentClaw 有 20 多个工具，每个工具有名字、描述、参数定义。我们测量过：**每次调用约 1,500-2,000 token 只用于工具定义。**

在 15 轮的 Agent 循环中，同样的工具定义被发送了 15 次 = 30,000 token，内容完全一样。

你可能会问：这些 token 不是应该被缓存吗？理论上是的。但工具定义在消息序列中的位置决定了它是否能被缓存——它必须在系统提示词之后、对话历史之前。如果你的实现把工具定义放在了消息体里而不是系统提示词前缀中，缓存就无法覆盖到它们。

更隐蔽的问题是**多余的字段**。我们在审计工具定义时发现，有一个 `_intent: string` 字段被注入到了每个工具的 Schema 里，用于追踪目的。这一个字段大约 20 个 token × 20 个工具 = 每次调用多花 400 个 token。一个 15 轮对话就是 6,000 个 token——只是为了在每次调用时问模型"你为什么调用这个工具？"

把这个字段改成可选之后，立刻省下了这些开销。

**审计你的工具定义。删掉每个多余的字段。每一个 token 都在计费。**

## 观察压缩：80% 的工具输出可以扔掉

前面讲的是对话级别的压缩——把旧对话变成摘要。但还有另一层压缩，针对的是单条工具结果。

大多数工具输出，从 Agent 的角度来看都是噪音。shell 命令的详细日志、API 返回的完整 JSON、文件列表的几百行输出——Agent 真正需要的只是其中一小部分：有没有报错？成功了没有？关键字段是什么？

观察压缩（Observation Compression）对最近 2 条工具结果不做处理（它们是"热"的，Agent 还在用），对更早的工具结果做四步提取：

**第一步：提取错误行（最高优先级）。** 扫描关键词：`error`、`exception`、`fail`、`ENOENT`、`denied`。错误是 Agent 最需要的信息——它决定了下一步是重试、换方法还是放弃。

**第二步：提取状态行。** 扫描关键词：`success`、`created`、`updated`、`total`、`count`。状态行告诉 Agent 操作是否成功、结果的数量级。

**第三步：提取 JSON 关键字段。** 如果输出是 JSON 格式，只保留 `id`、`name`、`status`、`error`、`message`、`path`、`url` 这几个字段。其他的嵌套结构、分页信息、元数据全部丢掉。

**第四步：取首尾。** 如果前三步提取的结果太少（少于 3 条），取前 3 行和最后 2 行作为结构化上下文。

另外还有一个去重机制：如果两条工具结果的前 200 个字符完全相同（内容指纹匹配），第二条压缩成"*重复结果——同前*"

真实的压缩效果：

| 内容类型 | 压缩前 | 压缩后 | 节省 |
|---|---|---|---|
| 带错误的 shell 输出 | 2,100 字符 | 380 字符 | **82%** |
| JSON API 响应 | 4,500 字符 | 290 字符 | **94%** |
| 文件列表（200 个文件） | 8,400 字符 | 350 字符 | **96%** |

96% 的压缩率意味着：原来需要 8,400 个字符才能装下的文件列表，现在只需要 350 个字符——Agent 仍然知道"有 200 个文件"，只是不知道每个文件的详细信息。如果它需要细节，可以再调一次 `file_read`。

来看一个具体的例子。Agent 执行了 `ls -la` 命令，输出了 200 个文件的完整列表，8,400 个字符。观察压缩之后，Agent 看到的是：

```
[Tool: shell] ls -la
Results: (compressed from 8400 chars)
  total 200 files
  First 3 lines:
    drwxr-xr-x  12 user staff  4096 Mar 15 src/
    -rw-r--r--   1 user staff  2048 Mar 14 package.json
    drwxr-xr-x   8 user staff  2048 Mar 13 node_modules/
  Last 2 lines:
    -rw-r--r--   1 user staff   512 Mar 15 tsconfig.json
    drwxr-xr-x   4 user staff  2048 Mar 15 tests/
```

350 个字符。Agent 知道了项目结构的大致样貌——有 `src/`、`node_modules/`、`tests/`、`package.json`、`tsconfig.json`。如果它需要看 `src/` 目录下的具体文件，可以再调一次工具。但大多数时候，这个概览已经足够它做出下一步决策了。

另外还有一个基础压缩层，对所有工具结果无条件运行：合并多余的空行、把 Tab 转成空格、精简格式化的 JSON。这些变换不影响语义，单独就能省下不少 token。

## 写入工具的特殊处理：LLM 写的东西不需要再读一遍

这是一个单行修复，效果出乎意料地好。

当你的 Agent 调用 `file_write` 写了 500 行代码，这 500 行代码会作为工具参数留在对话历史里。下一轮调用 LLM 时，模型会再次看到这 500 行——但它自己写的，它知道里面是什么。

这是纯粹的浪费。模型不需要重新阅读自己刚写的东西。

AgentClaw 的处理方式很直接：对于 `file_write`、`file_edit`、`execute_code` 这类写入工具，执行完成后，参数中的长内容（比如 `content`、`code` 字段）被截断到前 50 个字符，加上 `"... [truncated]"`。

模型看到的是 `content: "import { useState } from 'react';\n\nex... [truncated]"`，而不是完整的 500 行。它知道自己写了什么，不需要重读。

这一个改动，节省了大量 token——写入操作在 Agent 对话中很常见，每次写入都省一笔，累积效果可观。

为什么是 50 个字符而不是 0？因为模型需要一个"锚点"来回忆自己写了什么。50 个字符足够让它想起"我写了一个 React 组件的 import 语句"，但不会浪费太多空间。如果截断到 0（完全删除），模型可能不记得自己写过什么文件、用了什么框架——它需要一个最小的上下文片段来触发回忆。

这个优化也有一个前提：**你信任 LLM 的输出。** 如果你不确定模型写的代码是否正确（比如你在做代码审查），那截断参数就太早了——你需要保留完整内容来验证。AgentClaw 只在"模型生成 → 工具执行 → 结果反馈"这个流程完成后才截断，也就是说，工具已经执行了，代码已经写入了文件，模型的输出已经产生了实际效果。这时候保留完整参数就是纯浪费。

---

## 大内容卸载：写到磁盘，只给预览

有些工具输出天然就很大。读一个 2000 行的文件——约 16,000 token。执行一个返回完整 JSON 的 API——可能 10,000 token。把这些全部塞进上下文，三四个这样的调用就能把窗口撑满。

AgentClaw 的策略是：超过 12,000 字符的工具结果，完整内容写入磁盘文件，上下文里只保留一个 1,500 字符的预览和文件路径引用。

```
完整内容 → data/tmp/{conversationId}/full_result.txt
上下文里 → "文件已保存到 data/tmp/xxx/full_result.txt（前 1,500 字符预览：...）"
```

Agent 如果需要看完整内容，可以调用 `file_read` 工具去读。这需要一次额外的工具调用，但比把 16,000 token 塞进每一轮调用要便宜得多。

这个模式的灵感来自人类的工作方式：你开会时的笔记是摘要，完整录音是备份。99% 的时候你看笔记就够了，只有需要回溯细节时才去听录音。

**有损压缩 + 无损备份——在上下文管理和信息完整性之间找到平衡。**

这个设计还有另一个好处：它解决了"摘要太简略"的问题。如果 Agent 觉得摘要不够详细，它可以主动调用 `file_read` 去看完整记录。这让压缩变成了一种"按需加载"——大多数时候用摘要就够了，偶尔需要细节时可以追溯。

在实际使用中，Agent 调用 `file_read` 去回溯完整记录的概率大约是 5%。也就是说，95% 的情况下，200 token 的摘要就已经足够 Agent 做出正确决策了。只有当任务特别复杂、需要回溯很早的细节时，才会用到完整记录。

## 过时输出替换：三轮前的工具结果不值得保留

除了观察压缩，还有一种更简单的优化：**把过时的工具结果替换成占位符。**

什么意思？Agent 在第一轮读了 `config.json`，在第五轮又读了一遍（可能是因为忘了）。第一轮的工具结果——完整的 JSON 内容——还在对话历史里，但对 Agent 来说已经没有价值了，因为第五轮有了更新的版本。

AgentClaw 的做法是：超过 3 轮的工具结果，自动替换成 `[previous tool_name result]`。不需要 LLM 参与，纯字符串替换，时间复杂度 O(n)。

这个策略的前提是：**工具结果的价值随时间递减。** 三轮前的 shell 输出、五轮前的 API 响应——Agent 已经处理过了，留着只是占空间。把它们替换成占位符，Agent 知道"这个工具之前调用过"，但不需要重新看完整输出。

这个修复节省了大量 token，而且实现极其简单——一个 `if` 判断加一个字符串替换。

## 看不见的开销：后台 LLM 调用

你的 trace 看板显示的是主循环的 token 消耗。但有些 LLM 调用发生在后台，不出现在任何看板上。

**记忆提取**：AgentClaw 有一个后台流程，定期调用 LLM 来提取长期记忆。每次调用都要把最近的对话和已有记忆发给 LLM。一个长对话可能触发好几次提取——每次都是你看不见的 token 消耗。

早期版本提取频率太高（每 3 轮一次），后台调用次数太多。调低频率后，后台调用大幅减少，记忆质量没有可感知的下降。

**摘要缓存失效**：当上下文太长时，系统会调用 LLM 来摘要旧对话。早期版本的缓存键是 `conversationId + 最新轮次ID`。每来一条新消息，缓存就失效，重新摘要——即使被摘要的旧对话根本没有变化。

修复方案：用 `轮次数` 作为缓存键。同样的旧轮次 = 同样的摘要 = 缓存命中 = 零 LLM 调用。

这些看不见的开销加起来，可能占到总 token 消耗的 15-20%。如果你只看主循环的 trace，会觉得"我优化得很好啊，为什么总用量还是这么高？"——因为后台调用也在持续消耗 token。

## 完整管线：九个步骤，每一步都有原因

把所有压缩策略组装在一起，AgentClaw 的上下文管理是一个九步管线。每一步的顺序很重要——乱了就会出问题。

这九个步骤按顺序执行：

1. 从 SQLite 加载对话历史，最多加载 `maxHistoryTurns` 轮。
2. 检查触发条件：轮数是否超过 20，或者估算 token 是否超过 56K；如果都没有触发，直接使用原始历史。
3. 分割历史：旧轮次进入压缩区，最近 32 条消息作为新鲜尾部受到保护。
4. 用三层瀑布压缩旧轮次。
5. 把完整记录卸载到磁盘。
6. 重新组装消息：摘要 + 确认信息 + 新鲜尾部。
7. 构建或复用冻结快照：动态上下文包括记忆和技能。
8. 处理大内容：工具结果超过 12K 字符时写入磁盘，只保留预览。
9. 做最后清理：观察压缩、tool pair 修复、空白规范化、JSON 精简，然后组装系统提示词、动态后缀和处理后的消息发给 LLM。

为什么顺序重要？

大内容提取必须在观察压缩之前——否则观察压缩器会试图处理一个 100K 字符的巨型输出，可能卡住或遗漏。

工具对清理必须在所有消息修改操作之后——这样才能捕捉到所有被前面步骤制造出来的孤儿。

基础压缩放在最后——它做的是无损的格式优化（去多余空行、JSON 精简），不应该干扰前面的结构化提取。

九个步骤，每一个都对应一个真实的生产事故。没有一个是"以防万一"加上去的。

## 我们删掉的方案：过度设计的教训

在走到现在这套方案之前，我们试过几种看起来聪明、实际行不通的做法。

**激进历史截断：只保留最近 N 条消息。**

最简单的方案：对话太长？只保留最后 20 条，前面的全删。

问题是：多步任务会断裂。Agent 五步前读了一个文件、做了决定，截断后完全不记得了。它会重新读一遍那个文件，重新做一遍那个决定，甚至可能做出相反的决定。

最终方案保留了"发生了什么"的语义摘要，同时丢弃了原始输出。这是一个关键区别：知道"我读了 config.json 并把端口从 3000 改成了 8080"和知道"config.json 的完整内容是..."是两回事。前者 20 个 token，后者 2,000 个 token。

**会话内记忆刷新：每轮更新系统提示词中的记忆。**

用户在对话中存了一条新记忆，立刻注入系统提示词，让 Agent 在下一轮就能"知道"。

问题是：这会杀死提示词缓存命中率。系统提示词每轮都在变，Anthropic 的缓存折扣从第二轮开始就享受不到了。10% 的记忆新鲜度提升，换不回 90% 的缓存折扣损失。

最终方案是冻结快照：会话内的新记忆写入数据库但不更新系统提示词。Agent 可以通过对话历史回顾刚学到的东西。

**字典压缩：用短编码替换常见字符串。**

比如把 `"The file was not found"` 替换成 `$F01`，把 `"Permission denied"` 替换成 `$F02`。理论上能节省 4-8% 的 token。

问题是：LLM 需要"理解"这套编码方案。你在系统提示词里加了一张解码表，但模型不一定每次都正确解码。更糟的是，如果解码错了，Agent 会基于错误的理解做出决策。4-8% 的节省，换来的是不可预测的行为退化——不值得。

**按工具类型定制压缩策略：不同工具用不同的压缩逻辑。**

`file_read` 的输出用一种压缩方式，`shell` 的输出用另一种，`web_search` 的结果用第三种。

问题是：复杂度爆炸。每新增一种工具就要写一套压缩逻辑，测试矩阵指数增长。实际收益微乎其微——四步提取（错误→状态→JSON→首尾）对所有工具类型都有效，不需要定制。

**完全丢弃旧工具结果：只保留对话文本，工具结果全部删掉。**

最激进的方案：工具结果用完就没用了，直接删掉，只保留用户和 Agent 的对话。

问题是：Agent 经常需要引用之前的工具结果来做决策。"我之前读的那个文件里有个配置项叫什么来着？"——如果工具结果全删了，Agent 只能重新读一遍。而且有些工具结果本身就是重要信息（比如搜索结果、API 返回的数据），删掉就丢了。

最终方案是分层处理：最近 2 条完整保留，更早的做观察压缩（提取关键信息），太大的卸载到磁盘。不是"全删"也不是"全留"，而是"按价值分级处理"。

**这些被删掉的方案有一个共同点：它们都是"非黑即白"的策略。** 要么全保留，要么全删。要么每次都更新，要么永远不更新。现实中的上下文管理需要的是灰度——按时间衰减、按重要性分级、按大小分层。简单策略在测试中看起来能用，但在真实用户的长对话中会暴露出各种边界问题。

## 优化前后的对比

这些优化做完之后，效果主要体现在三件事上：每轮输入 token 明显下降，后台 LLM 调用减少，长对话里的上下文溢出和"失忆"明显变少。对用户体验来说，后两点比降低 token 更重要：用量决定系统能不能长期跑下去，不失忆决定用户愿不愿意继续用。

下面是一次典型 3 轮编程对话的对比：

| 轮次 | 优化前输入 token | 优化后输入 token | 发生了什么 |
|---|---|---|---|
| 第 1 轮 | 42,350 | 42,350 | 列出文件、读取配置（4 次工具调用） |
| 第 2 轮 | 58,200 | 25,975 | 读了两个源文件，共 1,800 行 |
| 第 3 轮 | 61,400 | 17,454 | 问关于之前结果的问题 |

第 3 轮用了比第 1 轮**少 59%** 的 token——尽管对话历史严格来说更长了。这就是压缩管线在起作用。优化前，第 3 轮还在发送前两轮的完整工具输出。优化后，旧的工具结果被压缩了，只保留了关键信息。

## 上下文管理的本质：信息优先级

回顾这一章，所有的压缩策略、保护机制、缓存技巧，本质上都在做同一件事：**决定什么信息在什么时候值得出现在 LLM 面前。**

这不是一个技术问题。这是一个判断问题。

系统提示词——永远值得，但必须稳定。它是 Agent 的"人格"和"规则"，每一轮都需要。但如果你每轮都重建它，缓存就废了。

工具定义——永远需要，但可以缓存。它们是 Agent 的"能力清单"，告诉模型"你能做什么"。但 20 个工具的完整定义每轮都发，是一个不小的开销。审计每个字段，删掉多余的。

新鲜对话——永远保留，因为 Agent 正在用。这是它的工作记忆，压缩掉它就是在制造混乱。

旧对话——压缩成语义摘要，丢弃原始输出。知道"做了什么决定"就够了，不需要知道"每个细节"。

旧工具结果——提取关键信息，丢弃噪音。错误和状态最重要，完整输出不重要。

大内容——写到磁盘，只留预览。99% 的时候预览够用，1% 的时候再去读完整文件。

写入操作——截断参数，模型知道自己写了什么。它不需要重读自己刚写的东西。

过时输出——替换成占位符。三轮前的结果已经过时了，留着只是浪费。

这些策略加在一起，形成了一个完整的信息优先级体系。每一层都在回答同一个问题：**这条信息值不值得占用宝贵的上下文空间？**

这不是"怎么把更多东西塞进有限空间"的问题。这是一个信息经济学问题：每一轮 LLM 调用，你有一个固定的 token 预算，你需要把预算花在最高价值的信息上。

这里有一个反直觉的事实：**压缩越激进，不一定越省 token。** 如果你把旧对话压缩得太狠，Agent 丢失了关键上下文，它会重新读文件、重新执行命令、重新问用户问题。每一次"重新"都是一次额外的工具调用 + 一次额外的 LLM 调用。你在压缩上省下来的 token，可能被这些"重新"操作吃掉，甚至消耗更多。

AgentClaw 在早期版本中犯过这个错误：摘要指令写得太简单（"一句话总结"），结果摘要丢失了太多细节，Agent 频繁回去"补课"。后来把摘要指令改成了"3-5 个要点，保留关键决策和状态"，虽然每个摘要多用了 100 个 token，但 Agent 回头补课的概率从 40% 降到了 8%。

**好的压缩不是"扔掉最多的东西"，而是"扔掉对下一步决策最没用的东西"。**

上下文窗口是你最贵的资源——比算力贵，比存储贵，比网络贵。管理它的方式，决定了你的 Agent 是能稳定完成任务，还是聊几轮就失忆。

## 动手试一试

**任务 1：测量你自己的对话 token 消耗。**

用 AI 编程工具（Cursor 或 Claude Code）完成一个小任务，比如"帮我写一个 Python 脚本，从 CSV 文件中统计每个城市的平均气温"。让它跑完之后，问它：

> "回顾一下这次对话，估算每一轮的 token 消耗。列出每轮的输入 token 大约是多少，其中工具输出占了多少。"

你会发现工具输出的占比远超你的预期。

**任务 2：给一段长对话做三种压缩版本。**

找一段你和 ChatGPT 或 Claude 的长对话（10 轮以上），把内容复制出来，然后让 AI 编程工具做三种压缩：

> "把下面这段对话压缩成三个版本：
> 1. 5 句话的摘要（保留所有关键决策）
> 2. 200 字的详细摘要（保留决策和重要细节）
> 3. 50 字的极简摘要（只保留最终结论）
>
> 哪个版本能让你重新接上工作？哪个版本丢失了关键信息？"

你会直观地感受到压缩率和信息损失之间的权衡。

**任务 3（可选）：设计一个"新鲜尾部保护"策略。**

假设你有一个对话，30 轮，你想从第 10 轮开始压缩。但 Agent 在第 25 轮刚读了一个文件、第 27 轮刚执行了一个命令。用纸笔画出：哪些消息应该被压缩成摘要，哪些必须完整保留？如果你在第 10 轮切一刀，第 25 轮和第 27 轮的工具结果会怎样？

**任务 4：审计你常用的 AI 工具的上下文消耗。**

如果你在用 Cursor、Claude Code 或其他 AI 编程工具，注意观察：

> "当你让 AI 读了一个大文件之后，后续的对话是不是变慢了？AI 是不是开始'忘事'了？它有没有重新读过已经读过的文件？"

这些都是上下文被撑满的信号。记录下你在第几轮开始感觉到这些问题，以及你的对话大概有多少轮。这个数据能帮你理解"128K 上下文在实际使用中能撑多久"。

**替代方案**（没有 AI 编程工具时）：拿一段你写过的会议纪要，试试把它压缩成三句话。然后问自己：如果明天你要根据这三句话继续工作，哪些信息是你绝对不能丢的？那就是你的"新鲜尾部"。

## 本章小结

1. **128K 上下文的真实可用空间远没有 128K。** 系统提示词、工具定义、安全余量占了相当一部分固定开销，还没开始对话就少了一截。一个 `file_read` 调用就能吃掉几千 token——几个工具调用就用掉了不少。

2. **上下文是预算，不是容器。** 问题不是"还能塞多少"，而是"什么值得留下"。AgentClaw 的默认预算大约是上下文窗口的 60%，在 70% 时触发压缩。

3. **三层压缩瀑布让失败可控。** LLM 正常摘要 → LLM 激进摘要 → 确定性截断。即使 LLM 两层都失败，系统仍然能产出粗糙但可用的压缩结果。

4. **新鲜尾部保护：最近 32 条消息不压缩。** Agent 的工作记忆不能丢。压缩掉它正在用的信息，它会重新读文件、重新跑命令——比不压缩还费钱。

5. **工具对安全是隐藏炸弹。** 压缩消息可能制造"孤儿"——有调用没结果，或有结果没调用。任何一种都会让 API 直接报错。必须在所有修改之后清理。

6. **冻结快照让系统提示词命中缓存。** 每个会话只构建一次动态上下文，后续轮次直接复用。代价是会话内的新记忆不会立刻出现在系统提示词里，但缓存收益远大于此。这是一个反直觉但算过账的权衡。

7. **大部分工具输出是噪音。** 观察压缩通过四步提取（错误→状态→JSON→首尾）大幅压缩工具结果，Agent 仍然知道发生了什么，只是不知道每个细节。

8. **写入工具的参数可以截断。** 模型自己写的内容不需要再读一遍。一个 `if` 判断，效果立竿见影。

9. **完整记录写到磁盘。** 有损压缩 + 无损备份。摘要在上下文里随时可见，完整记录通过 `file_read` 按需访问。

## 如果你只记住一件事

上下文窗口管理的所有工程，归结为一个判断：**在每一轮 LLM 调用中，什么是当前最有价值的信息？**

系统提示词和工具定义是"基础设施"——每轮都需要，但应该尽量稳定以享受缓存折扣。新鲜对话是"工作记忆"——Agent 正在用，应优先保留。旧对话和旧工具结果是"档案"——有用但不紧急，可以压缩、可以卸载、可以替换成占位符。大内容是"附件"——存到磁盘，按需加载。

如果你的 Agent 经常"聊着聊着就忘了"，问题几乎总是出在上下文管理上，而不是模型能力上。模型足够聪明——它只是看不到该看的信息了。

管理上下文窗口，就是管理 Agent 的注意力。注意力是有限的，把它花在刀刃上。
