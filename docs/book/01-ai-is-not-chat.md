# 第 1 章 AI 不只是聊天的对象

---

## 一个实验

打开 ChatGPT，输入这句话：

> "帮我调查一下 2025 年中国新能源汽车的出口情况，整理成一份报告。"

它会给你一段很不错的回答。可能有数据、有分析、有趋势总结。看起来像一份报告。

但它是"像"一份报告，不是一份报告。

它没有真的去搜索最新数据。它用的是训练时学到的知识，可能已经过时了。它没有打开任何网页，没有下载任何文件，没有对比不同来源的数据。它只是根据你的问题，生成了一段看起来合理的文字。

如果你追问："这个数据是从哪来的？"它可能会编造一个来源。不是因为它想骗你，而是因为它的工作方式就是"生成看起来合理的文字"，不是"查找真实的信息"。

现在换一种方式。你把同样的要求告诉一个 Agent。它会这样做：

1. 搜索"2025 年中国新能源汽车出口数据"
2. 打开搜索结果中的几个网页，提取关键数据
3. 搜索"2024 年中国新能源汽车出口数据"做对比
4. 搜索"2025 年新能源汽车出口政策变化"
5. 把所有信息整理成一份结构化的报告
6. 保存到一个文件里，发给你

这不是假设。这是 AgentClaw 实际能做到的事。

区别在哪里？**ChatGPT 回答你的问题。Agent 把你的事情做完。**

## 问一句答一句 vs 自己把事情做完

这个区别值得展开说，因为它决定了你后面所有设计决策的方向。

**聊天机器人** 的工作模式是：

```
用户问 → AI 答
```

一问一答。AI 的能力边界是"生成文字"。它能写文章、翻译、解释概念、回答问题——但它不能做任何超出文字生成范围的事。它不能打开文件、不能搜索网页、不能执行命令、不能调用 API。它是大脑，但没有手脚。

你问它"今天天气怎么样"，它会告诉你"我无法获取实时天气信息"。不是因为它不知道天气是什么，而是因为它没有连接天气 API 的能力。

你问它"帮我把这个 CSV 文件里的数据做个统计"，它会告诉你"请把文件内容粘贴给我"。不是因为它不会做统计，而是因为它不能自己打开文件。

**Agent** 的工作模式是：

```
用户说目标 → AI 自己规划 → 调用工具执行 → 看到结果 → 调整计划 → 继续执行 → ... → 完成
```

它不是等你问一个问题再答一个问题。它是接到一个目标后，自己决定该做什么、怎么做、做到什么程度算完。

你告诉它"帮我调查某个行业"，它会自己搜索、自己读网页、自己整理信息、自己写报告。中间遇到问题（搜索结果太少）会自己调整（换个关键词再搜）。遇到它处理不了的事（需要付费才能看的数据）会回来问你（"这个数据源需要付费，要继续吗？"）。

打一个比方：

聊天机器人像一个电话客服。你打电话问"我的订单什么时候到？"，客服查一下系统告诉你"三天后到"。一问一答。你问完就挂了。

Agent 像你的私人助理。你说"帮我安排下周的出差"，助理会自己查机票、订酒店、看天气、安排接送、发日程提醒。中间遇到问题（航班取消了）会自己处理（换一班）或者回来问你（"下午的航班取消了，改到晚上行吗？"）。你不用每一步都盯着。

**这个区别不是模型能力的区别，是系统设计的区别。**

GPT-4 和 Claude 足够聪明，能做复杂推理。但如果你只给它们一个聊天界面，它们就只能聊天。你需要给它们一个循环——让它们能"做事"而不是只"说话"——它们才能变成 Agent。

## Agent 的最小定义

说了这么多，Agent 到底是什么？

去掉所有花哨的术语，Agent 的核心就是四个步骤的循环：

```
感知（Perceive）→ 推理（Reason）→ 行动（Act）→ 观察（Observe）
```

然后回到"感知"，再来一轮。

**感知**：Agent 获取信息。可能是用户的指令、一个文件的内容、一次搜索的结果、一个 API 的返回值。感知是 Agent 了解世界的方式。没有感知，Agent 就是在黑暗中猜。

**推理**：Agent 思考下一步该做什么。这是大语言模型（LLM）的活。它根据当前信息决定：是直接回复用户，还是调用某个工具继续做事。推理是 Agent 的大脑——它决定了 Agent 的决策质量。

**行动**：Agent 执行一个具体操作。读一个文件、搜索一个网页、执行一段命令、调用一个 API。行动是 Agent 改变世界的方式。没有行动，Agent 就只能说不能做。

**观察**：Agent 看到行动的结果。成功了还是失败了？返回了什么信息？需要调整计划吗？观察是 Agent 学习的方式——它根据结果调整下一步的计划。

然后带着新的信息，回到"推理"，再来一轮。

用做饭来解释：

你打开冰箱（感知），发现有鸡蛋、西红柿和面条（信息）。你想了想（推理）：可以做西红柿鸡蛋面。你开始切西红柿、打鸡蛋、烧水煮面（行动）。煮完尝了一口（观察）：淡了。加点盐（新的行动）。再尝一口（观察）：刚好。出锅。

如果把做饭的过程写成伪代码：

```
while 没做好:
    看冰箱有什么（感知）
    想好做什么菜（推理）
    动手做（行动）
    尝一口（观察）
    if 做好了:
        出锅
```

把这个循环对应到 Agent 的世界里：

| 做饭 | Agent | 例子 |
|------|-------|------|
| 打开冰箱 | 感知 | 读取用户消息、打开文件、搜索网页 |
| 看到食材 | 信息获取 | 搜索结果、文件内容、API 返回值 |
| 想菜单 | 推理 | LLM 分析当前信息，决定下一步 |
| 切菜炒菜 | 行动 | 调用工具执行具体操作 |
| 尝味道 | 观察 | 检查工具执行结果 |
| 太淡了加盐 | 调整 | 根据结果修改计划，再来一轮 |
| 出锅 | 终止 | 目标完成，返回最终回答 |

你会发现，做饭的循环和 Agent 的循环几乎一一对应。这不是巧合——**任何需要"做事情"的过程，本质上都是感知-推理-行动-观察的循环。**

区别在于：人做饭是凭经验和直觉，Agent 做事是靠 LLM 的推理能力。人尝一口就知道该加盐，Agent 需要看工具的返回值才能判断。人的循环是下意识的，Agent 的循环需要被显式地编程出来。

这个循环有几个关键特点：

**它是有目标的。** Agent 不是漫无目的地乱转。它有一个明确的目标（做一顿饭、写一份报告、修一个 bug），所有的行动都指向这个目标。

**它是有反馈的。** 每一步行动之后，Agent 都会看到结果。结果好就继续，结果不好就调整。没有反馈的循环是盲目的。

**它是有终止条件的。** 循环不会永远跑下去。要么目标完成了（饭做好了），要么达到了某个限制（最多做三次尝试），要么遇到了无法解决的问题（没有食材了）。知道什么时候停，和知道什么时候开始一样重要。

**它是有边界的。** Agent 不能做所有事。它只能使用它被赋予的工具。一个只有搜索工具的 Agent 不能帮你做饭，一个只有文件操作工具的 Agent 不能帮你订机票。工具决定了 Agent 的能力边界。

## 一个更贴近生活的场景：货比三家

做饭的类比可能还是太"厨房"了。让我换一个你一定经历过的场景：买东西。

你想买一台降噪耳机。预算 1000 块左右。你打开京东，搜"降噪耳机"，出来几百个结果。你点开几个，看参数、看评价、看价格。发现索尼的 WH-1000XM5 超预算了，苹果的 AirPods Pro 2 刚好在预算内但你用的是安卓手机。你又搜了几个品牌，对比了半小时，最后在三个候选里纠结。

这个过程你做了什么？感知（看商品页面）、推理（对比参数和价格）、行动（点开下一个商品）、观察（这个太贵了/那个评价不好）。循环了好几轮，最后做出决策。

现在想象你把这个任务交给一个 Agent。你说："帮我找一副 1000 块以内的降噪耳机，要求佩戴舒适、续航长、安卓兼容好。"

Agent 会这样做：

1. 搜索"2026 年降噪耳机推荐 1000 元以内"
2. 打开几个评测网页，提取各品牌的关键参数
3. 搜索具体型号的价格——京东、淘宝、拼多多分别查
4. 搜索用户评价，重点关注"佩戴舒适度"和"安卓兼容性"
5. 整理一个对比表格：型号、价格、续航、舒适度评分、安卓兼容性
6. 给出推荐："综合来看，XXX 最符合你的需求，理由是……"

你花半小时做的事，Agent 几分钟就做完了。而且它查的信息可能比你更全面——你可能只看了京东，它把三个平台都查了。

但这里有一个关键区别：**你买东西时，"舒适"这个标准是模糊的。** 你知道自己戴过哪些耳机觉得不舒服，但你没法给 Agent 一个精确的定义。Agent 能查到"舒适度评分 4.2"，但它不知道你对"舒适"的体感标准是什么。

这就是 Agent 的能力边界：它能帮你收集信息、做对比、给建议，但最终的判断还是你的。它能帮你把"货比三家"的过程从半小时压缩到几分钟，但它不能替你决定"哪个更舒服"。

这个场景说明了 Agent 和聊天机器人的核心差异：**聊天机器人能告诉你"降噪耳机有这些品牌"，Agent 能帮你"跑完整个选购流程直到给出推荐"。** 一个是在回答问题，一个是在帮你做事。

## Agent 不是自动化脚本

你可能会想：这不就是自动化吗？我用 Python 写一个脚本，也能搜索网页、读取文件、执行命令。Agent 跟自动化脚本有什么区别？

区别在于**决策**。

自动化脚本是人写的固定流程："先搜索 A，再搜索 B，把结果合并，写入文件。"每一步都是预先确定的。如果搜索 A 的结果变了、或者 B 不存在了，脚本要么报错，要么跳过——它不会自己决定"既然 B 不存在，那换个关键词搜搜看"。

Agent 是**运行时决策** 的。你给它一个目标（"调查某个行业"），它自己决定该搜索什么、该读哪些文件、该整理成什么格式。每一步的决策都基于上一步的结果。如果搜索结果不好，它会换关键词。如果网页打不开，它会换一个来源。如果信息矛盾，它会回去再查证。

打个比方：

自动化脚本像一条流水线。每个螺丝按固定的顺序拧上去。如果某个螺丝孔位置偏了，流水线停下来。

Agent 像一个工匠。目标是组装一件家具。如果某个螺丝孔位置偏了，工匠会自己判断：换个螺丝？还是先扩一下孔？还是这个零件该换一个？

**自动化脚本是"如果 A 则 B"的确定性逻辑。Agent 是"根据当前情况决定下一步"的概率性推理。**

这带来了巨大的灵活性——也带来了巨大的不确定性。Agent 可能做出你意想不到的决策（有时候是好的，有时候是坏的）。它可能走一条你没预设的路径。它可能在你没考虑过的边界情况下犯错。

这就是为什么造 Agent 比写自动化脚本复杂得多。你需要处理的不是"如果出错怎么办"，而是"如果它做了我没想到的事怎么办"。

## 20 行代码的魔法

如果你想自己试一下，用 AI 编程工具（比如 Cursor 或 Claude Code）告诉它：

> "帮我写一个最小的 Agent 循环：接收用户消息，调用 LLM API，如果 LLM 返回了工具调用就执行工具，把结果反馈给 LLM，循环直到 LLM 给出最终回答。"

它大概会给你 20 行代码。核心逻辑就是：

```
while 迭代次数 < 最大限制:
    回复 = 调用LLM(对话历史, 工具列表)

    if 回复里没有工具调用:
        输出最终回答
        结束

    for 每个工具调用:
        结果 = 执行工具(工具名, 参数)
        把结果加入对话历史
```

20 行。一个能调用工具、能多轮思考、能自己决定什么时候停的 Agent 就跑起来了。

你可以让它搜索网页、读取文件、执行代码。它会根据你的问题自己选择工具、执行、看结果、再决定下一步。

**这就是为什么现在是造 Agent 的时机。** 核心逻辑就这么点。以前难在模型不够聪明、API 不够便宜。现在模型够聪明了，API 够便宜了，20 行代码就能造出一个能做事的 Agent。

但接下来你会遇到真正的问题。

## 它跑起来了，然后呢？

你让这个 20 行代码的 Agent 帮你做一件事。它开始搜索、读文件、写代码、执行命令。循环了好几轮。看起来挺能干的。

然后你会发现第一个问题：**它不知道什么时候该停。**

你让它"优化这段代码"。它优化了一次，觉得可以再优化一次。优化完觉得还能再改改。改完觉得有个地方可以更好。十五轮之后，你的代码被改得面目全非，Agent 还在继续。

你让它"搜索最新的 AI 新闻"。它搜了一次，觉得结果不够多。又搜了一次，用不同的关键词。又搜了一次。十次搜索之后，它消耗了大量 token，给你返回了一大堆它自己都整理不完的信息。

你让它修一个 bug。它调了一个工具，工具报错。它又调了一次，同样的错误。又调了一次。又调了一次。八次同样的失败调用，每次都在浪费一次 LLM 调用，它完全不知道该停下来换个方法。

你让它帮你读一个大文件。文件有五万字。它把五万字全部塞进了对话历史。下一轮调用 LLM 的时候，上下文窗口被撑满了。Agent 开始"忘事"——它记不住你最早说的需求了。

你让它帮你改三个文件。它改了第一个，改了第二个，改到第三个发现第一个改错了，又回去改第一个。改完发现第二个跟第一个冲突了。来回改了六轮，三个文件都被改得面目全非。

你跟它聊了半小时，关掉了对话。第二天再打开，它完全不认识你了。你昨天告诉它的所有信息——你的项目结构、你的代码风格、你的偏好——全部消失了。

**启动一个 Agent 循环很容易。控制这个循环才是真正的工程。**

## 这些问题的答案

这本书的每一章，都在回答上面这些具体问题中的一个。

"它不知道什么时候该停"→ 第 2 章讲循环控制，第 9 章讲错误处理。

"它一直重试同一个失败命令"→ 第 4 章讲工具容错：重复检测、失败上限、全局安全网。

"它把五万字塞进上下文"→ 第 6 章讲上下文管理：什么时候压缩、怎么压缩、压缩到什么程度。

"它改文件改得面目全非"→ 第 5 章讲并行执行：哪些操作可以同时做，哪些必须按顺序来。

"它不认识你了"→ 第 7 章讲长期记忆：怎么让 Agent 跨会话记住你是谁。

"每次都在重复消耗 token"→ 第 8 章讲成本控制：怎么通过压缩、缓存和审计减少重复输入。

"它被恶意输入控制了"→ 第 10 章讲安全：六类威胁，逐层防御。

这些不是理论问题。它们是你造 Agent 的第一天就会遇到的真实问题。

## 为什么是现在

如果你三年前想造一个 Agent，你需要：

**一个足够聪明的模型。** 三年前的模型做不到稳定的工具调用。你告诉它"请用 JSON 格式返回工具名和参数"，它可能会返回一段自然语言，或者返回一个格式错误的 JSON，或者干脆忽略你的指令。现在，GPT-4、Claude、Gemini、DeepSeek 都能做到稳定的结构化输出。这不是小事——它是 Agent 循环能跑起来的前提。

**一套工具调用的标准。** 以前各家 API 各搞各的。你为 OpenAI 写的工具定义，换到 Anthropic 就不能用。现在，OpenAI 定义了 function calling 的格式，其他厂商跟进。你写一套工具定义，Claude 能用、GPT 能用、Gemini 也能用。Model Context Protocol（MCP）甚至让第三方工具可以即插即用。

**一个能帮你写代码的 AI。** Cursor、Claude Code、Codex——你用自然语言描述需求，它们帮你实现。你不需要先具备职业程序员级别的编码熟练度，但你必须知道自己想要什么功能、什么结果算正确、出了偏差怎么验收和纠正。这是最大的变化：以前"写不动代码"意味着"很难造软件"。现在，它更多意味着"你需要一个 AI 搭档，并且要有清晰的判断"。

**足够可控的 API 调用成本。** 一次 Agent 对话（5-10 轮工具调用）的成本已经降到个人开发者和小团队可以承受的范围。成本仍然重要，但不再是完全无法跨越的门槛。

这四个条件同时满足，创造了一个窗口：**现在是个人能造 Agent 的时代。**

不是因为个人突然变强了。是因为工具变好了。你不需要先成为职业程序员再开始造东西——你可以一边造一边学，让 AI 承担大量实现工作。你必须做好一件事：想清楚你要造什么，以及为什么这样造。

具体来说，AI 编程工具在这两年发生了什么变化？

**Cursor**：一个代码编辑器，内置了 AI。你打开一个空白文件，用中文写一句"帮我实现一个 HTTP 服务器，监听 3000 端口，返回 hello world"，它几秒钟就生成完整的代码。你看到代码里有个变量名拼错了，选中它，按一下快捷键，AI 帮你改好。你写了一半不确定接下来怎么写，它根据上下文自动补全。它不是在"辅助"你编程——它在"替你"编程。

**Claude Code**：一个命令行工具。你在终端里输入"帮我给这个项目加一个用户登录功能"，它会自己读你的代码库、理解项目结构、创建新文件、写好代码、甚至帮你跑测试。你不需要一行一行地写，你只需要描述你想要什么。它会主动问你："你希望用 JWT 还是 session 来管理登录状态？"——它在帮你做设计决策。

**Codex**：OpenAI 的编程 Agent。你给它一个任务描述，它在后台自己写代码、自己测试、自己修 bug，最后给你一个完成的 pull request。你全程不需要看代码，只需要在最后审查一下结果。

这些工具的共同点是：**造软件不再只属于职业程序员。** 你需要的是"会描述需求"和"会判断结果"。前者是产品经理和创业者的核心能力，后者来自技术常识、业务经验和持续验证。

这就是为什么现在是造 Agent 的时机——不只是模型够聪明、API 够便宜，更是因为**造软件本身的过程被 AI 改变了**。以前你有了想法，得先找程序员。现在你有了想法，打开 Cursor 就能开始。

## 一个真实的数字

AgentClaw 从第一个 commit 到可运行骨架，只用了几天。

到 4 月底，代码量已经增长到约 37,000 行 TypeScript，分布在 8 个包里。工具系统、记忆系统、上下文管理、错误处理、安全防护、多渠道接入——一个完整的 Agent 框架该有的东西都有了。

大量代码由 AI 编程工具生成，我负责需求定义、架构判断、结果验收和持续纠偏。每个工具的接口怎么设计、每个超时设多少秒、每个错误怎么处理、每个安全边界在哪里——这些判断加起来，才是造这个框架的真正工作量。

这个比例不是特例。它代表了一种新的工作方式：**你负责方向、拆解和验收，AI 负责大量实现。** 你不一定要亲手实现每个功能，但你必须知道什么是好的设计、什么是危险的实现、什么结果才算真正正确。

2026 年 2 月 20 日，春节假期，我打开了 Cursor，输入了第一句话："帮我创建一个 TypeScript monorepo 项目，包含 gateway、core、tools、memory 四个包。"

那天结束的时候，AgentClaw 已经能跑起来了：一个 CLI 界面，能接收消息，调用大语言模型，执行工具，返回结果。11 个工具——读文件、写文件、执行命令、搜索网页——全部在第一天就位。连 Telegram 机器人的接入都做好了。

第一天的感觉是：这也太快了。AI 写代码的速度远超我的预期。我说"加一个搜索工具"，两分钟后它就写好了。我说"加一个记忆系统"，十分钟后 SQLite 持久化就跑通了。我说"支持 WebSocket 连接"，半小时后前端就能跟后端对话了。

但第一个 bug 也来得很快。

第二天，我让 Agent 帮我搜索一个东西。它搜了，返回了结果。然后它又搜了一遍。又搜了一遍。我看着它连续搜了十五次，每次用的关键词都稍微有点不同，但结果大同小异。它不知道什么时候该停。

这个问题困扰了我好几天。我试了很多方法：限制搜索次数、限制总 token 消耗、在提示词里告诉它"搜两次就够了"。都不管用。最后我意识到，这不是一个"告诉它该怎么做"的问题——这是一个**架构问题**。你需要在框架层面设置硬限制，而不是依赖 LLM 自己判断该不该停。

这个认知改变了我对整个项目的理解。Agent 框架不是"给 LLM 提供工具"那么简单。它是"给 LLM 套上缰绳"——让它在安全的范围内做事，而不是放任它自由发挥。

随后几周里，它不断补齐上下文管理、记忆、浏览器、多渠道、安全和生产化能力。它接入了 Telegram、WhatsApp、钉钉、飞书、企业微信、QQ、Web 等渠道。每一个渠道的接入都踩过坑——不同平台的消息格式、文件上传限制、回调超时机制，没有一个是文档里完全写清楚的，全是边做边验证出来的。

## 这本书教什么

这本书不教你怎么用 ChatGPT。不教你怎么写 prompt。不教你怎么训练模型。

这本书教你：**怎么造一个 Agent 框架。**

Agent 框架是 Agent 的"操作系统"。它负责：

- 运行 Agent 的思考循环
- 管理 Agent 可以使用的工具
- 处理 Agent 和用户之间的对话
- 存储 Agent 的记忆
- 控制 Agent 的成本
- 保护 Agent 的安全
- 把 Agent 部署到真实的用户面前

你每天用的 ChatGPT、Claude、Gemini 背后都有一个框架。只是你看不到它。

这本书把它拆开给你看。

每一章解决一个具体问题。每一章都有真实的数据和踩坑经历。每一章末尾都有一个"动手试一试"，让你用 AI 编程工具自己试一下。

最终，你会理解一个 Agent 框架的完整结构——不是从教科书上学到的抽象概念，而是从一个真实项目中拆解出来的实战经验。
