# 第 9 章 LLM 会犯错，而且经常犯

---

## 凌晨两点的报警

2026 年 3 月的一个凌晨两点，我被手机震醒了。

不是闹钟，是 Telegram bot 的报警消息。AgentClaw 在处理一个用户的批量任务——帮他对二十份合同做条款摘要——突然卡住了。用户等了五分钟没有回复，发了一条"？"，又等了三分钟，发了一条"算了"。

我打开日志，看到了一串让人心凉的数据：

```
02:07:14  POST /v1/messages  429 Too Many Requests
02:07:15  retry in 1s...
02:07:16  POST /v1/messages  429 Too Many Requests
02:07:18  retry in 2s...
02:07:20  POST /v1/messages  429 Too Many Requests
02:07:24  retry in 4s...
...
02:10:31  POST /v1/messages  429 Too Many Requests
02:10:39  give up after 12 retries
```

Agent 在三分钟内疯狂重试了十二次。每一次都失败，每一次都浪费一次调用。指数退避（Exponential Backoff）——教科书上推荐的重试策略——在这个场景下变成了一个笑话：用户早就走了，Agent 还在忠实地等一秒、两秒、四秒、八秒地重试。

更糟糕的是，这次失败污染了整个会话。错误信息留在了对话历史里。当用户第二天重新打开对话时，Agent 看到了那些 429 错误记录，把它们当成了"上一次对话的上下文"，试图继续处理那些合同——但 API 限速已经解除了，它不需要"继续"，它需要"重新开始"。

这就是我要讲的第一个教训：**在 Agent 框架里，LLM 的错误不是异常，而是常态。**

三个半月的生产数据告诉我们：LLM 调用时不时会返回"不是干净回复"的东西。不是罕见事件——经常发生。

如果你的 Agent 框架只是在 LLM 调用外面套一层 `try/catch`，然后"catch 了就重试"——那你迟早会遇到和我一样的凌晨两点。

---

## 先分类，再反应

让我用一个生活类比来说清楚为什么"catch 了就重试"是错的。

想象你打电话给一家餐厅订位。电话打不通。你的反应取决于"为什么打不通"：

- **占线**：餐厅太忙了。等五分钟再打。大概率能通。
- **关机**：餐厅今天不营业。等没用，换一家。
- **空号**：号码错了。永远打不通。查一下正确号码。
- **欠费停机**：你手机欠费了。换手机打也没用，先把费交了。

如果你不区分原因，一律"等一秒再打"——占线的时候你等一秒就能打通，但关机的时候你会等一秒、两秒、四秒、八秒……等到天荒地老也打不通。空号的时候你换十部手机也打不通。欠费的时候你换一百个号码也打不通。

LLM API 的错误也是一样。一个 429（限速）和一个 401（密钥错误）需要完全不同的处理策略。如果你把它们当成一回事，要么浪费时间重试一个永远不会成功的请求，要么过早放弃一个只需要等一等就能恢复的请求。

AgentClaw 做的第一件事，就是建了一个错误分类器（Classifier）。每一个从 LLM API 返回的错误——不管 HTTP 状态码是什么，不管响应体（Response Body）里写的是什么——都会经过这个分类器，被归入七种类型之一。

## 七种错误，七种反应

让我用一个表格把它们列出来。这个表格不是理论——它是 AgentClaw 在生产环境运行一个多月后总结出来的真实分类：

| 错误类型 | 什么情况下触发 | 能重试吗 | 正确反应 |
|----------|---------------|---------|---------|
| 密钥错误（auth_failure） | HTTP 401、403，但不含"quota"关键词 | 不能 | 停下来。API 密钥错了或被撤销了，重试一百次也没用。 |
| 配额耗尽（quota_exceeded） | HTTP 403，响应体里有"quota" | 不能 | 停下来，通知运维。你花超预算了。 |
| 限速（rate_limited） | HTTP 429，或者响应体里有"rate limit" | 能 | 等 60 秒再试，或者换一个模型。 |
| 服务过载（overloaded） | HTTP 529、503，响应体里有"overload" | 能 | 等 15 秒。服务器忙，不是坏了。 |
| 服务器错误（server_error） | 任何 5xx，但不属于上面两种 | 能 | 立刻换模型，给出错的模型 30 秒冷却。 |
| 配置错误（config_error） | HTTP 400、413，"context length"、"too long" | 不能 | 这个模型处理不了你的请求。跳过它——可能是上下文太长了。 |
| 网络错误（network_error） | 连接重置、超时、"fetch failed" | 能 | 等 10 秒重试。临时抖动，大概率能恢复。 |

为什么是七种，而不是简单的"能重试 / 不能重试"两种？

因为**每种错误的"正确反应"完全不同**：

一个限速错误意味着**这个模型、此刻**——等一等就行。一个服务器错误意味着**这个模型可能挂了**——换一个试试。一个配置错误意味着**这个请求、这个模型、永远**——重试没用，要么换模型，要么改请求。一个密钥错误意味着**这个供应商、永远**——别浪费 API 调用了。

如果你只区分"能重试 / 不能重试"，你就不知道该等多久、该不该换模型、该不该通知用户。冷却时间、降级策略、日志记录全都取决于具体类型。

打个比方：你去看医生，"头疼"是一个症状。但医生不会只说"吃止痛药"。他会问你：是偏头痛？是高血压？是感冒？还是脑子里长了东西？同样是头疼，治疗方案天差地别。错误分类就是 Agent 的"诊断"——先搞清楚是什么病，再决定怎么治。

## 200 OK 的谎言

现在我要讲一个让很多人翻车的陷阱。

你可能以为，HTTP 状态码 200 就代表"成功"。在传统 Web 开发里，这基本是对的。但在 LLM API 的世界里，200 可以是一个谎言。

AgentClaw 曾经遇到过这样的情况：

```
HTTP/1.1 200 OK
Content-Type: application/json

{"error": "service temporarily unavailable"}
```

HTTP 状态码是 200。响应头（Response Header）说"一切正常"。但响应体（Response Body）里藏着一个错误。

为什么会这样？因为某些 LLM 提供商在过载时，会返回一个 200 状态码加上一个 JSON 格式的错误信息。也许是架构设计的问题，也许是负载均衡器（Load Balancer）的锅——具体原因不重要，重要的是你的 Agent 需要能处理这种情况。

如果你只检查状态码，你的 Agent 会把这个"假成功"当成真正的回复，传给 LLM 去处理。LLM 会试图理解这段错误信息，可能给出一个莫名其妙的回答，可能尝试"修复"它，可能浪费一个迭代周期。用户看到的是一个莫名其妙的回复，不知道发生了什么。

AgentClaw 的做法是：**即使状态码是 200，也要检查响应体。** 如果响应体里出现"service temporarily"或"temporarily unavailable"这类关键词，就把它归类为服务过载（overloaded），走和 503 一样的处理流程——等 15 秒，换模型。

永远不要只信任状态码。读响应体。

这个教训看起来简单，但很多框架——包括早期的 AgentClaw——都栽在这里。因为在 Web 开发的世界里，"200 就是成功"是常识。但在 LLM API 的世界里，常识不总是对的。

## 卡住了，但没报错

这是最微妙的一种失败模式。

没有 429，没有 500，没有错误信息。Agent 在正常运行。但如果你仔细观察它的行为，会发现一个令人不安的模式：

```
第 7 轮：让我检查一下配置文件...  file_read(/app/config.json)
第 8 轮：我来看看配置文件...    file_read(/app/config.json)
第 9 轮：让我再看一下配置...    file_read(/app/config.json)
```

每一轮的措辞略有不同——"检查一下"变成了"来看看"，又变成了"再看一下"。但做的事情完全一样：读同一个文件，用同样的路径。每次读完都得到同样的错误（比如文件不存在），然后它"想了想"，决定再读一次。

它不是在报错。它是在**原地打转**。

这在人类身上也会发生。你找钥匙，在客厅找了一圈没找到，想了想，又在客厅找了一圈。你的大脑告诉你"我在用不同的方式找"，但实际上你在重复同样的动作。区别在于：人找三圈之后会意识到"客厅没有，去卧室看看"。LLM 可能找三十圈也不会自己意识到。

AgentClaw 的解决方案叫做**输出指纹（Output Fingerprint）检测**。

原理很简单：把每一轮 LLM 输出的前 300 个字符（去掉空白后）和它调用的工具名拼在一起，形成一个"指纹"。保留最近三轮的指纹。如果这三个指纹有 80% 以上是相同的，就认为 Agent 卡住了。

一旦检测到卡住，AgentClaw 会做一件有意思的事：不是直接停掉 Agent，而是往对话里注入一段提示：

> "你似乎陷入了循环——你最近三次的回复非常相似。停止重复同样的方法。尝试完全不同的策略：使用不同的工具、改变参数、或者向用户解释是什么在阻碍你。"

然后给 Agent 一次机会自己调整。

这个方法在生产环境的效果出乎意料地好：多数卡住的情况，Agent 在看到这段提示后会自己"醒过来"，换一种方式继续。剩下的会继续打转，直到触发迭代次数上限，返回一个"半成品"回复。

为什么不是 100% 有效？因为有些情况下，Agent 确实只有一个"正确"的做法，但它做错了。比如它被要求修改一个不存在的文件，它反复尝试是因为它不知道文件不存在——每次它都在"换个方式试试"，但文件就是不存在。这时候提示它"换策略"也没用，因为没有其他策略可换。

但多数情况已经够了。关键洞察是：**一个卡住的模型不是一个 bug，而是一个缺失的信号。** 模型不知道自己在重复。告诉它就够了。很多时候，这就是它需要的全部信息。

这有点像你在考试时反复检查同一道题。监考老师走过来说"你已经在第三题上花了二十分钟了，先做后面的题"——这句话本身不解决第三题，但它打破了你的执念。

## 格式错误回滚：坏 JSON 不该永远留着

还有一种失败，比 API 错误更隐蔽，比卡住更难发现。

Agent 调用了一个工具，但传给工具的参数是坏掉的 JSON（JavaScript 对象标记，一种常用的数据格式）。工具解析失败，返回错误。这个错误被记录到对话历史里。下一轮，LLM 看到了错误，试图"修复"——但修出来还是坏 JSON。再下一轮，又修。又失败。

三轮下来，对话历史里塞满了三段坏 JSON、三条错误信息。Agent 的"工作台"——上下文窗口——被垃圾占了一大半。更糟糕的是，LLM 看到这么多错误记录，可能开始"焦虑"，输出质量明显下降。

在 AgentClaw 的早期版本里，这种情况每烧掉一轮迭代（Iteration），就扣掉一笔"预算"。如果你的 Agent 总共只有 15 轮迭代的预算——这是个很常见的设置——三轮坏 JSON 就吃掉了 20% 的预算，而且产出为零。

我们的解决方案叫做**格式错误回滚（Format Error Rollback）**。

核心思路：当一轮迭代中，所有工具调用都因为格式错误（JSON 解析失败或者工具名不存在）而失败时，我们认为这不是模型的错——是我们的解析器处理不了模型的输出。惩罚模型、扣它的迭代预算是罚错了对象。

所以我们会：

1. 删除这一轮的助手回复和所有工具结果，从对话历史里彻底抹掉
2. 把迭代计数器减一
3. 归还这一轮消耗的迭代预算
4. 让循环继续——模型获得了一次"重新来过"的机会，没有被垃圾污染的历史干扰

当然，不能无限回滚。如果模型连续三次都产出格式错误，说明它确实处理不了这个请求。三次之后，我们让错误保留，停止循环。

这里有一个设计判断值得展开说：**格式错误到底是谁的错？**

直觉上，你可能觉得"模型输出了坏 JSON，当然是模型的错"。但换个角度想：模型的工作是思考和表达意图。它说"我想编辑 app.ts 文件的第 10 行"——这个意图是清楚的。只是它表达的方式（JSON 格式）我们解析不了。如果一个翻译把法语翻成了半通不通的中文，你能说原始想法是错的吗？

所以格式错误回滚的本质是：**给模型一个"再表达一次"的机会，而不是因为表达方式的问题扣它的分。**

三次上限的意义在于：如果连续三次"再表达"都不行，那可能不只是表达的问题了——可能是这个任务本身超出了模型的能力。这时候该停就停。

## 降级链：首选不行，就换一个

现在我们知道了怎么分类错误、怎么处理卡住、怎么回滚格式错误。但还有一个关键问题：**当一个模型不行的时候，怎么换另一个？**

这个机制叫做降级链（Fallback Chain），或者叫故障转移（Failover）。名字听起来很高级，其实就是一句话：**首选不行，用备选；备选不行，用任何能用的。**

AgentClaw 的降级链分三层：

**第一层：显式配置。** 你提前告诉系统：做编程任务的时候，先试 Claude Sonnet，不行试 GPT-4o，再不行试 DeepSeek。系统按顺序走，跳过那些挂了的或者在冷却期的模型。

**第二层：按任务类型匹配。** 如果没有显式配置，系统会根据任务类型选模型。比如"规划"类任务用旗舰模型，"聊天"类任务用快速模型，"编程"类任务用标准模型。在对应的层级里选第一个能用的。

**第三层：兜底。** 如果上面两层都不行——你配置的模型全挂了，对应层级的模型也全挂了——就随便抓一个还能响应的模型。有总比没有好。

这个设计的关键在于：**冷却（Cooldown）是按模型追踪的，不是按供应商追踪的。**

什么意思？OpenAI 的 gpt-4o 被限速了，不代表 gpt-4o-mini 也不行。Anthropic 的 Claude Sonnet 过载了，不代表 Claude Haiku 也不行。每个模型的状态独立追踪：

```
anthropic::claude-sonnet    → 限速中，45 秒后恢复
openai::gpt-4o              → 可用
openai::gpt-4o-mini         → 服务器错误，20 秒后恢复
deepseek::deepseek-chat     → 可用
```

下一个请求来了，系统选第一个"可用"的模型——openai::gpt-4o。

不同错误类型的冷却时间也不同：限速等 60 秒（大多数提供商的限速在一分钟内重置），过载等 15 秒（过载通常是短暂的），服务器错误等 30 秒（给服务器恢复的时间），网络错误等 10 秒（临时抖动，等一下就好）。

你可能会问：为什么不直接用指数退避——等 1 秒、2 秒、4 秒、8 秒？因为指数退避在 Agent 场景下有一个致命问题：**用户在等。**

想象一下：用户发了一条消息，Agent 开始处理。LLM 被限速了。如果用指数退避，用户要等 1 秒 + 2 秒 + 4 秒 + 8 秒 + 16 秒 = 31 秒。而且每一轮等待之间还有 API 调用的延迟。用户盯着屏幕上的加载图标，等了半分钟，可能已经关掉了页面。

固定冷却时间的好处是**可预测**。限速就是 60 秒——不是 59 秒也不是 120 秒。运维可以告诉用户"请等一分钟再试"，而不是"请等一会儿"。可预测比"正确"更有用——至少在用户体验层面是这样。

## 流式输出的约束

这里有一个很多人忽略的约束：**在流式输出（Streaming）模式下，一旦第一个字已经发出去了，就不能换模型了。**

什么是流式输出？当你用 ChatGPT 或 Claude 的时候，回复不是一次性出现的——而是一个字一个字"流"出来的。这就是流式输出。好处是用户不用等整个回复生成完才看到内容，第一个字出来就能开始阅读。

但这也带来了一个问题。

想象一下：Agent 开始流式输出。前十个字出来了。然后 LLM API 返回了一个错误。这时候你有两个选择：

- **选择一：切到另一个模型继续。** 但新模型不知道前十个字是什么。它从零开始生成，接在那十个字后面。结果就是一段话的前半部分是 Claude 说的，后半部分是 GPT 说的——牛头不对马嘴。
- **选择二：丢掉这十个字，从头来。** 但用户已经看到了那十个字。突然消失了，换了一段完全不同的话。体验很差。

AgentClaw 的做法是：**在第一个字出来之前可以切模型，出来之后就不行了。**

具体来说：

- 流还没开始 → 捕获错误，标记模型不可用，尝试下一个。
- 流已经开始（第一个文本块或工具调用块已经发出去了）→ 抛出错误。部分回复无法恢复。

这是一个取舍。它意味着有些错误（比如模型在输出到一半时突然断开）会被直接暴露给用户，而不是被静默处理。但替代方案更糟——把两个不同模型的输出拼在一起，产生一段毫无意义的文字。

大多数框架要么完全不做流式故障转移（错过了"第一个字之前"可以切换的机会），要么试图把两个模型的输出拼接在一起（产生垃圾）。AgentClaw 选了一条中间路线：能切就切，不能切就诚实报错。

这个约束还有一个更实际的影响：**它决定了你的降级链在流式模式下能走多远。** 在非流式模式下，一个请求失败了，你可以从降级链的第一个模型试到最后一个。但在流式模式下，一旦开始输出，你就被"锁定"在当前模型上了。这意味着你需要在流开始之前就做好模型选择——不能"先随便选一个，不行再换"。预判比补救更重要。

## 一个完整的错误处理状态机

把上面所有机制放在一起，AgentClaw 的错误处理是一个状态机（State Machine）。让我用一段简化的伪代码来展示它的骨架：

状态机的核心分支如下：

- 格式错误：回滚本轮消息，最多回滚 3 次，然后继续同一轮。
- 全部工具调用失败：`consecutiveErrors++`；如果连续失败达到 3 次，提前停止。
- 连续 3 轮输出 80% 相似：注入一次升级提示，要求 Agent 换策略。
- 任何成功：重置所有错误计数器和相似度窗口。
- LLM API 错误：先分类，再决定冷却、降级或停止。

几个关键规则：

1. **任何成功都重置所有计数器。** 连续错误计数器、相似度滑动窗口——全部归零。一次成功就意味着"恢复正常"。
2. **格式错误和 API 错误分开处理。** 格式错误是"解析失败"，回滚重试。API 错误是"调用失败"，走降级链。
3. **卡住检测和错误计数并行运行。** 一个 Agent 可能同时"卡住了"（输出相似）和"报错了"（工具调用失败）。两个机制独立工作，不互相干扰。
4. **全局安全网。** 不管中间发生了什么，总工具调用次数超过 40 次就强制停止。这是最后一道防线——如果所有其他机制都失败了，这个数字保证 Agent 不会无限运行。

---

## 我们试过但删掉的方案

诚实地说，AgentClaw 的错误处理不是一开始就这么设计的。有几个方案我们试过，后来删掉了。

**指数退避。** 教科书推荐的重试策略。我们用了两周，然后删了。原因前面说过——用户等不了。一个 429 错误用指数退避重试五次，用户要等 31 秒。换成固定 60 秒冷却后，用户体验好了很多——因为运维可以给用户一个明确的预期："请等一分钟。"

**413 错误时自动裁剪上下文。** 当模型返回"context too long"的时候，自动删掉最老的消息再试。听起来合理，但我们发现：盲目裁剪会破坏工具调用和工具结果的配对关系（一个工具调用后面必须跟它的结果），删掉一半上下文后模型的输出质量比换一个更大上下文窗口的模型差得多。所以现在 413 被归类为"配置错误"——跳过这个模型，换一个能处理的。

**对认证失败做重试。** 有些框架在遇到 401/403 时会重试一次，"以防万一它是临时的"。我们试了。认证失败从来不是临时的。你的密钥要么是错的，要么过期了，要么被撤销了。每次重试都是浪费 API 调用，而且会让提供商的限速计数器多加一，让情况更糟。删了。

**这三个被删掉的方案有一个共同点：它们都是"看起来合理"的默认做法。** 指数退避是教科书标准。自动裁剪是直觉上的好主意。重试认证失败是"以防万一"的保守策略。但在 Agent 的具体场景下，它们都产生了比不做的更差的结果。

这再次印证了这本书反复说的一个观点：**Agent 工程的难点不是找到"正确"的方案，而是识别"看起来正确但实际上不对"的方案。**

## 一个真实的数字

让我分享一些生产环境的数据。

AgentClaw 运行一个多月后，我翻了一遍 LLM 调用的错误日志。其中：

- 限速（rate_limited）是最常见的错误——大家都在用 API 的时候
- 服务器错误（server_error）通常发生在深夜——提供商在做维护
- 网络错误（network_error）——临时抖动，什么时候都可能冒出来
- 过载（overloaded）——通常是某个热门模型的高峰期
- 配置错误（config_error）——通常是上下文太长
- 认证或配额问题——通常是因为环境变量没配对

降级链时不时会被触发——如果没有降级链，这些调用要么失败，要么让用户等很久。

格式错误回滚不常触发，但每次触发都省下了一到三轮迭代预算。

输出指纹检测更少见，但一旦触发，多数情况能通过注入提示词解决。

这些情况单独看都不大。但加在一起意味着：**LLM 调用时不时需要框架介入处理。** 如果你的框架不处理这些情况，它们就会变成用户看到的"莫名其妙的错误"、"为什么 Agent 在说胡话"、"怎么等了这么久还没回复"。

更有意思的是这些错误的时间分布。限速错误集中在工作日的白天——大家都在用 API 的时候。服务器错误集中在凌晨——提供商做维护的时候。网络错误则完全随机，什么时候都可能冒出来。如果你只在开发环境测试（通常是工作时间），你可能从来没见过凌晨的服务器错误。但你的用户在凌晨也会用你的 Agent。

还有一个容易忽略的成本：在没有错误处理机制的早期版本里，一个失控的重试循环就能烧掉不少 token。加上降级链和格式错误回滚之后，这部分浪费降到了接近零。

## 为什么不能让 LLM 自己处理错误

你可能会想：LLM 那么聪明，为什么不直接告诉它"如果出错了就换一种方式"？

我们试过。在系统提示词里加了这么一段："如果你的工具调用失败了，请分析错误原因并尝试不同的方法。"

结果是：

- 对于简单的错误（比如文件路径写错了），LLM 确实能自己修。但修的速度比框架直接重试慢三倍——因为它要先"分析"，再"推理"，再"尝试"。
- 对于 API 级别的错误（429、503），LLM 完全无能为力。它不知道该等多久，不知道该不该换模型。它唯一能做的就是"再试一次"——而这正是我们不想要的。
- 对于卡住的情况，LLM 完全意识不到自己卡住了。它每一轮都在"认真思考"，只是思考的结果恰好和上一轮一样。

**LLM 是大脑，不是神经系统。** 大脑负责思考和决策。神经系统负责反射——当你手碰到火的时候，你的手会先缩回来，然后大脑才知道"烫"。错误处理应该像反射一样快速、自动、不经思考。

框架应该做"反射"：检测错误、分类、冷却、降级——这些不需要 LLM 的"智能"，只需要确定性的规则。LLM 应该做"思考"：决定下一步做什么、怎么调整策略——这些需要推理能力，规则做不了。

把错误处理交给 LLM，就像让你的大脑来控制心跳——理论上可以，实际上不行，而且效率极低。

这里还有一个更深层的原因：LLM 处理错误需要消耗上下文窗口的空间。每一条错误信息、每一次失败的工具调用，都要占用本来可以用来"做事"的空间。如果你让 LLM 自己处理三轮错误，对话历史里就多了三条错误记录和三条"分析"。这些信息不但没用，还会干扰 LLM 对真正任务的理解。我们见过一个极端案例：Agent 在八轮重试之后，上下文窗口被错误信息填满了 40%，导致它"忘记"了用户最初的需求——把合同摘要做成了另一件事。

所以框架层处理错误不只是"更快"，还是"更干净"。回滚格式错误、冷却限速模型、降级到备选——这些操作的副作用是零：不增加上下文负担，不消耗迭代预算，不影响 LLM 的判断。而让 LLM 自己处理错误，每一步都在污染它的工作环境。

## 一个你应该问自己的问题

如果你正在造一个 Agent 框架，或者评估一个现有的框架，你应该问这个问题：

**"当 LLM API 返回 429 的时候，我的系统具体做了什么？"**

如果答案是"重试"——你需要追问：重试几次？每次等多久？等的时候用户看到什么？如果所有重试都失败了呢？

如果答案是"换模型"——你需要追问：怎么选下一个模型？怎么知道它没挂？如果所有模型都挂了呢？

如果答案是"我不知道"——那这就是你下一个要解决的问题。

错误处理不是"锦上添花"的功能。它是 Agent 能不能在真实世界里稳定运行的关键。一个没有错误处理的 Agent，就像一辆没有刹车的车——在车库里可能看着挺酷，上了路就是灾难。

## 本章小结

1. **LLM 错误是常态，不是异常。** 生产环境里时不时会出问题。你的框架必须假设每一次调用都可能失败。

2. **先分类，再反应。** 七种错误需要七种不同的处理策略。"catch 了就重试"是最差的方案。

3. **不要只信状态码。** 200 OK 的响应体里可能藏着错误。读响应体。

4. **卡住比报错更难发现。** 输出指纹检测能抓住多数卡住情况。告诉模型"你在重复"比让它自己猜有效得多。

5. **格式错误是框架的错，不是模型的错。** 回滚、不扣预算、给模型重新表达的机会。

6. **降级链要按模型追踪，不是按供应商。** 一个模型被限速不代表同供应商的其他模型也不行。

7. **流式输出有约束。** 第一个字之后不能切模型。能切就切，不能切就诚实报错。

**任务 1：为七种错误各写一条处理策略。**

拿一张纸，画一个七行的表格。左边写错误类型，右边写你的处理策略。要求：策略必须包含"等多久""换什么""通知谁"三个要素。如果你的 Agent 只用一个模型，"换什么"可以写"告诉用户模型暂时不可用"。

**任务 2：检查你的 200 OK。**

如果你有 LLM API 的调用日志，搜索一下 HTTP 状态码是 200 但响应体里包含"error"、"unavailable"、"overloaded"的记录。你可能会惊讶于有多少"假成功"。

**任务 3（可选）：设计一个"卡住检测"。**

假设你有一个 Agent，它每一轮都会输出一段文字和调用一个工具。你怎么判断它是不是在原地打转？写出你的检测规则——什么算"相似"？看多少轮？检测到了怎么办？
