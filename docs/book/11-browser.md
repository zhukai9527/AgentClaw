# 第 11 章 浏览器：Agent 的眼睛

---

## 一个订机票的故事

我让 Agent 帮我订一张机票。

"查一下下周三从北京到上海的航班，下午的，经济舱。"

Agent 很听话，打开了航空公司的网页。然后它卡住了。

不是网络问题，不是权限问题。是它"看"不懂那个网页。

航空公司的首页有一堆广告、一个搜索框、一个日期选择器、一个舱位下拉菜单、一个"搜索"按钮。人类一眼就知道该填哪里、点哪里。但 Agent 看到的是一份 340KB 的 HTML 文档——2,847 个 DOM 节点、186 个 CSS 类名、14 层嵌套的 `div` 才包住一个输入框。

如果把这份 HTML 原封不动塞给大语言模型（LLM），要烧掉 85,000 个 token。更糟的是，模型会在里面"幻觉"出一堆不存在的交互元素——它会告诉你"点击第 17 个按钮"，但那个按钮根本不是搜索按钮，是一个广告的关闭图标。

**浏览器自动化的最大难题不是控制浏览器，而是怎么让 AI 看懂一个网页。**

这一章，我们来解决这个问题。

---

## 核心判断

> **AI 不需要看到一个网页的全部。它只需要知道：这个页面上有什么可以点的、有什么可以填的、有什么可以读的——用最少的 token 告诉它这些就够了。**

这是整章的出发点。你选择的"网页表示方式"，决定了三件事：花多少钱、能不能成功操作、能不能处理没见过的页面。

---

## 三种看网页的方式

想象你是一个盲人，要在一个陌生的办公室里找打印机。

你有三种方式了解这个房间：

**方式一：让人给你拍张照片。** 照片很直观——你能看到房间的布局、家具的位置、打印机大概在哪个角落。但你是盲人，你看不到照片。你需要一个"读照片的人"告诉你"左边有一个白色的机器，大概在第三张桌子旁边"。这个过程很慢，而且读照片的人可能会把饮水机认成打印机。

**方式二：让人把房间里所有东西的标签都念一遍。** "地板是灰色瓷砖，尺寸 4 米乘 6 米。天花板有 12 块石膏板，每块 60 厘米乘 60 厘米。墙上有 3 个插座，型号分别是……"你会听到一大堆你不需要的信息，听了十分钟还没听到打印机在哪里。

**方式三：让人只告诉你"能用的东西"。** "门口有一个电灯开关。右手边有一台打印机，型号 HP LaserJet，正在待机。打印机旁边有一个放纸的柜子。再往前是一张桌子，上面有一台电脑，已开机。"两分钟就听完了。你知道什么能用、在哪里、怎么操作。

这三种方式，对应了 Agent 看网页的三种技术方案：截图、原始 HTML、无障碍快照（Accessibility Snapshot）。

---

### 截图：看得见，摸不着

截图是最直觉的方案。让浏览器截一张图，发给支持图片理解的模型（比如 GPT-4V 或 Claude 的视觉能力），让模型"看"图然后决定下一步。

听起来很美好。实际用了之后，问题一大堆。

**第一个问题：贵。** 视觉 token 的价格是文字 token 的 2 到 3 倍。一张网页截图消耗的 token 以万计。如果你的 Agent 需要浏览多个页面才能完成任务，光截图就是一大笔开销——只是为了让 AI "看"网页，还没开始做事。

**第二个问题：坐标不可靠。** 模型看到截图后，需要告诉你"点击坐标 (347, 215) 的位置"。但网页上的按钮位置会因为窗口大小、字体大小、浏览器版本而变化。今天 (347, 215) 是"搜索"按钮，明天可能就偏了几像素，点到了旁边的"高级搜索"。这种操作的成功率很低。

**第三个问题：丢失语义信息。** 模型看到一个蓝色的长方形，它不知道这是一个"提交按钮"还是一个"取消按钮"还是一个装饰性的色块。它看到一个输入框，不知道里面应该填邮箱还是密码还是手机号。截图只保留了视觉信息，丢失了所有语义。

截图方案的结论：**直观但脆弱，昂贵且不可靠。** AgentClaw 早期尝试过，很快放弃了。

---

### 原始 HTML：信息完整，噪声爆炸

既然截图丢了语义信息，那直接把网页的 HTML 拿给模型看呢？HTML 里什么都有——标签名告诉你这是按钮还是输入框，`placeholder` 属性告诉你该填什么，`href` 属性告诉你链接指向哪里。

理论上完美。实际上更糟糕。

一个典型的单页应用（SPA）页面，HTML 可能有几万个 token。一个电商产品页、一个社交媒体信息流、一个 SaaS 仪表盘——都是 token 大户。

这些 token 里有多少是有用的？很少。

剩下 95% 是什么？CSS 样式声明、SVG 图标代码、追踪脚本、广告占位符、一堆没有实际内容的嵌套 `div`——前端开发者管这叫"div 汤"。就像你去找打印机，有人把整个房间的装修图纸、电线走向、地板材质清单都念给你听了。信息是完整的，但你听了半小时还没听到打印机在哪里。

我们试过一个折中方案：不给 HTML，给 `innerText`——只提取页面上的文字。好了一点，但丢失了所有交互信息。模型知道页面上写了什么，但不知道哪些东西可以点、哪些输入框可以填。就像你知道房间里有"HP LaserJet"这几个字，但不知道它是一台真正的打印机还是一张打印机的广告图片。

原始 HTML 方案的结论：**信息完整但 token 灾难，噪声淹没信号。**

---

### 无障碍快照：为机器设计的页面摘要

第三种方案来自一个意想不到的地方：屏幕阅读器（Screen Reader）。

屏幕阅读器是视障人士使用的软件。它不能"看"屏幕，但它能读取网页的无障碍树（Accessibility Tree）——浏览器内部维护的一个结构化数据，记录了页面上每个可交互元素的角色、标签和状态。

"这是一个按钮，标签是'提交'。"
"这是一个输入框，类型是邮箱，标签是'电子邮箱地址'。"
"这是一个链接，文字是'忘记密码'，指向 /reset。"

这恰好是 Agent 需要的信息。不多不少。

无障碍快照的工作原理很简单：遍历 DOM 树，找到每一个可交互的元素（按钮、链接、输入框、有 ARIA 角色的元素），给它们编上号，然后返回一个紧凑的文本树。跳过所有 LLM 不需要的东西——样式、脚本、SVG 图标、隐藏元素。

一个登录页面的快照长这样：

```
URL: https://example.com/login
Title: 登录

# 登录你的账户
  [e1] input[email] "电子邮箱地址"
  [e2] input[password] "密码"
  [e3] a "忘记密码？" → /reset
  [e4] button "登录"
  [e5] a "创建账户" → /register
```

6 行。120 个 token。

同一份页面的原始 HTML 有上千行、几万个 token。**快照压缩了 99% 以上。**

Agent 读到这个快照，就知道该做什么了：往 e1 里填邮箱，往 e2 里填密码，然后点 e4。每个操作都用编号引用，不需要 CSS 选择器，不需要坐标计算，不需要猜。

---

## 为什么无障碍快照能工作

无障碍快照能工作，靠的是三个关键设计决策。

### 决策一：先标号，再读取

在生成快照之前，我们先往网页的 DOM 里注入 `data-ac-ref` 属性。每个可交互的元素——按钮、链接、输入框、有 ARIA 角色的元素——都被标上一个顺序编号：`e1`、`e2`、`e3`……

然后快照读取这些编号。

当 Agent 说"点击 e4"的时候，系统把它翻译成 `[data-ac-ref="e4"]`——一个 CSS 选择器，永远能找到对应的元素，因为我们刚刚才创建了它。

为什么这个顺序很重要？如果反过来——先读页面，再标号——那标号和页面之间就有一个时间差。在动态网页上，元素可能在你读取和标号之间发生变化。先标号再读取，保证了快照和实际页面的一一对应。

打个比方：你去一个陌生的图书馆找书。先标号再读取，就像管理员先把每本书贴上编号，再给你一份书目清单。你拿着清单上的编号去找书，一定能找到。反过来，先给你看一遍书架再贴编号，可能你记住的位置和编号对不上。

### 决策二：交互过滤

有时候 Agent 不需要读页面上的所有文字，只需要找到可以点击的按钮。比如一个设置页面有 200 个数据点和 8 个操作按钮，Agent 只需要知道那 8 个按钮在哪里。

`filter: "interactive"` 模式会把所有纯文本节点和非交互元素全部去掉，只留下可操作的部分。在上面的例子中，token 数量再砍 50% 到 60%。

这就像你去图书馆不是为了读书，而是为了找到借书处。管理员不需要给你念每本书的标题，只需要告诉你"前台在左手边，自助借还机在右手边"。

### 决策三：深度限制和可见性裁剪

DOM 树可以嵌套得很深。一个看起来简单的页面，背后可能有 20 层嵌套的 `div`。大多数深层嵌套对 Agent 没有意义。

我们做两个限制：

- DOM 遍历深度上限 10 层。超过 10 层的嵌套大概率是样式容器，没有交互意义。
- 跳过不可见元素。`offsetParent === null`（隐藏的）或 `aria-hidden="true"`（被标记为对辅助技术不可见）的元素直接跳过。没展开的下拉菜单、没打开的弹窗、屏幕外的内容——全部忽略。

---

## 实测效果

我们在真实网页上做了对比测试。原始 HTML 动辄几万 token，而无障碍快照只需要几千——压缩率在 90% 以上。登录表单这类简单页面更夸张，快照只需要原始 HTML 的不到 1%。

**无障碍快照就是网页的压缩版——为机器设计的压缩。** 屏幕阅读器需要这种压缩已经几十年了。LLM 现在需要它，原因一模一样：带宽有限、需要结构、不能容忍噪声。

这个差距直接反映在成本上。假设你的 Agent 完成一个任务需要浏览 5 个页面：截图方案的 token 消耗以万计，原始 HTML 以十万计，而无障碍快照只需要几千。同样的任务，成本差了一个数量级。

## 登录态：让 Agent 不用每次都登录

你让 Agent 帮你操作一个需要登录的网站。它打开了登录页，填了邮箱和密码，点了登录。成功了。

第二天，你让它再做同样的事。登录态过期了。它又得走一遍登录流程——填邮箱、填密码、可能还要过验证码、处理两步验证、关掉 Cookie 同意弹窗。六个工具调用，花了两分钟，就为了重复昨天做过的事。

如果每天都要登录一次，一个月就是 30 次。如果 Agent 需要操作 5 个需要登录的网站，一个月就是 150 次。每次 6 个工具调用，就是 900 次工具调用——纯粹浪费。

解决方法是**登录态持久化**（Browser State Persistence）。原理很简单：登录成功后，把浏览器的 Cookie 和 localStorage 保存成一个 JSON 文件。下次启动时，加载这个文件，浏览器就自动恢复到已登录状态。

AgentClaw 实现了两个命令：

- `save_state`：保存当前浏览器的登录状态到文件
- `load_state`：启动浏览器时加载之前保存的状态

```
# 第一次：Agent 登录后保存状态
Agent: browser_cdp { action: "save_state", name: "github" }
→ 登录状态已保存: data/browser-states/github.json

# 之后：直接加载，不需要重新登录
Agent: browser_cdp { action: "load_state", name: "github" }
→ 浏览器已加载保存的 Cookie 和 localStorage
```

一个容易踩的坑：Chrome 扩展产生的 Cookie，`sameSite` 属性的值可能是 `"unspecified"` 或 `"no_restriction"`——Playwright 不认识这些值，加载时会报一个莫名其妙的验证错误。需要在保存时做一次标准化：`unspecified` 和 `no_restriction` 变成 `"None"`，`lax` 变成 `"Lax"`，`strict` 变成 `"Strict"`。

还有一个更聪明的路径：通过 Chrome 扩展，直接从用户自己的浏览器里导出登录状态。用户在自己的 Chrome 里已经登录了 GitHub，扩展通过 WebSocket 连接到 AgentClaw 的网关，把当前标签页的 Cookie 和 localStorage 导出。Agent 完全不需要走登录流程——它直接复用用户的登录态。

这就像你让助理帮你去银行办事。你不用告诉他密码，直接把你的银行卡给他就行。他用你的卡办事，不需要自己再注册一个账户。

## 无头部署：服务器没有屏幕

你自己的电脑有屏幕。浏览器打开，你能看到它在做什么。但当你把 Agent 部署到服务器上——一台 Linux VPS 或者一个 Docker 容器——没有屏幕。

"没有屏幕怎么跑浏览器？"这个问题的答案是**无头模式**（Headless Mode）。浏览器在后台运行，该渲染渲染、该执行 JavaScript 执行 JavaScript，只是不显示窗口。就像一个演员在空房间里排练——没有观众，但该做的动作一个不少。

但这里面有一个关键的技术选择：用哪种无头模式？

Chrome 有两种无头模式。旧的 `--headless` 模式运行的是一个完全不同的渲染引擎——它和有头模式的渲染结果不一样。现代的单页应用经常在旧无头模式下渲染出不同的结果，甚至直接崩溃。

新的 `--headless=new` 模式（Chrome 112+）运行的是完整的浏览器——同样的渲染引擎、同样的 JavaScript 引擎、同样的 DevTools 协议。只是没有窗口。

AgentClaw 的检测逻辑是自动的，不需要手动配置：

```
Windows 或 macOS → 有头模式（显示浏览器窗口）
Linux 有 DISPLAY 或 WAYLAND_DISPLAY 环境变量 → 有头模式
Linux 没有显示环境 → 无头模式（--headless=new）
```

你的笔记本上能看到浏览器窗口，VPS 上自动切到无头模式，Docker 容器里也一样。开发者不需要管这件事。

Docker 环境还需要三个额外的启动参数，否则会出各种奇怪的问题：

| 参数 | 为什么需要 |
|------|-----------|
| `--no-sandbox` | Docker 容器默认用 root 运行，Chrome 的沙箱和容器本身的隔离冲突 |
| `--disable-dev-shm-usage` | Docker 默认的 `/dev/shm` 只有 64MB，Chrome 在那里写共享内存，满了就崩 |
| `--disable-gpu` | 大多数容器没有 GPU，不加这个会刷一堆（无害但烦人的）错误日志 |

这三个参数加了不会有任何副作用，但少了任何一个都可能在生产环境出问题。就像出门带伞——晴天无所谓，下雨就知道它有多重要。

## 连接可靠性：浏览器也会"死"

CDP（Chrome DevTools Protocol）连接会断。Chrome 会崩溃。远程调试端口会停止响应。

如果你的 Agent 框架不处理这些情况，用户会看到"Browser error: Target closed"，然后所有后续的浏览器命令全部失败，直到有人手动重启进程。

这不是偶发问题。在长时间运行的 Agent 会话中——比如一个需要执行半小时的自动化任务——浏览器连接中断是必然会发生的。

AgentClaw 实现了三层连接检查：

**第一层：连接状态检查。** 每次执行操作前，先调 `browser.isConnected()`。如果返回 `false`，说明 Chrome 已经退出了或者远程端口关了。重置所有状态，重新连接。

这一层能捕获"干净的断开"——Chrome 正常退出、端口正常关闭。

**第二层：活跃页面探针。** 即使 `isConnected()` 返回 `true`，连接也可能是"僵尸"状态——WebSocket 还开着，但浏览器进程已经死了。这时候调一个轻量的 CDP 命令（比如 `activePage.title()`），如果抛异常，就说明连接是僵尸的，重置并重新连接。

这就像你打电话给朋友。电话打通了（第一层通过），但对面一直没声音（第二层检测到异常）。你需要挂掉重拨。

**第三层：优雅的启动回退。** 当连接 `127.0.0.1:9222` 失败时，自动启动一个新的 Chrome 实例，然后最多重试 10 次连接，每次间隔 500 毫秒。如果 Chrome 已经在运行（带着调试端口），就连上它。如果没有，就启动一个新的。用户完全不需要手动管理 Chrome 进程。

三层下来，结果是：**Agent 可以在一个持续数小时的会话中使用浏览器，开发者不需要处理任何连接错误。**

## 两条路径：自己开浏览器 vs 用用户的浏览器

AgentClaw 支持两条不同的浏览器自动化路径，对应两种不同的使用场景：

**路径一：直接 CDP 模式。** Agent 自己启动一个 Chrome 实例（本地或远程），通过 CDP 协议控制它。适合自主任务、服务器部署、无头抓取。完全自包含，不依赖用户的浏览器。

```
Agent → browser_cdp 工具 → Playwright → Chrome（本地）
```

**路径二：Chrome 扩展模式。** 用户的 Chrome 浏览器安装了一个扩展，扩展通过 WebSocket 连接到 AgentClaw 的网关。Agent 通过网关的 HTTP API 发送命令，扩展在用户的标签页里执行，返回结果。适合需要用户登录态的场景、交互式工作流。

```
用户的 Chrome → 扩展（background.js）→ WebSocket → 网关 → Agent
```

两条路径产生相同的输出格式——带编号的无障碍快照。Agent 不需要知道当前用的是哪条路径。表示层（快照格式）就是抽象层。

这就像你让助理帮你办事，不需要关心他是自己开车去的还是坐地铁去的。他回来给你看结果，格式一样。

## 其他方案：我们评估过但放弃的

在找到无障碍快照之前，我们认真评估过几个替代方案。

**Lightpanda** ——一个用 Zig 写的无头浏览器，专门为 AI Agent 和网页抓取设计。号称资源消耗比 Chrome 低 10 到 50 倍。架构很有前途，但目前不支持 Windows，JavaScript 引擎兼容性也不够。我们在关注它的进展，但今天的生产环境，Chrome 是唯一能渲染出和用户看到的完全一致的页面的浏览器。

**Puppeteer** ——Google 官方的 Chrome DevTools Protocol 客户端。我们选了 Playwright 而不是 Puppeteer，三个原因：Playwright 的 `storageState` API 更完整、它的自动等待机制减少了不稳定的交互、它还支持 Firefox 和 WebKit 以备不时之需。底层的 CDP 层是一样的。

**Selenium** ——测试领域的老牌工具。但它的 WebDriver 协议在 CDP 之上多了一层翻译，引入了额外的延迟，也限制了对现代浏览器功能的访问。CDP 是浏览器的原生协议，不需要翻译。

**browser-use** ——一个 Python 库，把浏览器自动化包装成 LLM Agent 可用的形式。能力不错，但它是一个独立的 CLI 工具，不是一个可以嵌入 TypeScript 框架的库。我们需要浏览器自动化作为二十个工具中的一个，而不是作为整个 Agent 架构。

## AgentClaw 的实现骨架

把上面的设计落地，浏览器工具的核心接口其实很简单。以下是伪代码：

```
browser_cdp 工具支持的操作：

  navigate(url)       → 打开网页，返回无障碍快照
  click(ref)          → 点击编号为 ref 的元素
  type(ref, text)     → 在编号为 ref 的输入框里输入文字
  get_content()       → 获取当前页面的无障碍快照
  save_state(name)    → 保存登录状态
  load_state(name)    → 加载登录状态
  screenshot()        → 截图（主要用于调试）
```

Agent 调用 `navigate("https://example.com/login")`，得到一个快照。读快照，决定往 `e1` 填邮箱、`e2` 填密码、点 `e4`。三次工具调用，登录完成。如果需要保存登录态，再调一次 `save_state("example")`。

整个过程 Agent 不需要知道底层用的是 Playwright 还是 Puppeteer，是有头模式还是无头模式，是 CDP 还是 WebSocket。它只需要知道：我拿到了一个快照，快照里有编号，我用编号操作。

**抽象的力量在于：上层不需要知道下层的实现细节。**

## 一个完整的例子

让我们走一遍 Agent 登录 GitHub 的完整流程。

**第一步：打开登录页**

```
Agent: browser_cdp { action: "navigate", url: "https://github.com/login" }

返回：
URL: https://github.com/login
Title: Sign in to GitHub

# Sign in to GitHub
  [e1] input "Username or email address"
  [e2] input "Password"
  [e3] a "Forgot password?" → /password_reset
  [e4] button "Sign in"
  [e5] a "Create an account" → /signup
```

**第二步：填入凭据**

```
Agent: browser_cdp { action: "type", ref: "e1", text: "my-email@example.com" }
Agent: browser_cdp { action: "type", ref: "e2", text: "my-password" }
Agent: browser_cdp { action: "click", ref: "e4" }
```

**第三步：检查登录结果**

```
Agent: browser_cdp { action: "get_content" }

返回：
URL: https://github.com
Title: GitHub

# Dashboard
  [e1] a "Pull requests" → /pulls
  [e2] a "Issues" → /issues
  [e3] a "Marketplace" → /marketplace
  ...
```

登录成功。Agent 看到了 Dashboard 的快照。

**第四步：保存登录态**

```
Agent: browser_cdp { action: "save_state", name: "github" }
→ 登录状态已保存
```

下次需要操作 GitHub 时，直接 `load_state("github")`，不需要重新登录。

5 次工具调用，完成了登录和状态保存。如果没有登录态持久化，每次都要重复这 5 步。

## 真实世界的挑战

无障碍快照不是万能的。在实际使用中，我们遇到了几个没有完美解决方案的问题。

**动态内容加载。** 很多现代网页的内容是异步加载的——页面框架先出来，数据通过 API 后填充。如果在数据还没加载完的时候就截取快照，拿到的是一个空壳。解决方法是在 `get_content` 之前等一下，但"等多久"是一个经验值。等太短，内容没加载完；等太长，浪费时间。AgentClaw 的做法是等待页面的网络请求基本结束（2 秒内没有新请求），但不是所有场景都适用。

**Canvas 和 WebGL。** 有些网页的核心内容画在 Canvas 上（比如在线设计工具、地图、数据可视化）。Canvas 内部没有 DOM 节点，无障碍快照看不到里面的内容。这是一个目前没有好解法的问题——Canvas 对屏幕阅读器来说也是黑盒。

**验证码。** CAPTCHA 的设计初衷就是"不让机器通过"。无障碍快照能看到验证码的存在，但不能帮你解决它。如果遇到验证码，Agent 只能回来问用户："需要你手动完成验证码。"这不是 Agent 的能力边界，是验证码的设计目标。

**复杂的嵌套弹窗。** 有些网站的弹窗不是标准的 `dialog` 元素，而是一个绝对定位的 `div` 加上半透明遮罩。无障碍快照可能把弹窗和背后的内容混在一起。深度限制和可见性裁剪能缓解，但不能完全解决。

这些问题没有一个有完美的技术方案。**承认边界比假装完美更重要。** 你知道无障碍快照在什么场景下有效、在什么场景下会出问题，比相信它"能搞定一切"要好得多。

## 本章小结

1. **三种看网页的方式，一个最优解。** 截图贵且坐标不可靠，原始 HTML 噪声爆炸，无障碍快照把网页压缩成机器可读的结构——token 压缩率在 90% 以上。

2. **先标号再读取，不是反过来。** 给 DOM 元素注入 `data-ac-ref` 属性，然后用编号引用。Agent 说"点击 e4"，系统翻译成确定性的 CSS 选择器，不需要猜。

3. **登录态持久化是 ROI 最高的优化。** 一次登录，多次复用。`save_state` 和 `load_state` 让 Agent 不用每天重复登录流程。

4. **无头模式自动检测。** 有屏幕就显示窗口，没屏幕就后台运行。开发者不需要手动配置。

5. **三层连接检查保证可靠性。** `isConnected()` + 活跃页面探针 + 启动回退。浏览器连接断了自动恢复，用户无感知。

6. **抽象是关键。** 不管底层是 CDP 还是 Chrome 扩展，是有头还是无头，输出格式统一——带编号的无障碍快照。上层不需要知道下层细节。

7. **承认边界。** Canvas 内容、验证码、动态加载——有些问题目前没有完美解法。知道边界在哪里，比假装没有边界更重要。

**任务 1：手工制作一个无障碍快照。**

打开任意一个登录页面（比如 GitHub、Twitter、或者任何一个你常用的网站），用浏览器的开发者工具（按 F12），在 Console 里输入 `document.querySelectorAll('button, a, input, [role]')`，看看有多少可交互元素。然后试着把它们整理成无障碍快照的格式：

```
[e1] input "邮箱"
[e2] input "密码"
[e3] button "登录"
```

对比一下原始 HTML 有多少行。感受一下 95% 的压缩率是什么概念。

**任务 2：用 AI 编程工具写一个页面摘要器。**

提示词参考：

> "写一个函数，输入是一个网页的 HTML 字符串，输出是一个只包含可交互元素（button、a、input、select、textarea、有 role 属性的元素）的文本摘要。每个元素标注类型、文字标签和编号（e1, e2, e3...）。不需要运行，写伪代码即可。"

重点不是代码能不能跑，而是你理解"从 HTML 到快照"这个转换过程做了哪些过滤。

**任务 3（可选）：思考一个你希望 Agent 能自动操作的网站。**

列出这个网站的关键操作流程（比如"登录 → 进入设置页 → 修改头像 → 保存"），估算每一步需要几次工具调用，以及如果每次都要重新登录会多花多少次。这就是登录态持久化的价值。
