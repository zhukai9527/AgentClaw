# 第 13 章 从"能跑"到"能用"

---

## 那个周末，一切都在我电脑上好好的

2026 年 3 月的一个周五晚上，我在本地把 AgentClaw 跑通了。

搜索能用，记忆能存，Telegram 能回消息，Web 界面能打开。我兴奋地截图发了朋友圈："周末部署到服务器，给大家玩。"

周六早上，我租了一台便宜的 VPS（虚拟专用服务器，Virtual Private Server），SSH 上去，`git clone`，`npm install`，`node index.js`。跑起来了。我把 Telegram bot 链接分享给三个朋友。

一个小时后，第一个朋友发消息说"它回了一半就不动了"。第二个朋友说"它好像忘了我刚才说什么"。第三个朋友什么都没说——因为 bot 已经不回消息了。

我上去一看：进程崩了。没有日志，没有错误信息。SQLite 数据库被锁住了。有个朋友让它执行了一个 shell 命令，不知道怎么搞的，把 `data/` 目录删了大半。

没有备份。

那天下午我花了四个小时重建数据、排查问题、手动修复数据库。然后我又花了两周时间，把一个"在我电脑上能跑"的系统，改造成一个"在服务器上能用"的系统。

这两周做的事，就是这一章要讲的内容。

如果你正在考虑把自己的 Agent 部署到服务器上给别人用，这一章是你的"上线前必读"。不是因为它讲了什么高深的技术——恰恰相反，它讲的都是最朴素的东西：打包、备份、关闭、迁移、检查、日志、安全。没有一件是新鲜事。但每一件都是你跳过之后会在凌晨三点后悔的事。

**本章核心判断：生产可用（Production Readiness）不是一个功能，而是七件无聊但救命的事。跳过任何一件，你都会在凌晨三点学到它的教训。**

这一章可能不如前面的章节"有趣"。没有巧妙的架构设计，没有令人惊叹的 AI 能力，没有让你想发朋友圈的酷炫功能。这一章讲的全是"出了问题怎么办"——备份、关闭、迁移、检查、日志、安全。每一件都很无聊。每一件都是你上线后能不能睡好觉的决定因素。

我喜欢用一个比喻：生产可用就像房子的地基。没有人会参观你的房子时说"哇，你的地基真漂亮"。但如果你的地基有问题，整栋房子都会塌。这一章就是地基——你看不到它，但它撑着你所有的功能。

## 七件无聊的事

让我先列出来，后面逐一展开：

1. **容器化（Containerization）**——把你的环境打包，而不是依赖服务器装了什么
2. **自动备份（Automated Backup）**——数据库坏了能恢复，而不是从零开始
3. **优雅关闭（Graceful Shutdown）**——用户正在对话时别把进程杀掉
4. **数据库迁移（Schema Migration）**——升级功能时别把数据搞丢
5. **健康检查（Health Check）**——知道哪里坏了，而不是等用户来投诉
6. **日志与监控（Logging & Monitoring）**——出问题时能查到原因，而不是猜
7. **安全加固（Security Hardening）**——你的 Agent 能执行命令，那它也能被利用来干坏事

这七件事，没有一个能在 demo 里炫酷地展示。没有一个能让你在朋友圈获得点赞。但每一个都是你上线后能不能睡好觉的决定因素。

AgentClaw 在生产环境同时跑着六个渠道——Telegram、WhatsApp、钉钉、飞书、企业微信和 QQ。持续运行期间，它能扛住多渠道消息、工具调用、定时任务和部署重启，不是因为它从来不坏，而是因为每一层都设计了兜底：自动备份兜数据丢失，优雅关闭兜部署中断，健康检查兜服务降级，安全加固兜恶意输入。

这一章就是这些兜底的完整清单。

## 容器化：别让你的环境跟着代码走

### 为什么不能直接 `node index.js`

你可能会想：我在服务器上装好 Node.js，`git clone` 代码，`npm install`，`node index.js`，不就行了？

能跑。但有三个问题会让你半夜起来修服务器。

**第一个问题：环境不一致。** 你的本地机器是 Node 20，服务器是 Node 18。你的本地装了 Python 3.11（某些原生模块需要），服务器没有。你的本地有 `ffmpeg`（处理音频用的），服务器没装。代码一样，行为不一样——这就是经典的"在我电脑上能跑"（Works on My Machine）综合征。

**第二个问题：不可复现。** 你的服务器跑了一个多月，装了各种依赖。现在你想换一台服务器，或者加一台服务器做同样的事。你得回忆三个月里装了什么东西，按什么顺序装的，改了哪些配置。没有人记得住。

**第三个问题：安全隐患。** 你的 `.env` 文件里有 API 密钥、数据库密码、各种敏感信息。如果直接在服务器上跑，这些密钥就明文躺在磁盘上。如果有人拿到了服务器的访问权限，他就拿到了你所有的密钥。

容器化（Containerization）解决了这三个问题。Docker 是最常用的容器工具——它把你的代码、依赖、配置打包成一个标准的"箱子"，这个箱子在任何装了 Docker 的机器上都跑得一模一样。

### 多阶段构建：扔掉不需要的东西

Docker 有个关键技巧叫多阶段构建（Multi-stage Build）。打个比方：你要寄一个包裹，里面是一个组装好的家具。多阶段构建就是：先在一个工坊里把零件组装好（这个工坊很大，有很多工具），然后只把组装好的家具放进包裹箱寄出去。工坊里的锯子、锤子、螺丝刀不寄——收件人不需要它们。

AgentClaw 的 Dockerfile 就是这样做的：

**第一阶段（构建阶段）**：安装源代码、开发依赖、编译工具链（Python、make、g++——这些是用来编译 SQLite 原生模块的）。编译完成后，这个阶段的所有东西都可以扔掉。

**第二阶段（运行阶段）**：从一个干净的 `node:20-slim` 镜像开始，只复制编译好的产物和生产依赖。再装上 Agent 运行时真正需要的工具：`ffmpeg`（处理音频）、`git`（版本控制工具）、`curl`（网络请求）、`python3`（某些脚本依赖）。

结果是：构建阶段的源代码、测试文件、编译器全部不进最终镜像。最终镜像只包含运行 Agent 需要的最小集合。

为什么这很重要？两个原因。

**安全**：镜像里不包含源代码和测试文件，攻击面更小。你不会不小心把 API 密钥的备份文件打包进去（后面会专门讲 `.dockerignore`）。

**体积**：完整构建环境可能有几个 GB，运行时镜像可能只有几百 MB。更小的镜像意味着更快的部署、更少的存储成本、更快的扩缩容。

### 非 root 用户：一行代码，最高 ROI

容器默认以 root 用户运行。这意味着：如果你的 Agent 被骗执行了 `rm -rf /`（还记得第 10 章讲的 Prompt Injection 吗？），它能删掉容器里的所有文件。

解决方案很简单：在 Dockerfile 里创建一个专用用户，然后用 `USER agentclaw` 切换过去。这是一行代码——但它是你能加的安全措施中投资回报率（ROI, Return on Investment）最高的一个。

为什么 ROI 最高？因为这一行代码同时防护了所有场景：

- Prompt Injection 让 Agent 执行了危险的 shell 命令？非 root 用户权限有限，破坏范围被限制在 `/app/data` 目录内。
- Agent 的文件操作工具被利用来读取系统文件？`/etc/shadow`、`/proc` 等敏感路径对非 root 用户不可读。
- 某个依赖库有安全漏洞被远程利用？攻击者拿到的是一个低权限用户的 shell，不是 root。

**一行代码，限制了一整类攻击的破坏范围。** 这是安全工程里少见的高回报动作。

### .dockerignore：别把密钥打包进去

这是一个容易被忽略但后果严重的细节。

当你运行 `docker build` 时，Docker 会把当前目录下的所有文件发送给构建引擎。如果你的 `.env` 文件在当前目录里，它就会被复制进镜像的构建层（Build Layer）。

即使你在后面的步骤里删了它，它仍然存在于镜像的历史层中。任何能拉取这个镜像的人，都能用 `docker history` 命令看到它。

`.dockerignore` 文件的作用和 `.gitignore` 类似——告诉 Docker 哪些文件不要打包进镜像。AgentClaw 的 `.dockerignore` 排除了这些：

- `.env`、`.env.*`——所有环境变量文件
- `*.pem`、`*.key`——所有证书和密钥文件
- `data/`——整个数据目录（包含数据库、备份、临时文件）

**如果你从这一章只记住一件事，记住这个：`.dockerignore` 必须排除密钥文件。** 一个被推送到 Docker Registry 的镜像，如果里面包含 API 密钥，那就是一个定时炸弹——你不知道什么时候会爆炸，但你知道它一定会。

### 把它们串起来：docker-compose

实际部署时，AgentClaw 不是一个单独的容器，而是三个容器的组合：

1. **Agent 本体**——你的 Agent 框架
2. **SearXNG**——自托管的搜索引擎，替代付费搜索 API
3. **Valkey**（Redis 的开源替代）——缓存服务

三个容器通过 Docker 的内部网络通信，对外只暴露 Agent 的端口。而且绑定的是 `127.0.0.1`（本机回环地址），不是 `0.0.0.0`（所有网络接口）——这意味着只有本机能访问，外部网络访问不到。你需要自己配反向代理（比如 Nginx）来暴露服务。

为什么要自托管搜索引擎？因为商业搜索 API 通常按调用量计费。SearXNG 是开源的，部署在自己服务器上后，直接 API 调用成本会下降，但你仍然要承担服务器、维护和稳定性的成本。对于一个 Agent 框架来说，搜索是最常用的工具之一，成本会快速累积。

docker-compose 用一个 40 行的配置文件就能把这三个容器编排好。一条 `docker compose up` 命令启动全部，一条 `docker compose down` 停止全部。

**你可能会问：为什么不用 Kubernetes？** 因为我们的部署是一个 Agent、一个搜索引擎、一个缓存。Docker Compose 对这个规模来说刚刚好。Kubernetes 会增加学习曲线、运维复杂度和至少五个 YAML 文件——在当前规模下带来零收益。如果未来需要水平扩展或零停机滚动更新，再切换也不迟。

### 容器不是虚拟机

很多人第一次接触 Docker 时会把它想成"轻量级虚拟机"。这个类比大方向没错，但容易误导你做出错误的安全假设。

虚拟机模拟了整套硬件，有自己的操作系统内核。容器共享宿主机的操作系统内核，只是在用户空间做了隔离。这意味着容器比虚拟机轻得多（启动只要几秒，内存占用小得多），但也意味着隔离不如虚拟机彻底。

这对你的安全决策有什么影响？

**容器不是安全的银弹。** 如果宿主机的内核有漏洞，容器里的进程也可能利用这个漏洞"逃逸"到宿主机。Docker 的默认安全配置（seccomp 配置文件、AppArmor、capabilities 限制）能挡住大多数已知的攻击路径，但它不是万无一失的。

所以我们的安全策略是：容器是最后一道防线，不是唯一一道。前面的软件层防御（shell 沙箱、文件黑名单、SSRF 防护）是第一道到第四道防线。容器是第五道。五道防线叠加，比依赖任何单一道防线都安全得多。

### 端口绑定：127.0.0.1 vs 0.0.0.0

docker-compose 配置里有一个容易被忽略但很重要的细节：端口绑定。

`ports: "127.0.0.1:3100:3100"` 和 `ports: "3100:3100"` 看起来差不多，实际区别巨大。前者只允许本机访问，后者允许任何网络接口访问——也就是说，全世界都能连你的 Agent。

AgentClaw 绑定 `127.0.0.1`，意味着你必须自己配一个反向代理（比如 Nginx 或 Caddy）来对外暴露服务。反向代理负责 HTTPS 加密、请求限流、日志记录——这些是生产环境必需的，但不是 Agent 框架本身该管的事。

为什么要这样做？因为"默认安全"比"默认开放"好。一个绑定了 `0.0.0.0` 的 Agent，如果忘了配防火墙，全世界都能连上。一个绑定了 `127.0.0.1` 的 Agent，即使你什么安全措施都没加，外部也访问不到。安全应该从默认配置开始，而不是从"等我想起来再加"开始。

## 自动备份：出事了能恢复，而不是从零开始

### SQLite 的优势和风险

AgentClaw 用 SQLite 作为数据库。SQLite 是一个单文件数据库——你的所有数据（对话历史、长期记忆、任务列表、用户配置）都在一个文件里。

这是 SQLite 最大的优势：不需要安装数据库服务器，不需要配置连接池，不需要管理数据库进程。一个文件就是一个完整的数据库。你可以用 `cp` 命令复制它，用 `scp` 命令传输它，用 `ls -lh` 看它有多大。

这也是 SQLite 最大的风险：一个文件就是一个完整的数据库。如果这个文件损坏了——磁盘故障、写入中断、误删除——你的所有数据都没了。

你可能会说："我的服务器有 RAID，不会丢数据。"RAID 防的是硬件故障，不防人为错误。一个 `rm -rf data/` 命令，RAID 救不了你。

你可能会说："我有云服务商的快照。"快照是整个磁盘的镜像，恢复一个文件要恢复整个磁盘。而且快照通常是每天一次——如果损坏发生在两次快照之间，你丢失的是最后一次快照到损坏之间的所有数据。

**你需要的是：针对数据库文件的、每天自动运行的、有保留策略的备份。**

### copyFileSync：朴素但有效

AgentClaw 的备份策略极其简单：

```
每天凌晨 2:00 →
  复制数据库文件到 backups/ 目录（带日期戳） →
  删除超过 7 天的旧备份
```

就这些。没有复杂的增量备份，没有压缩，没有远程同步。就是一个文件复制，加上一个保留策略。

为什么这么朴素？因为**有没有备份，差别是无穷大；备份方案精不精细，差别是有限大。** 一个每天跑的 `copyFileSync`，比一个"计划做但还没做"的完美备份方案强无穷倍。

AgentClaw 在 WAL（Write-Ahead Logging，预写日志）模式下运行 SQLite。在这个模式下，`copyFileSync` 复制出来的文件是一致的——不会出现"复制到一半，数据库状态不完整"的问题。前提是：Agent 的流量不太高，文件在大多数时候处于一致状态。如果你的 Agent 每秒处理几十个并发请求，你应该用 SQLite 的在线备份 API，而不是简单的文件复制。

### 为什么保留 7 天

保留策略是：只保留最近 7 天的备份，更早的自动删除。

7 天是一个刻意的选择。它覆盖了这个场景：你做了一次数据库迁移（后面会讲），迁移看起来成功了，但一周后你才发现某个数据被搞坏了。7 天的备份窗口让你能回滚到迁移之前的状态。

为什么不保留更多？因为备份目录会无限增长。SQLite 数据库可能有几百 MB，7 天的备份就是几个 GB。如果你的 VPS 磁盘只有 40 GB，备份目录占满磁盘会导致整个系统崩溃——那比没有备份更糟。

为什么不保留更少？因为有些问题需要更长时间才会被发现。比如一个数据迁移改坏了某条历史记录，你可能几周后才注意到。

**7 天是一个平衡点：够长到覆盖大多数延迟发现问题，够短到不会撑爆磁盘。**

### 备份的真相

诚实地说：AgentClaw 的备份策略非常基础。它没有异地备份（备份文件和数据库在同一台服务器上），没有加密（备份文件明文存储），没有自动恢复（需要手动复制回来）。

这些在什么时候会成为问题？

**同一台服务器**：如果服务器硬件故障或被入侵，备份和数据一起没了。对于个人项目或小团队来说，这个风险可以接受——你可以在本地再保留一份。对于正式的商业部署，你应该把备份同步到另一台机器或云存储。

**没有加密**：如果有人拿到了服务器权限，备份文件也是明文可读的。对于存储了用户对话的 Agent 来说，这可能有隐私问题。加一层 GPG 加密不难，但 AgentClaw 目前没做。

**手动恢复**：如果数据库损坏，你需要自己把备份文件复制回来，重命名为数据库文件名。这个过程不复杂，但需要你 SSH 到服务器上手动操作。

**这些都是合理的简化——只要你清楚自己做了什么妥协。** 备份系统的关键不是做到完美，而是做到"有"。一个有基本备份的系统，比一个没有备份但"计划做完美方案"的系统，安全一万倍。

### 什么时候需要升级备份策略

AgentClaw 的简单备份策略有一个隐含的前提：**单实例、低流量、非关键数据。**

如果你的 Agent 存储了用户的真实对话记录、支付信息、或者任何有隐私法规要求的数据（比如 GDPR），你需要更认真的备份策略：

**异地备份**：备份文件和数据库在同一台服务器上，一场硬件故障或一次服务器入侵就能同时毁掉两者。至少把备份同步到另一台机器或云存储（S3、OSS 等）。可以用 `rsync` 命令定时同步，也可以用云存储的 CLI 工具上传。

**加密**：备份文件如果包含用户对话，就是敏感数据。用 GPG 或 OpenSSL 加一层加密，即使备份文件被泄露，没有密钥也读不到内容。

**自动恢复测试**：备份最大的谎言是"我有备份"——直到你需要恢复时才发现备份文件是坏的。每个月做一次恢复测试：把备份文件复制到一个临时目录，用 SQLite 的 `.open` 命令打开它，确认数据完整。这叫"备份演练"（Backup Drill），和消防演习一个道理——你不想在真正着火时才发现灭火器是空的。

**恢复点目标（RPO, Recovery Point Objective）和恢复时间目标（RTO, Recovery Time Objective）**：RPO 是"你能接受丢失多少数据"——如果 RPO 是 24 小时，每天备份一次就够了。RTO 是"你能接受多久不能用"——如果 RTO 是 1 小时，你需要一个能在 1 小时内完成恢复的方案。对于个人 Agent 项目，RPO 24 小时、RTO 几小时完全可接受。对于商业部署，你需要根据业务需求来定。

## 优雅关闭：别杀死正在对话的用户

### 为什么 `Ctrl+C` 不够

你按 `Ctrl+C`（或发送 `SIGTERM` 信号），操作系统告诉进程"请退出"。大多数程序收到这个信号就直接退出了。

但 Agent 不是"大多数程序"。当你按 `Ctrl+C` 的时候，可能有三个用户正在和 Agent 对话。其中一个用户的 Agent 正在调用 LLM API，已经等了 15 秒，还差几秒就能拿到回复。另一个用户的 Agent 正在执行一个工具调用，工具的结果还没返回。第三个用户刚发了消息，Agent 还没来得及处理。

如果你直接退出：

- 第一个用户：LLM 调用被中断，对话历史里留下一个没有对应结果的工具调用（tool_call）。下次打开对话时，这个悬空的调用会导致上下文混乱。
- 第二个用户：工具执行到一半被中断。如果这个工具是写文件，文件可能写了一半。如果是执行命令，命令可能执行到一半。
- 第三个用户：消息丢失。Agent 从没收到过这条消息。

**这不是理论上的问题。这是 AgentClaw 上线第一周就遇到的真实事故。** 部署新版后重启进程，三个用户的对话直接断了。其中一个用户的任务执行到一半——Agent 刚搜索了网页、读了三个文件、正在整理报告，重启后这一切都白做了。

### 三阶段关闭

优雅关闭（Graceful Shutdown）的核心思想是：不要立刻退出，而是分步骤有序退出。

**第一阶段：停止接受新工作。** 收到关闭信号后，立刻停止心跳、定时任务、渠道监听器和任务调度器。不再接受新的用户消息，不再启动新的 Agent 循环。但已经在运行的循环继续运行。

打个比方：餐厅要打烊了。门口挂上"停止接待"的牌子，但已经在吃饭的客人可以继续吃完。

**第二阶段：排空活跃对话。** 每隔一秒检查一次：还有多少个活跃的 Agent 循环？给它们最多 15 秒的时间完成。如果 15 秒后还有没完成的，记录警告日志，继续进入下一阶段。

为什么是 15 秒？因为 LLM 调用通常需要 5-30 秒。等 15 秒能覆盖大多数正在进行的调用，但不会让部署变得痛苦地慢。如果一个调用跑了 20 秒还没完成，它大概率是卡住了——一个被截断的回复，比一个永远挂着的部署进程要好。

**第三阶段：关闭服务器 + 强制退出超时。** 调用服务器的关闭方法，断开所有连接，然后退出进程。但设一个 30 秒的最终超时——如果 30 秒后进程还没退出，强制 `process.exit(1)`。

为什么需要这个 30 秒的"安全网"？因为有些情况下进程会永远无法自然退出：LLM 调用卡在网络超时上、某个资源没有被正确释放、某个 Promise 没有被正确 reject。没有这个强制退出，你需要手动 SSH 到服务器上 `kill -9` 进程——在凌晨三点，这不是你想做的事。

### 为什么 unref 很重要

30 秒的强制退出定时器有一个细节：它调用了 `unref()`。这意味着如果所有其他东西都正常关闭了，这个定时器不会阻止进程自然退出。

打个比方：你设了一个闹钟提醒自己"30 分钟后必须出门"。但如果 20 分钟的时候你已经出门了，闹钟不应该在你家空无一人时响——`unref()` 就是告诉 Node.js："如果其他事情都做完了，不用等这个定时器。"

这个细节看起来微不足道，但在实际部署中很重要。没有 `unref()`，即使所有对话都正常完成了，进程也会等 30 秒才退出。有 `unref()`，所有对话完成后进程立刻退出，只在有东西卡住时才等到 30 秒。

### 关闭和部署的关系

你可能会问：我又不经常重启服务器，优雅关闭真的重要吗？

重要。因为每次你更新代码、修复 bug、部署新版本，都需要重启进程。如果你的 Agent 24 小时在线服务用户，每天可能要重启 1-3 次（修 bug、发新功能、配置变更）。没有优雅关闭，每次重启都是对当前活跃用户的一次"事故"。

有了优雅关闭，重启变成了一件安全的事：正在对话的用户会收到当前回复（或者一个友好的"我需要重启，请稍后再试"的提示），不会看到一半就断了，对话历史也不会被破坏。

**优雅关闭不是一个"nice to have"的功能。它是你能正常迭代产品的前提。**

### 重启的标准流程

AgentClaw 有一个标准的重启脚本 `restart.ps1`（Windows PowerShell 脚本），它自动化了整个重启流程：

1. 停止旧进程（发送 SIGTERM，等待优雅关闭）
2. 构建新版本（`npm run build`）
3. 启动新进程

为什么需要一个脚本？因为手动重启有太多可以出错的地方：忘了停旧进程就启动新进程（端口冲突），忘了构建就启动（跑的还是旧代码），停了旧进程忘了启动新的（服务中断）。一个脚本把这些步骤固化下来，每次重启都走同一条路。

这个脚本接受一个 `-NoBuild` 参数，跳过构建步骤——当你只改了配置没改代码时，不需要重新构建。

**标准流程是生产运维的基础。** 不是因为流程有多复杂，而是因为手动操作一定会出错——尤其是在凌晨三点、你刚被电话叫醒的时候。把可以自动化的步骤自动化，把需要判断的步骤留给人。重启脚本属于前者。

### 当 Agent 有定时任务时

AgentClaw 运行着定时任务：数据库备份（每天凌晨 2 点）、健康检查（每小时）、对话标题生成（异步，用户第一条消息后触发）。优雅关闭需要处理这些定时任务：停掉定时器，等正在执行的任务完成，然后才进入下一步。

这比处理用户对话更复杂。用户对话有明确的"完成"状态——LLM 返回了最终回答。定时任务没有——备份脚本可能正在复制一个大文件，健康检查可能正在等待一个慢响应。你不能简单地"等 15 秒然后杀掉"，因为备份到一半的文件是损坏的。

AgentClaw 的做法是：在优雅关闭的第一阶段就停止调度器（不再启动新的定时任务），然后在第二阶段等待所有正在执行的任务完成。备份脚本会检查一个"是否正在关闭"的标志，如果在复制文件时收到了关闭信号，它会完成当前文件的复制再退出。

**优雅关闭的复杂度和你的系统状态量成正比。** 一个无状态的 Web 服务器只需要断开连接就行。一个有定时任务、有活跃对话、有数据库连接的 Agent，需要处理的状态多得多。这是生产化的真实成本——它不是一个功能，而是一层覆盖所有功能的协调机制。

## 数据库迁移：加列容易，改列要命

### 数据库会变

你的 Agent 在第一个版本可能只有两张表：`conversations`（对话）和 `messages`（消息）。一个月后你要加记忆功能，需要 `memories` 表。再过一个月你要加任务管理，需要 `tasks` 表。再过一个月你要给 `tasks` 表加一个 `priority` 字段。

数据库结构（Schema）会随着功能增长而不断变化。每次变化都是一次"迁移"——把旧结构变成新结构，同时不丢失已有数据。

传统的 Web 应用有成熟的迁移框架：Knex、Prisma Migrate、Alembic——它们能生成迁移脚本、回滚到之前的版本、在团队间同步数据库结构变化。但对于一个嵌入式 SQLite 数据库、一个单人维护的 Agent 框架来说，这些框架太重了。

### addColumnIfMissing：最朴素的迁移

AgentClaw 用一个极其简单的模式来处理列的新增：

```
addColumnIfMissing(db, 表名, 列名, 类型)
  → 查询表的当前结构（PRAGMA table_info）
  → 如果列不存在：ALTER TABLE ADD COLUMN
  → 如果列已存在：跳过
```

这个模式的核心是"幂等"（Idempotent）——执行一次和执行一百次的效果一样。每次 Agent 启动时，它都会运行所有的迁移检查。如果列已经存在了，就跳过。如果列不存在，就加上。

为什么每次启动都跑？因为这样你不需要手动记录"哪些迁移已经执行过了"。不管你启动多少次，结果都一样。你不需要维护一个迁移历史表，不需要担心"这个迁移在生产环境跑了没有"。

AgentClaw 累积了 25 次以上的列新增操作，涉及 5 张表，每次启动的总耗时不到 50 毫秒。这个成本完全可以忽略。

### 当朴素方案遇到墙

`addColumnIfMissing` 只能做一件事：加列。它不能改列名、改列类型、删列。这些操作在 SQLite 里需要重建整个表。

更麻烦的是 CHECK 约束（检查约束）。AgentClaw 的 `tasks` 表有一个 CHECK 约束，限制 `status` 字段只能是特定的值（比如 `pending`、`done`）。后来我们需要加新的状态值（`triaged`、`queued`、`running`），但 SQLite 不支持 `ALTER TABLE ... DROP CONSTRAINT`。

解决办法是"重建表"：

1. 创建一个新表 `tasks_new`，带有正确的结构（这次没有 CHECK 约束）
2. 把 `tasks` 的所有数据复制到 `tasks_new`
3. 删除 `tasks` 表
4. 把 `tasks_new` 重命名为 `tasks`
5. 重建索引

这个过程用数据库事务包裹，确保要么全部成功，要么全部回滚——不会出现"表删了但数据没复制完"的中间状态。

在执行之前，还会先"探测"：往旧表里插入一条带新状态值的测试行，如果失败了（CHECK 约束拒绝），才需要重建。这样对于已经不需要重建的数据库，就跳过了整个过程。

### 一个痛苦的教训

在两次表重建之后，AgentClaw 学到了一个教训：**不要在数据库层用 CHECK 约束来限制字段的可选值。** 因为可选值一定会变（业务需求会变），而 SQLite 的 CHECK 约束修改起来代价很高。

更好的做法是在应用代码里做校验：数据库层只管存什么类型的数据（字符串、数字、日期），具体值的合法性由代码来检查。代码改了重新部署就行，不需要动数据库结构。

### 诚实的边界

`addColumnIfMissing` 是一个极简的迁移方案，它有明确的边界：

**能做的**：加列、加表、加索引。这些都是"只增不改"的操作，幂等、安全、简单。

**不能做的**：改列名、改列类型、删列、改约束。这些需要表重建，复杂度高得多。

**适用场景**：单人维护的项目，SQLite 数据库，以"新增"为主的 schema 演化。

**不适用场景**：多人团队、需要回滚、需要跨环境同步、需要处理复杂的数据迁移逻辑。

如果你的项目发展到了那个阶段，你应该迁移到一个真正的迁移框架。但在此之前，`addColumnIfMissing` 够用了。够用就好——不要过度设计。

### 迁移失败怎么办

迁移可能失败。网络中断、磁盘空间不足、代码 bug——任何原因都可能导致迁移中途失败。

AgentClaw 的迁移用数据库事务包裹。事务（Transaction）是数据库的一个基本特性：要么里面的操作全部成功，要么全部回滚到事务开始前的状态。不会出现"表删了但数据没复制完"的中间状态。

但事务不是万能的。SQLite 的 `ALTER TABLE` 操作（比如 `ADD COLUMN`）是隐式提交的——它不能被回滚。这意味着如果 `addColumnIfMissing` 成功加了一列，但后面的某个迁移失败了，你不能回滚"加列"这个操作。

这在实践中不是大问题——加一列是幂等的，多加一次不会出错。但它提醒你：**迁移脚本的顺序很重要。** 如果你的迁移脚本有依赖关系（B 依赖 A 的结果），你需要确保 A 成功后才执行 B。AgentClaw 的做法是按表分组，每组内的迁移按顺序执行，组间互不依赖。

**迁移是生产环境中风险最高的操作之一。** 它直接修改数据结构，一次错误可能导致数据丢失。所以在执行迁移之前，先备份数据库（前面讲的自动备份在这里派上用场），然后在测试环境验证迁移脚本，最后才在生产环境执行。

## 健康检查：在用户投诉之前知道哪里坏了

### Agent 的依赖比你想的多

一个普通的 Web 应用可能只依赖数据库和缓存。但一个 Agent 框架依赖的东西多得多。

AgentClaw 的依赖清单：

- LLM API（Claude、OpenAI、Gemini 等）——Agent 的大脑
- SearXNG 搜索引擎——搜索工具的后端
- IMAP 邮件服务器——邮件相关工具
- 浏览器扩展——浏览器自动化工具
- ComfyUI 图像生成服务——图片生成工具

这些服务中的任何一个都可能独立挂掉。SearXNG 可能因为服务器重启而断开连接。LLM API 可能因为限速而返回错误。浏览器扩展可能因为 Chrome 更新而崩溃。

如果健康检查只是简单地返回"服务正常"或"服务异常"，它只能告诉你"有东西坏了"，但不能告诉你"是什么坏了"、"坏了多久了"、"坏了之后 Agent 该怎么做"。

### 只报变化

AgentClaw 的健康检查每小时运行一次，并发检查所有已配置的服务（每个检查 3 秒超时）。关键设计是：**只报告状态变化，不重复报告。**

如果 SearXNG 挂了，你会收到一条通知："SearXNG 搜索引擎连接超时。"如果它第二天还是挂的，你不会收到第二条通知。当它恢复时，你会收到一条通知："SearXNG 搜索引擎已恢复。"

为什么这样设计？因为"持续报错"比"不报错"更糟糕。如果 SearXNG 挂了 24 小时，你收到 24 条一模一样的通知，你会开始忽略所有通知——包括那些真正重要的、新出现的问题。

**只报变化意味着每条通知都值得关注。**

### 让 AI 知道哪里坏了

这是健康检查最不寻常的设计：**把检查结果注入到 LLM 的系统提示词里。**

当 SearXNG 不可用时，Agent 的系统提示词里会多出一段："[注意] 以下服务当前不可用：SearXNG 搜索引擎（连接超时）。涉及这些服务的请求请告知用户。"

这意味着 LLM 在处理用户消息之前就知道搜索功能不可用。它不会尝试调用搜索工具、等 3 秒超时、收到错误、再告诉用户"搜索不行"——它直接告诉用户："搜索功能目前不可用，但我可以用已有知识帮你。"

这个设计节省了一次工具调用的往返时间（几秒），也减少了一次 LLM 调用的 token 消耗，更重要的是——用户体验好了很多。用户不会看到"搜索失败"的错误信息，而是看到一个坦诚的告知。

**底层原则是：不要让 LLM 通过试错来发现系统故障。在系统提示词里直接告诉它。** 这个原则不仅适用于健康检查，也适用于所有你能提前知道的系统状态。比如：如果你知道某个 API 密钥过期了，不要等 LLM 调用失败再告诉它，在系统提示词里直接说。

### 健康检查 vs 心跳

有些人会把健康检查（Health Check）和心跳（Heartbeat）搞混。它们是两个不同的东西。

**心跳** 是 Agent 主动向外报告"我还活着"。通常是定期（比如每 30 秒）向一个监控服务发送一条消息。如果监控服务连续几分钟没收到心跳，就认为 Agent 挂了。心跳检测的是"进程是否存活"。

**健康检查** 是外部主动来问 Agent"你怎么样？"。Agent 检查自己的各个依赖服务，然后返回状态。健康检查检测的是"功能是否正常"。

一个 Agent 可以进程活着（心跳正常）但功能不可用（搜索挂了、LLM 限速了）。反过来，一个 Agent 可以某个依赖暂时挂了（健康检查报告异常）但进程本身还在运行（心跳正常）。

AgentClaw 同时使用了两者。心跳确保进程没崩。健康检查确保各个功能模块正常工作。两者互补，缺一不可。

### 检查什么、怎么检查

健康检查的价值取决于你检查了什么。检查得太少，漏掉了关键故障；检查得太多，产生噪音。

AgentClaw 检查的项目：

- **LLM API 可用性**：发一个最简单的请求（比如 `echo`），确认 API 能响应。这不检查模型是否正常（那是另一个问题），只检查"API 能不能连上"。
- **SearXNG 搜索引擎**：发一个 HTTP 请求，确认搜索引擎在运行。
- **IMAP 邮件服务器**：尝试连接 IMAP 端口，确认邮件服务可达。
- **浏览器扩展**：通过 WebSocket 发一个 ping，确认扩展在线。
- **ComfyUI 图像服务**：发一个 HTTP 请求，确认服务在运行。

每个检查 3 秒超时。为什么是 3 秒？因为健康检查每小时跑一次，如果某个检查卡住太久，会阻塞后续检查。3 秒足够判断"服务可达"，但不会因为网络抖动而误报。

一个重要的设计原则：**健康检查不应该调用 LLM。** 健康检查本身不应该产生 token 消耗。它检查的是"服务是否可达"，不是"服务是否能正常完成复杂任务"。用最简单的请求——一个 HTTP GET、一个 TCP 连接、一个 WebSocket ping——就够了。

## 日志和监控：出问题时能查到原因

### console.log 的诚实评估

让我们坦诚地说：AgentClaw 的日志方案是 `console.log` 加一个 `[tag]` 前缀。不是结构化日志，不是日志聚合，不是 ELK Stack。就是 `console.log`。

```
[gateway] Bootstrapping...
[gateway] Creating server...
[gateway] Server listening on http://0.0.0.0:3100
[health-check] 定时检查完成，无状态变化
[db-backup] Backed up to data/backups/agentclaw-2026-03-21.db
```

这够用吗？

**对于单实例部署，够用。** 你可以 `grep "[db-backup]"` 查找备份日志，`grep "[shutdown]"` 追踪关闭过程，`grep "[error]"` 查找错误。对于一个单人维护的系统，这覆盖了 90% 的排查需求。

**对于多实例部署，不够。** 如果你有三个 Agent 实例在运行，日志分散在三个地方，没有集中的日志查询，排查问题就是在三个终端窗口之间来回切。

### traces 表：最有价值的日志

AgentClaw 有一个比 `console.log` 更有价值的"日志"：SQLite 数据库里的 `traces` 表。

每次 LLM 调用，都会在 `traces` 表里记录：

- 输入和输出的 token 数量（成本追踪）
- 调用耗时（性能追踪）
- 使用了哪些工具（行为追踪）
- 是否出错，什么错误（错误追踪）
- 使用的模型（模型追踪）
- 来自哪个渠道（渠道追踪）

这给了你结构化的查询能力：

- "过去一小时有多少 LLM 错误？"——一条 SQL 查询
- "哪个工具最慢？"——一条 SQL 查询
- "今天的 token 消耗是多少？"——一条 SQL 查询
- "Telegram 渠道的平均响应时间是多少？"——一条 SQL 查询

这些查询如果用 `console.log` 来做，你需要写脚本去解析日志文件。用数据库来做，一条 SQL 就够了。

**务实的建议：如果你是单实例部署，`console.log` 加 `traces` 表覆盖 90% 的需求。如果你要扩展到多实例，在投资新功能之前，先投资结构化日志（比如 pino 或 winston）和日志聚合。你没法修复你在日志里看不到的问题。**

### 什么时候该升级日志方案

`console.log` 有几个固有的问题，在你规模变大时会越来越痛：

**没有日志级别。** 你没法区分"这条是调试信息"和"这条是严重错误"。所有日志都是同一个颜色、同一个通道。当你想找一个错误时，你得在几百条调试信息里翻。

**没有结构化数据。** 日志是纯文本。你想查"过去一小时有多少请求来自 Telegram 渠道"，你得写正则表达式去匹配。如果日志格式变了，你的正则就废了。

**没有上下文。** 一条日志说"LLM 调用失败"。哪个用户？哪个会话？用了什么模型？你得往前翻几十行才能找到这些信息。结构化日志会自动附带这些上下文。

什么时候该升级？当你发现自己在 `grep` 日志上花的时间超过 30 分钟时。当你需要回答"这个用户遇到了什么问题"却无法从日志里找到答案时。当你开始写脚本来解析日志时。

升级方案不复杂：用 pino（一个高性能的 Node.js 日志库）替换 `console.log`，日志输出为 JSON 格式。JSON 格式的日志可以被各种日志工具（Loki、Elasticsearch、甚至简单的 `jq` 命令）直接查询。

但在升级之前，先确认 `console.log` 真的不够用了。过早优化日志方案和过早优化其他任何东西一样——浪费精力。

### Sentry：错误不应该静默消失

除了 `console.log` 和 `traces` 表，AgentClaw 还集成了 Sentry（一个错误监控服务）。通过 `SENTRY_DSN` 环境变量启用，不配置时零开销。

哪些错误会被 Sentry 捕获？

- 启动失败（进程崩了，但你不知道）
- 未处理的异常（代码里有没 catch 的错误）
- 定时任务失败（备份脚本报错了，但你没看日志）

Sentry 的好处是：错误发生时它会发邮件或 Slack 通知给你。你不需要主动去看日志——日志是"出了问题你去找原因"，Sentry 是"出了问题它来找你"。

对于一个 24 小时运行的 Agent 来说，后者重要得多。你不可能 24 小时盯着日志。

### 监控的成本和收益

最后说一个诚实的判断：监控有成本。

Sentry 的免费额度对个人项目够用，但如果你的 Agent 每天处理上千次调用，你可能需要付费方案。结构化日志需要额外的存储和查询工具。`traces` 表会随着使用量增长——如果你每天记录 1000 条 trace，一年就是 36 万条，SQLite 文件会变大。

这些成本值不值得？

**取决于你的 Agent 服务多少人。** 如果只有你自己用，`console.log` 加偶尔看看数据库就够了。如果你服务 10 个朋友，加一个 Sentry 做错误通知。如果你服务 100 个用户，你需要 `traces` 表做成本和性能分析。如果你服务 1000 个用户，你需要结构化日志和日志聚合。

**监控的投入应该跟着用户规模走，不要超前也不要落后。** 超前会浪费精力在你不需要的基础设施上。落后会让你在出问题时抓瞎。

## 安全加固：你的 Agent 是一个攻击面

### 和传统 Web 应用不同的威胁模型

一个传统的 Web 应用，攻击者的目标是"进来"——通过 SQL 注入、XSS、CSRF 等手段突破你的防线，获取数据或执行操作。

一个 AI Agent 的威胁模型不同。攻击者不需要"进来"——用户本来就在和 Agent 对话。攻击者的目标是"控制你的 AI"——通过精心构造的输入（Prompt Injection），让 Agent 做它不该做的事。

Agent 能做的事越多，被控制后的危害就越大。AgentClaw 能读文件、执行 shell 命令、发 HTTP 请求、写入数据库——如果被控制，每一个能力都是一个攻击向量。

第 10 章详细讲了安全设计，这里只列出生产环境额外需要关注的加固措施。

### 容器化本身就是安全层

前面讲容器化时提到的"非 root 用户"，在这里再次出现——因为它同时也是一个安全措施。

当 Agent 以非 root 用户运行在容器里时，即使所有软件层的防御都失败了（Prompt Injection 绕过了 shell 沙箱、绕过了文件读取黑名单、绕过了 SSRF 防护），攻击者能做的事仍然有限。他不能安装软件包，不能修改系统文件，不能提升权限。他的活动范围被限制在容器内的 `/app/data` 目录。

**容器是最后的安全网。** 不是第一道，是最后一道。前面的防线（shell 沙箱、文件黑名单、SSRF 防护、记忆扫描）挡住大部分攻击。容器确保"即使前面都失败了，损失也是有限的"。

在安全工程里有一个概念叫"纵深防御"（Defense in Depth）——不指望任何单层防御能挡住所有攻击，而是叠加多层防御，每层挡住不同的攻击类型。AgentClaw 的安全层是这样的：

| 层级 | 防御措施 | 挡住什么 |
|------|---------|---------|
| 第 1 层 | Shell 命令校验 | 破坏性命令（`rm -rf /`、`mkfs`、`dd`） |
| 第 2 层 | 文件读取黑名单 | 敏感文件泄露（`.env`、`.pem`、`id_rsa`） |
| 第 3 层 | SSRF 防护 | 私有网络访问和云元数据泄露 |
| 第 4 层 | 记忆写入扫描 | 长期记忆被污染（持久性 Prompt Injection） |
| 第 5 层 | 非 root 容器 | 以上所有防御都失败后的损害限制 |

**没有哪一层是充分的。每一层挡住不同的攻击。** 五层叠加，才构成一个可接受的安全姿态。

### 安全的诚实评估

第 10 章讲过，安全不是"防住了"和"没防住"的二元状态。它是一个频谱——你降低了多少比例的风险，以及每次突破后损失有多大。

AgentClaw 的安全设计降低了哪些风险？

**拦住了大部分意外风险。** 一个普通用户让 Agent 执行危险命令，被 shell 沙箱拦住了。Agent 被 Prompt Injection 诱导读取 `.env` 文件，被文件黑名单拦住了。这些都是最常见的场景——意外比蓄意攻击多得多。

**拦住了大部分低成本自动化攻击。** 爬虫、脚本小子、自动扫描工具——它们用的是已知的攻击模式，软件层防御能挡住不少。

**但没有防住什么？** 一个足够聪明的、有针对性的攻击者，可能找到绕过 shell 沙箱的方法（正则表达式不是完美的）。一个有耐心的攻击者，可能通过多轮对话逐步诱导 Agent 泄露信息（每一轮看起来都无害，但组合起来有风险）。一个有资源的攻击者，可能利用零日漏洞逃逸容器。

**这些"没有防住"的不是失败——它们是你需要知道的风险。** 安全工程的精髓不是"造一个不可攻破的系统"（那不存在），而是"清楚地知道你的防线在哪里，以及突破后损失有多大"。

对于一个个人项目或小团队的 Agent，五层防御能把常见风险降到更可接受的范围。你不需要一开始就达到银行级别的安全，但需要比最常见的攻击多一层防御。

## 上线检查清单：做到 12 项以上才算生产系统

把前面所有内容汇总成一个清单。你可以用它来评估你自己的 Agent 项目离"生产可用"还差多远。

### 容器化

| 检查项 | 如果跳过会怎样 |
|--------|--------------|
| 多阶段 Docker 构建 | "在我电脑上能跑"综合征 |
| 容器内用非 root 用户 | 一条危险命令 = 整个系统沦陷 |
| `.dockerignore` 排除 `.env` 和密钥 | API 密钥被永久刻入镜像层 |
| 显式声明运行时依赖 | 基础镜像更新后"昨天还好好的" |

### 备份

| 检查项 | 如果跳过会怎样 |
|--------|--------------|
| 每日自动数据库备份 | 一次数据损坏 = 全部从零开始 |
| 保留策略（保留 N 天，删除旧的） | 备份目录增长直到磁盘满 |

### 关闭

| 检查项 | 如果跳过会怎样 |
|--------|--------------|
| 信号处理（SIGTERM/SIGINT） | 对话中断、历史损坏 |
| 活跃对话排空（带超时） | 部署要等很久或杀掉正在回复的用户 |
| 强制退出超时（带 unref） | 进程卡死需要手动 kill |

### 迁移

| 检查项 | 如果跳过会怎样 |
|--------|--------------|
| 启动时运行幂等迁移 | 每次部署手动执行 ALTER TABLE |
| 表重建策略处理约束变更 | 卡在旧 schema，无法加新功能 |

### 健康检查

| 检查项 | 如果跳过会怎样 |
|--------|--------------|
| 依赖服务健康检查 | LLM 浪费迭代次数来发现工具坏了 |
| 状态变化通知 | 你从用户那里得知服务挂了 |
| 健康状态注入 LLM 上下文 | Agent 反复尝试调用坏掉的工具 |

### 监控

| 检查项 | 如果跳过会怎样 |
|--------|--------------|
| 错误追踪（Sentry 或同等工具） | 静默失败不断累积 |
| 每次调用的 trace 记录 | 无法回答"为什么这么慢/这么贵？" |
| 结构化日志或带标签的 console.log | 调试只能靠肉眼读原始输出 |

### 安全

| 检查项 | 如果跳过会怎样 |
|--------|--------------|
| Shell 命令校验 | Prompt Injection 执行 `rm -rf /` |
| 文件读取黑名单 | Prompt Injection 泄露 API 密钥 |
| SSRF 防护 | Prompt Injection 窃取云实例凭证 |
| 记忆写入扫描 | 一次坏对话 = Agent 的长期记忆被永久投毒 |
| 非 root 容器执行 | 以上全部失败后的最后防线 |

**总共 20 项。如果你的项目做到了 12 项以上，你可以有信心把它交给用户。如果少于 12 项，你跑的是一个 demo，不是生产系统。** demo 没什么不好——开发阶段就是需要 demo。但不要给 demo 赋予"生产"的期望。

## 我们评估过但放弃的方案

### PostgreSQL 替代 SQLite

对于单实例 Agent，SQLite 更简单、更快、不需要单独的数据库进程。迁移的故事更差（没有 `ALTER COLUMN`，有限的 `ALTER TABLE`），但运维的简单性——你的整个数据库就是一个文件，可以 `cp` 复制——超过了这个劣势。

什么时候切换？当多个 Agent 实例的并发写入压力导致 WAL 竞争时。

### Kubernetes 替代 Docker Compose

我们的部署是一个 Agent、一个搜索引擎、一个缓存。Docker Compose 用一个 40 行的 YAML 文件搞定。Kubernetes 会增加学习曲线、运维复杂度和至少 5 个 YAML 文件——在当前规模下零收益。

什么时候切换？当我们需要水平扩展或零停机滚动更新时。

### 完整迁移框架（Knex、Prisma Migrate）

这些工具在你有团队、有 staging 环境、有回滚需求时很优秀。对于一个单人维护的 SQLite 数据库、以"新增"为主的 schema 变化，`addColumnIfMissing` 在启动时运行更简单，且零依赖。

什么时候切换？当我们需要回滚或非新增式迁移时。

**这三个决策有一个共同模式：选择当前规模下最简单的方案，但清楚地知道什么时候需要升级。** 过度工程（Over-engineering）比技术债务更危险——因为过度工程消耗的是你现在的精力，而技术债务只是预支了你未来的精力。在 Agent 的早期阶段，你的精力应该花在功能和用户体验上，不是花在运维基础设施上。

### 其他被拒绝的方案

除了上面三个，还有一些方案我们认真考虑过但最终没有采用：

**Prometheus + Grafana 监控面板。** 漂亮的仪表盘，实时显示 CPU、内存、请求数、错误率。对于一个只有几个人用的 Agent 来说，这是用大炮打蚊子。`console.log` 加 `traces` 表够用了。什么时候上？当你需要在一个屏幕上同时看 10 个指标时。

**Nginx 反向代理自动化。** docker-compose 里加一个 Nginx 容器，自动配置 HTTPS 和反向代理。我们选择手动配 Nginx，因为：(1) HTTPS 证书用 Let's Encrypt 自动续期，需要和 Nginx 配合；(2) 手动配一次就行，不需要自动化。什么时候自动化？当你需要频繁添加新域名或新服务时。

**CI/CD 流水线。** GitHub Actions 自动构建、测试、部署。AgentClaw 目前是手动部署——SSH 到服务器，`git pull`，`docker compose up -d`。手动部署的好处是：每次部署你都知道改了什么，不会因为自动部署了一个坏版本而半夜起来修。什么时候上 CI/CD？当你团队超过一个人，或者部署频率超过每天一次时。

**这些"没做的事"和"做了的事"同样重要。** 它们代表了一个判断：当前阶段什么值得投入精力，什么不值得。每个项目都有自己的"当前阶段"——你需要根据自己的情况做判断，而不是盲目照搬别人的方案。

## 一个真实的数字

让我用一个具体案例把这些概念串起来。

AgentClaw 上线后的第一个月，发生了这些事：

- SQLite 数据库因为一次异常写入被锁住，进程卡死。没有优雅关闭，重启后三个用户的对话历史损坏了。**教训**：加了优雅关闭。

- 一次代码更新改了数据库结构，但忘了通知服务器。部署后 Agent 启动失败，报"column not found"错误。**教训**：加了启动时自动迁移。

- 用户在 Telegram 里让 Agent 搜索一个关键词，但 SearXNG 刚好挂了。Agent 调了搜索工具，等了 3 秒超时，报错。然后它又试了一次。又试了一次。三次搜索尝试，浪费了 9 秒和三次 LLM 调用的 token。**教训**：加了健康检查，把服务状态注入系统提示词。

- 内部测试中，有人通过 Prompt Injection 让 Agent 尝试读取 `.env` 文件。文件读取工具没有黑名单，Agent 成功读到了敏感配置。**教训**：加了文件读取黑名单，并把密钥轮换和日志审计纳入上线流程。

- 一次半夜的定时任务失败了（备份脚本因为磁盘空间不足而报错）。没人发现，直到三天后数据库损坏需要恢复备份，才发现备份文件已经三天没更新了。**教训**：加了 Sentry 错误监控。

这五个事故，每一个都对应了前面讲的一个系统。没有哪个系统是"锦上添花"的——每一个都是被真实事故逼出来的。

### Before 和 After

让我用具体数字展示这些系统上线前后的差异：

**备份**：上线前，数据库损坏意味着从零开始重建所有对话历史和记忆。上线后，数据库损坏意味着从备份恢复最近一天的数据。损失从"全部"变成了"最多一天"。

**优雅关闭**：上线前，每次部署都会中断 2-3 个活跃对话，用户看到"bot 不回消息了"。上线后，部署过程中用户会收到当前回复的完整内容，或者一个"正在重启，请稍候"的提示。用户感知到的"宕机时间"从"每次部署几分钟"变成了"几乎为零"。

**健康检查**：上线前，SearXNG 挂了之后 Agent 会反复尝试搜索，浪费 3-5 次工具调用的 token 和时间。上线后，Agent 直接告知用户搜索不可用，一次工具调用都不浪费。

**安全加固**：早期测试中，有一次 Prompt Injection 成功读到了 `.env` 文件。加固后，同样的攻击被文件读取黑名单拦截，Agent 返回"无法读取该文件"。从"直接暴露"变成"被规则拦截"。

这些数字不惊人。备份没有把恢复时间从小时降到秒，优雅关闭没有实现零停机部署，健康检查没有实现自动故障转移。但每一个数字都是从"出了问题不知道怎么办"变成"出了问题有预案"。

**生产可用不是追求完美，是把每一种"出了问题怎么办"都提前想好。**

## 本章小结

1. **生产可用不是功能，是七件无聊的事。** 容器化、备份、优雅关闭、数据库迁移、健康检查、日志监控、安全加固。跳过任何一件，你都会在凌晨三点学到它的教训。

2. **容器化解决三个问题：** 环境一致性、可复现性、安全性。多阶段构建扔掉不需要的东西，非 root 用户限制攻击面，`.dockerignore` 排除密钥文件。

3. **备份的关键是"有"，不是"完美"。** 一个每天跑的 `copyFileSync`，比一个"计划做但没做"的完美方案强无穷倍。保留 7 天够用。

4. **优雅关闭分三步：** 停止接受新工作 → 等待活跃操作完成（15 秒）→ 强制退出超时（30 秒）。这让你能安全地部署新版本。

5. **数据库迁移用最简单的方案：** `addColumnIfMissing` 幂等模式，启动时运行。简单有效，但有明确的边界——只能加列，不能改列。

6. **健康检查不只告诉你坏了，还要告诉 LLM 坏了。** 把服务状态注入系统提示词，让 Agent 不再尝试调用坏掉的工具。

7. **20 项检查，做到 12 项以上才算生产系统。** 如果你做不到 12 项，你跑的是 demo——demo 很好，但不要给它"生产"的期望。
