# 第 5 章 并行：让 Agent 同时做几件事

---

## 一个等待了 12 秒的任务

有一次，我让 AgentClaw 帮我分析一个项目的代码质量。需求是：找到所有 TypeScript 文件，统计每个文件的行数，检查有没有未使用的 import，最后整理成一份报告。

Agent 开始工作了。它先调用 `glob` 找所有 `.ts` 文件。等了 800 毫秒，拿到了文件列表。然后它调用 `grep` 搜索未使用的 import。又等了 600 毫秒。然后它调用 `file_read` 读取第一个文件。等了 200 毫秒。然后读第二个文件。又等了 200 毫秒。然后读第三个……

三个搜索操作，每个等 600-800 毫秒。如果串行执行，总共等 2 秒。如果并行执行，只需要等 800 毫秒——取最慢的那个。

但这不是最大的浪费。最大的浪费是：Agent 的 LLM 调用本身也要花时间。每次工具调用之后，结果要送回 LLM，LLM 要"思考"，然后决定下一步。这个"思考"过程通常要 1-3 秒。

一次对话如果有 10 轮工具调用，每轮等工具 0.5 秒 + 等 LLM 思考 2 秒 = 每轮 2.5 秒，总共 25 秒。

但如果能把 3 个搜索操作合并成 1 轮（并行执行工具），LLM 思考次数从 10 次减少到 8 次，总时间从 25 秒降到 20 秒。节省 5 秒，20% 的提速。

5 秒听起来不多？在用户体验的世界里，5 秒是"还行"和"有点慢"的分界线。更关键的是：**这个提速几乎不花任何成本。** 不需要更强的模型，不需要更快的网络，只需要在执行工具时做一点简单的分类。

## 为什么 Agent 可以同时做好几件事

你可能会问：LLM 不是"一个一个"回答的吗？它怎么可能同时调用多个工具？

答案是：**现代 LLM 可以在一次回复中返回多个工具调用请求。**

举个例子。你告诉 Agent："帮我找项目里所有的 TypeScript 文件，同时检查一下有没有未使用的 import。"

LLM 的回复可能是：

```
我来同时做这两件事：

工具调用 1：glob("**/*.ts")
工具调用 2：grep("import.*from", "**/*.ts")
```

一次回复，两个工具调用。LLM 在"思考"阶段就规划好了：这两个操作互不依赖，可以同时做。

如果你的框架是串行执行的，它会这样做：

```
执行 glob → 等待 800ms → 返回结果
执行 grep → 等待 600ms → 返回结果
把两个结果一起送给 LLM
总耗时：1,400ms
```

如果你的框架是并行执行的，它会这样做：

```
同时执行 glob 和 grep → 等待 800ms（取最慢的）→ 两个结果一起返回
总耗时：800ms
```

节省 600ms。一轮节省 600ms，10 轮节省 6 秒。

这个优化在技术上叫"批处理"（Batching）——把多个独立的操作打包成一批，同时执行，一起返回。

你可能在日常生活中已经见过这个模式。当你用洗衣机洗衣服时，你不会一件一件洗——你会把一周的衣服攒在一起，一次洗完。洗衣机的运转时间差不多（不管是洗 1 件还是 10 件），但"批处理"让你省掉了 9 次启动和等待的时间。

Agent 的并行执行也是同样的道理。一次 LLM 调用的"思考"时间是固定的（1-3 秒），不管你让它做 1 件事还是 3 件事。如果能让它一次想清楚 3 件事，就省掉了 2 次"思考"的等待。

## 餐厅厨房的启示

让我用一个类比来解释并行执行的核心问题。

想象一个餐厅厨房。厨师要同时做三件事：

1. 烧水煮面（需要 10 分钟）
2. 切西红柿（需要 3 分钟）
3. 炒鸡蛋（需要 5 分钟）

如果只有一个厨师，他必须串行：先煮面（10 分钟），再切西红柿（3 分钟），再炒鸡蛋（5 分钟），总共 18 分钟。

如果有三个厨师，可以并行：一个煮面，一个切西红柿，一个炒鸡蛋，总共 10 分钟（取最慢的那个）。

但有些操作不能并行。比如：

4. 把煮好的面捞出来（需要在煮面完成之后）
5. 把所有食材装盘（需要在所有食材准备好之后）

第 4 步必须等第 1 步完成。第 5 步必须等第 2、3、4 步都完成。这些是"依赖关系"——后面的步骤依赖前面的结果。

厨房的规则很清楚：

- **没有依赖关系的操作可以同时做。** 切西红柿和炒鸡蛋互不影响。
- **有依赖关系的操作必须按顺序来。** 捞面必须等水烧开。
- **装盘必须等所有食材准备好。** 这是"同步点"——所有并行的操作在这里汇合。

Agent 的工具执行也是同样的道理。有些工具互不影响，可以同时执行。有些工具有先后顺序，必须串行。关键是怎么区分。

## 纯工具和不纯工具

区分"可以并行"和"必须串行"的标准只有一个：**这个工具有没有副作用（Side Effect）。**

什么是副作用？简单说：**如果一个操作会改变外部世界的状态，它就有副作用。**

- `glob`（搜索文件列表）：没有副作用。它只是读取文件系统的信息，不改变任何东西。你调用 100 次，文件系统还是原来的样子。

- `grep`（搜索文件内容）：没有副作用。它只是读取文件内容，不改变任何东西。

- `file_read`（读取文件）：没有副作用。它只是读文件，不写文件。

- `web_search`（搜索网页）：没有副作用。它只是发送搜索请求，不改变网页内容。

- `file_write`（写入文件）：**有副作用。** 它会创建或修改文件。如果你同时写入同一个文件，结果是不可预测的——可能互相覆盖，可能写入乱序。

- `shell`（执行命令）：**有副作用。** 它可能修改文件、启动服务、删除目录。如果你同时执行两个 shell 命令，它们可能互相干扰。

- `file_edit`（编辑文件）：**有副作用。** 它会修改文件内容。如果你同时编辑同一个文件的不同部分，可能产生冲突。

AgentClaw 用一个简单的标记来区分这两种工具：**`pure` 字段。**

```typescript
interface Tool {
  name: string;
  description: string;
  parameters: ToolParameterSchema;
  pure?: boolean;  // true = 没有副作用，可以并行
  execute(input, context?): Promise<ToolResult>;
}
```

`pure: true` 的工具是"纯工具"——只读取，不修改。`pure: false`（或不填）的工具是"不纯工具"——会修改外部状态。

这个标记不是给 LLM 看的，是给框架看的。框架根据这个标记决定哪些工具可以并行执行。

在 AgentClaw 的 22 个工具中，有 11 个被标记为 `pure`：

| 纯工具（可并行） | 不纯工具（必须串行） |
|-----------------|-------------------|
| glob, grep, file_read, web_search, web_fetch, context_search | shell, file_write, file_edit, send_file, schedule, remember, update_db, subagent |

超过一半的工具是纯的。这意味着在大多数情况下，LLM 一次回复中的多个工具调用都可以并行执行。

## 批处理逻辑：三十行代码的艺术

有了 `pure` 标记，批处理逻辑就变得很简单。

当 LLM 在一次回复中返回多个工具调用时，框架按顺序扫描这些调用，把连续的纯工具分成一组，遇到不纯工具就"断开"。

举个例子。LLM 返回了 5 个工具调用：

```
[glob, grep, file_write, grep, grep]
```

框架的处理逻辑：

```
第 1 步：glob（纯）→ 加入当前批次
第 2 步：grep（纯）→ 加入当前批次
第 3 步：file_write（不纯）→ 当前批次结束，并行执行 glob+grep；
        file_write 单独串行执行
第 4 步：grep（纯）→ 开始新批次
第 5 步：grep（纯）→ 加入当前批次
第 6 步：没有更多调用 → 并行执行 grep+grep
```

执行计划可以分成三段：

- 第一段：`glob + grep`，并行执行。
- 第二段：`file_write`，串行执行。
- 第三段：`grep + grep`，并行执行。

整个批处理逻辑大约 30 行代码。核心伪代码：

```
batch = []
for 调用 in 工具调用列表:
    if 调用是纯工具:
        batch.push(调用)
    else:
        if batch 不为空:
            并行执行 batch（Promise.all）
            清空 batch
        串行执行 调用
if batch 不为空:
    并行执行 batch
```

就这么简单。30 行代码，每轮工具调用节省 200-800 毫秒。

## 并行的边界：什么时候不能并行

`pure` 标记解决了一大部分问题。但还有一些边界情况需要额外处理。

**边界一：同一个文件的多次写入。**

如果 LLM 返回了两个 `file_write` 调用，都是写入同一个文件，它们不能并行。即使 `file_write` 被标记为不纯工具（所以本来就不会并行），但如果两个不纯工具之间没有其他操作，你可能会想"它们是同一种工具，能不能并行？"

答案是不能。两次写入同一个文件，后一次会覆盖前一次的结果。必须串行。

**边界二：有因果关系的读写。**

如果 LLM 先调用 `file_read` 读取一个文件，然后调用 `file_write` 写入同一个文件（可能是修改后写回），这两个操作有因果关系——写入依赖读取的结果。

在 AgentClaw 的批处理逻辑里，这个问题自然地被解决了：`file_read` 是纯工具，`file_write` 是不纯工具。当扫描到 `file_write` 时，前面的纯工具批次已经执行完了，所以 `file_write` 拿到的是 `file_read` 的最新结果。

**边界三：外部 API 的限速。**

`web_search` 是纯工具，可以并行。但如果你同时发送 5 个搜索请求，搜索引擎可能会限速你。

AgentClaw 的处理方式是：对有速率限制的工具，在并行执行时加上一个"并发度上限"。不是不做并行，而是限制同时执行的数量。比如，最多同时 3 个搜索请求，超过的排队等待。

**边界四：数据库操作。**

如果你的 Agent 需要读写数据库，情况会更复杂。数据库读取通常可以并行，但写入需要考虑事务隔离。这不是 Agent 框架能单独解决的问题——它需要数据库层面的支持。

AgentClaw 的做法是：数据库操作全部标记为不纯，串行执行。宁可慢一点，也不要冒险产生数据不一致。

## 一个真实的性能对比

AgentClaw 在实现批处理之前和之后，做了一次性能对比测试。

测试场景：Agent 执行一个典型的"调查 + 整理"任务——搜索 3 个关键词，读取 5 个网页，写入 1 个报告文件。

**串行执行（实现前）：**

| 步骤 | 工具 | 耗时 |
|------|------|------|
| 1 | web_search("关键词 A") | 800ms |
| 2 | web_search("关键词 B") | 600ms |
| 3 | web_search("关键词 C") | 700ms |
| 4 | web_fetch("网页 1") | 500ms |
| 5 | web_fetch("网页 2") | 400ms |
| 6 | web_fetch("网页 3") | 600ms |
| 7 | web_fetch("网页 4") | 300ms |
| 8 | web_fetch("网页 5") | 500ms |
| 9 | file_write("report.md") | 100ms |
| **工具执行总耗时** | | ** 4,500ms** |

**并行执行（实现后）：**

LLM 在前几轮会把多个搜索/读取操作放在一次回复里。框架自动把它们分成并行批次：

| 步骤 | 工具 | 耗时 |
|------|------|------|
| 1 | web_search(A) + web_search(B) + web_search(C) 并行 | 800ms |
| 2 | web_fetch(1) + web_fetch(2) + web_fetch(3) + web_fetch(4) + web_fetch(5) 并行 | 600ms |
| 3 | file_write("report.md") 串行 | 100ms |
| **工具执行总耗时** | | ** 1,500ms** |

工具执行时间从 4,500ms 降到 1,500ms，**节省 67%。**

但更重要的是 LLM 调用次数的变化。串行执行需要 9 轮工具调用（每轮之后都要回 LLM），而并行执行只需要 3 轮。每轮 LLM 调用的"思考"时间约 2 秒，所以：

- 串行：9 轮 × 2 秒 = 18 秒思考时间
- 并行：3 轮 × 2 秒 = 6 秒思考时间

总时间（工具 + 思考）：
- 串行：4.5 + 18 = 22.5 秒
- 并行：1.5 + 6 = 7.5 秒

**总提速 67%。**

这个提速来自两个因素：工具并行执行节省了时间，LLM 调用次数减少也节省了时间。后者的影响往往更大——因为 LLM 调用的延迟（1-3 秒）通常比工具执行的延迟（0.1-0.8 秒）更长。

还有一个隐藏的收益：**token 节省。** 每一轮 LLM 调用，都要把整个对话历史（包括系统提示词、工具定义、之前的所有消息）重新发送给 API。对话越长，每轮调用的 token 消耗越大。如果并行执行能把 10 轮减少到 3 轮，就省掉了 7 轮重复发送对话历史的 token。

以一个中等长度的对话为例：如果并行执行能把 10 轮减少到 3 轮，就省掉了 7 轮重复发送对话历史的开销。轮次越少，token 节省越明显。

## LLM 的并行意识

有一个关键问题：**LLM 怎么知道哪些操作可以并行？**

答案是：它不总是知道。LLM 是概率模型，它的决策基于训练数据和上下文。有时候它会把可以并行的操作分成多轮（浪费时间），有时候它会把必须串行的操作放在一轮里（可能导致问题）。

AgentClaw 通过工具描述来"引导" LLM 的行为。在系统提示词里，有一段关于工具使用的指导：

> 当你需要同时做多件事时（比如搜索多个关键词、读取多个文件），请在一次回复中返回所有工具调用。框架会自动并行执行没有副作用的操作。

这段话的效果是：LLM 更倾向于在一次回复中返回多个工具调用，而不是一个一个来。

但这个引导不是强制的。LLM 可能仍然选择串行调用——特别是当它不确定操作之间是否有依赖关系时。框架不会因为 LLM 选择了串行就报错，只是慢一点而已。

**原则：框架应该适应 LLM 的行为，而不是强制 LLM 改变行为。** 如果 LLM 选择串行，框架就串行执行。如果 LLM 选择并行，框架就优化执行。框架是 LLM 的"后勤"，不是"指挥官"。

## 一个被删掉的设计：细粒度锁

在 AgentClaw 的早期版本里，我们考虑过一个更"精确"的并行方案：细粒度锁（Fine-Grained Locking）。

想法是这样的：不是简单地用 `pure` 标记区分"可以并行"和"不能并行"，而是为每个资源（文件、数据库表、API 端点）维护一把锁。两个操作如果访问不同的资源，即使都是"不纯"的，也可以并行。

比如，`file_write("a.txt")` 和 `file_write("b.txt")` 写的是不同的文件，理论上可以并行。细粒度锁会为 `a.txt` 和 `b.txt` 各维护一把锁，只有当两个操作访问同一个文件时才串行。

这个方案看起来更"正确"。但我们最终没有实现它。原因：

**第一，复杂度爆炸。** 你需要为文件、数据库、API、内存状态等各种资源维护锁。锁的粒度怎么定？文件级？目录级？字节级？每种选择都有取舍。

**第二，死锁风险。** 如果操作 A 持有锁 1 等待锁 2，操作 B 持有锁 2 等待锁 1，就死锁了。你需要实现死锁检测或超时机制，这又增加了复杂度。

**第三，收益不大。** Agent 的工具调用中，大部分操作都是读操作（搜索、读文件、读网页），写操作相对较少。用 `pure` 标记已经能把大部分读操作并行化了。为了那一小部分"不同文件的并行写入"而引入复杂的锁机制，投入产出比不划算。

**最终方案：简单胜于精确。** `pure` 标记 + 批处理逻辑，30 行代码，解决了 90% 的并行问题。剩下的 10%（不同文件的并行写入）串行执行，多花几百毫秒，但代码简单、没有死锁风险、容易理解和调试。

## 并行不是万能的

最后说一个常见的误解：并行不是越多越好。

有些任务天然是串行的。比如：

- "先搜索，再根据搜索结果写报告"——写报告依赖搜索结果，不能并行。
- "先读取配置文件，再根据配置执行命令"——执行依赖配置，不能并行。
- "先创建目录，再写入文件"——写入依赖目录存在，不能并行。

强行并行化这些任务，要么导致错误（读取到旧数据），要么导致结果不可预测（写入顺序不确定）。

AgentClaw 的批处理逻辑尊重这些依赖关系。它不会试图"智能地"分析依赖关系——它只是简单地把纯工具放在一起并行，不纯工具串行。这个策略不总是最优的，但它是安全的。

**原则：并行是优化手段，不是目标。** 目标是"正确地完成任务"，并行只是在不破坏正确性的前提下提升速度。如果不确定能不能并行，就串行。慢一点总比出错好。

## 本章小结

1. **并行执行能节省可观的时间。** 主要来自两个因素：工具并行执行和 LLM 调用次数减少。

2. **区分"可以并行"和"必须串行"的标准是副作用。** 只读操作（`pure: true`）可以并行，写操作必须串行。

3. **批处理逻辑只有 30 行代码。** 把连续的纯工具分成一组并行执行，遇到不纯工具就断开。

4. **并行的边界需要额外处理。** 同一个文件的多次写入、有因果关系的读写、外部 API 限速、数据库操作——这些情况需要特殊处理。

5. **简单胜于精确。** `pure` 标记 + 批处理逻辑解决了绝大多数问题。细粒度锁听起来更"正确"，但复杂度和风险不值得。

6. **框架应该适应 LLM 的行为。** 如果 LLM 选择串行，框架就串行执行。不要强制 LLM 改变行为。
