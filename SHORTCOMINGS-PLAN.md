## P0：建立 Eval 基准（无法度量 = 无法改进）

**现状**：`eval.ts` 框架 95% 完成，但没有真实测试集、没有运行入口、没有 CI 集成。

**目标**：能一条命令跑 eval，量化 agent 表现，后续改动有对比基线。

- [x] **P0-1** 建 `data/golden-testcases.json`（20 个测试用例，覆盖工具选择/安全/多步骤等场景）
- [x] **P0-2** 添加 Gateway 路由 `POST /api/eval/run`（`packages/gateway/src/routes/eval.ts`）
- [x] **P0-3** 添加 `npm run eval`（`scripts/eval.mjs`，CLI 一键运行）
- [x] **P0-4** Baseline：2/20 通过（1294 条 trace，大部分测试用例无匹配 trace——后续随使用积累数据自动改善）

## P1-A：记忆可主动检索（记了但想不起来）

**现状**：记忆在会话开始时查一次注入系统提示词，冻结不变。会话中途提到新话题，已存的相关记忆检索不到。

**目标**：agent 能在对话中途主动查记忆库。

- [x] **P1A-1** 新增 `recall` 只读工具（`packages/tools/src/builtin/recall.ts`，支持关键词 + 语义 + 类型过滤）
- [x] **P1A-2** 系统提示词增加 recall 使用指引（"用户提到上次/之前时，用 recall 搜索"）
- [x] **P1A-3** 端到端测试通过：新会话 → 请求 recall 搜索邮箱 → 成功找到两个邮箱地址

---

## P1-B：强化规划能力（复杂任务不靠直觉）

**现状**：系统提示词只引导"多步骤先列 todo"，没有依赖分析、验证条件、失败回退。

**目标**：复杂任务（5+ 步）有结构化计划，每步有验证条件。

- [x] **P1B-1** 系统提示词增加"复杂任务规划"引导（5+ 步骤时：列步骤 + 验证条件 + 失败回退 + update_todo）
- [x] **P1B-2** 提示词已生效（无需 build，系统提示词运行时读取），后续通过实际使用观察效果

---

## P2-A：从失败中自动学习

**现状**：MemoryExtractor 只提取用户信息，agent 自身的操作经验（哪些操作在什么条件下失败）没有积累机制。

**目标**：失败经验自动沉淀，同类错误不重犯。

- [x] **P2A-1** `MemoryExtractor.processTrace()` 方法：分析失败 trace → LLM 提取操作教训 → 语义去重后存储
- [x] **P2A-2** 复用 `episodic` 类型，格式："条件 → 失败 → 正确做法"
- [x] **P2A-3** orchestrator 在每次对话结束后自动检查最新 trace，有工具错误或 loop 耗尽时触发学习

---

## P2-B：关键工具结果验证

**现状**：工具执行后直接进入下一步，没有 post-condition 检查。

**目标**：关键操作（文件写入、代码执行）有自动验证。

- [x] **P2B-1** file_write 语法检查：JS/TS（Biome lint，已有）+ JSON（JSON.parse 验证，新增）+ Python（py_compile，新增）
- [x] **P2B-2** shell exit code 非零自动加警告（已有 preset hook）
- [x] **P2B-3** 构建通过，hooks 在 orchestrator 启动时自动注册
